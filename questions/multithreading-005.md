# Вопрос
Спроектируйте распределенный планировщик задач (Distributed Task Scheduler) для микросервисной архитектуры. Система должна выполнять миллионы отложенных задач (например, отправить push-уведомление ровно через 24 часа после события). 

У нас есть кластер из 10 одинаковых instance'ов нашего сервиса (нод), работающих с одной реляционной базой данных (PostgreSQL) или брокером (RabbitMQ/Kafka).

**Требования:**
1. Задача не должна выполняться одновременно двумя нодами (исключение дубликатов).
2. Если нода взяла задачу и "упала" (OOM/Crash), задача не должна зависнуть навсегда, ее должна подхватить другая нода.
3. Опишите trade-offs выбранного решения.

---

# Ответ

В распределенных системах полагаться на in-memory таймеры (типа `ScheduledExecutorService`) нельзя, так как состояние теряется при рестарте ноды. Нужен внешний координатор — База Данных или Брокер Сообщений.

**Подход 1: Database Polling с блокировками (Реляционная БД)**
Таблица `tasks (id, payload, execution_time, status, locked_by, locked_at)`.
1. Каждая нода раз в N секунд делает запрос к БД на поиск задач, чье время `execution_time <= NOW()` и `status = PENDING`.
2. Чтобы избежать захвата одной задачи разными нодами, используем **Row-level Locks** с директивой `SKIP LOCKED`.
   *SQL:* `UPDATE tasks SET status = 'PROCESSING', locked_by = 'node_id', locked_at = NOW() WHERE id IN (SELECT id FROM tasks WHERE execution_time <= NOW() AND status = 'PENDING' LIMIT 10 FOR UPDATE SKIP LOCKED) RETURNING *;`
3. Выполняем бизнес-логику (отправку пуша).
4. Обновляем статус: `status = 'COMPLETED'`.
5. **Обработка падения:** Фоновый job ищет задачи со статусом `PROCESSING`, у которых `locked_at` старше, чем разумный таймаут (например, 5 минут). У них статус сбрасывается обратно в `PENDING`, и другая нода заберет их при следующем поллинге.

**Подход 2: Broker с отложенной маршрутизацией (RabbitMQ + Delayed Message Plugin)**
При создании задачи мы кидаем сообщение в RabbitMQ с TTL = 24 часа (или используем плагин x-delayed-message).
1. Сообщение висит в exchange 24 часа, после কমপক্ষে чего падает в рабочую очередь.
2. Ноды подключены к очереди как Consumer-ы. RabbitMQ сам балансирует сообщения между 10 нодами (конкурентные консьюмеры). Исключение дубликатов из коробки.
3. **Обработка падения:** Используем ручное подтверждение (`manual ack`). Сообщение удаляется из RabbitMQ только после успешной отправки пуша. Если нода упала, соединение разрывается, и брокер автоматически возвращает сообщение в очередь, отдавая его другой ноде.

**Trade-offs (Компромиссы):**
- **DB Polling:** Проще в реализации, нет новых инфраструктурных зависимостей, легко мониторить. *Минусы:* Нагрузка на БД при частом поллинге, задержки выполнения зависят от интервала поллинга (может быть дрифт в несколько секунд).
- **RabbitMQ:** Выполнение миллисекунда в миллисекунду, нулевая нагрузка на БД, встроенная балансировка и `ack`. *Минусы:* Труднее отменить или изменить запланированную задачу (найти сообщение внутри брокера почти невозможно, придется реализовывать логику инвалидации на стороне консьюмера), требуется плагин.

**Идемпотентность (защита от повторов):** Оба варианта обеспечивают *at-least-once* доставку. Если нода отправила пуш, но упала до записи статуса/ack, задача повторится. Метод отправки пуша должен быть идемпотентным (например, внешнему API передается уникальный `Idempotency-Key` на основе `task_id`).

### Аналогия из жизни
База Данных с поллингом — это доска объявлений. 10 рабочих постоянно подбегают к доске. `SKIP LOCKED` — это когда рабочий срывает стикер с задачей, чтобы другой не взял ее. Если рабочий пропал на час (упала нода), прораб (фоновый джоб) видит это, печатает дубликат стикера и вешает обратно на доску.

### Ключевые моменты
- Локальные шедулеры не работают в кластере. Требуется внешнее хранилище состояния.
- При использовании БД паттерн `SELECT ... FOR UPDATE SKIP LOCKED` критически важен для конкурентной выборки без взаимных блокировок.
- Обработка сбоев ("зомби-задачи") решается через таймауты блокировок (locked_at) в БД или механизмы подтверждения (`ack` / `nack`) в брокерах очередей.
- Идемпотентность на стороне исполнителя обязательна, так как распределенные системы гарантируют at-least-once доставку.
