# Modeling Domain States: Sealed Classes vs. Enums

## Question
You are designing a payment processing system in Java 21. You need to represent the result of a payment, which can be exactly one of three states: `Success`, `Failed`, or `Pending`. 

A junior developer suggests using a standard Java `enum` for this. However, you know that `Success` needs to carry a transaction ID (String), `Failed` needs to carry an error code (int), and `Pending` carries no extra data. 

Explain why an `enum` is insufficient for this requirement, and describe how Java's Modern OOP features (specifically Sealed Classes/Interfaces and Records) provide a superior domain modeling solution.

## Analogy
An `Enum` is like a set of traffic lights: Red, Yellow, Green. They are static, singular states; there is only ever one "Red". 
A `Sealed Class` is like a specific type of vehicle allowed on a private road (Car, Truck, Motorcycle). You strictly control the categories, but there can be thousands of different Cars, each with their own unique license plate and passengers.

## Explanation

**The Limitation of Enums:**
Enums in Java are singletons. While they can have fields, those fields are shared across the entire JVM for that specific enum constant. You cannot create a `PaymentStatus.SUCCESS` with transaction "A12" and another `PaymentStatus.SUCCESS` with transaction "B99" simultaneously.

**The Power of Sealed Interfaces:**
Java 17+ introduced `sealed` types, which allow a class or interface to strictly control which classes are allowed to implement or extend it. Combined with `record` types, this gives Java Algebraic Data Types (ADTs).

This approach allows you to:
1. Guarantee exhaustive states (just like Enums).
2. Carry unique, instance-specific data for each state.

### Example Solution
```java
public sealed interface PaymentResult permits Success, Failed, Pending {}

public record Success(String transactionId) implements PaymentResult {}
public record Failed(int errorCode) implements PaymentResult {}
public record Pending() implements PaymentResult {}
```

By sealing the hierarchy, the compiler knows exactly what subtypes exist, allowing for exhaustive pattern matching in `switch` expressions without needing a `default` branch.

## Mandatory Key Points for a Correct Answer
1. **Enum Singleton Nature:** Identify that enums cannot hold instance-specific data per state occurrence because they are singletons.
2. **Sealed Hierarchy:** Propose using a `sealed interface` or `abstract sealed class` to restrict the hierarchy to only the allowed states.
3. **Data Carriers:** Use `record` or standard classes for the implementations so they can hold instance-specific data (e.g., transaction IDs).
4. **Exhaustive Pattern Matching:** Mention that sealed classes enable the compiler to enforce exhaustive `switch` statements, preventing unhandled states.
