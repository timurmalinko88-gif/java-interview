In Java, we frequently hear the advice to "favor composition over inheritance." 
However, inheritance is a fundamental pillar of OOP. 
Explain this principle. In what scenarios would you explicitly choose composition over inheritance, and when is inheritance actually the right choice? Provide concrete examples.

---ANSWER---

### Life Analogy
Inheritance is like buying a pre-furnished house: you get everything built-in, but changing the layout is hard, and you might get furniture you don't need. Composition is like renting an empty apartment and buying your own furniture: you plug in exactly what you need, and you can easily replace the couch later without rebuilding the walls.

### Structuring the Answer
- **Inheritance (`is-a` relationship)**: Creates tight coupling. Subclasses inherit both behavior and implementation, which can lead to fragile base classes.
- **Composition (`has-a` relationship)**: Promotes loose coupling. You build complex behavior by combining simpler, independent objects, typically via interfaces.

### Key Points for a Correct Answer
- **Flexibility**: Composition allows behavior to change at runtime (Strategy pattern), whereas inheritance is statically resolved at compile time.
- **Encapsulation**: Inheritance exposes protected members and internal workings of the parent class to the child class, breaking encapsulation.
- **Single Inheritance Limit**: Java only allows extending one class. Tying up that single inheritance slot for code reuse limits future architectural choices.
- **When to use Inheritance**: Only when there is a true, undeniable "is-a" relationship, and the subclass strictly adheres to the Liskov Substitution Principle.
- **When to use Composition**: When you need code reuse without sharing an identity, or when an object needs to dynamically change its behavior.
