# The Tale of Two Collectors: Latency vs. Throughput

## Question

You are migrating two distinct Java 21 applications from an older Java 8 environment. 
1. **App A** is a High-Frequency Trading (HFT) matching engine where any pause over 5 milliseconds results in severe financial loss. 
2. **App B** is an overnight batch processing engine that crunches terabytes of data to generate a single report by morning.

Which Garbage Collector (GC) would you configure for each application (choosing between G1GC, ZGC, and Parallel GC), and what trade-offs are you accepting with your choices?

## Analogy
Choosing a GC is like hiring a cleaning service for an office. 
* A **Background Cleaner (ZGC)** works silently while everyone is at their desks; nobody gets interrupted, but the cleaner takes up some space in the hallways (CPU overhead).
* A **Deep Cleaning Crew (Parallel GC)** forces everyone out of the building for an hour. Nobody can work, but the cleaning is done incredibly fast and efficiently with zero ongoing distractions.

## Answer Structure

### 1. App A: High-Frequency Trading Engine
**Choice: ZGC (Z Garbage Collector) or Shenandoah GC**
* **Why:** ZGC is a scalable, low-latency garbage collector designed to guarantee pause times well under 1 millisecond, regardless of heap size. 
* **Trade-off:** ZGC achieves this by doing almost all its work concurrently with application threads. This uses up a portion of the application's CPU cycles (throughput reduction) and requires colored pointers/load barriers, which adds a slight baseline overhead.

### 2. App B: Overnight Batch Processor
**Choice: Parallel GC (or G1GC heavily tuned for throughput)**
* **Why:** The batch job doesn't care about micro-pauses; it only cares about total execution time. Parallel GC maximizes throughput by dedicating all CPU resources to the application, then completely pausing the application ("Stop-The-World") to clean up garbage as fast as possible using all cores.
* **Trade-off:** Long "Stop-The-World" pauses. If this were an interactive web server, users would experience severe lag, but for a batch job, this results in the fastest total completion time.

### 3. Where does G1GC fit?
G1GC is the default in modern Java because it provides a configurable balance. It aims for high throughput while allowing you to specify a target maximum pause time (e.g., 200ms). It's a general-purpose compromise.

## Mandatory Key Points
1. **ZGC** optimizes for **latency** (sub-millisecond pauses) at the expense of overall CPU throughput.
2. **Parallel GC** optimizes for **throughput** (total work done per hour) at the expense of long "Stop-The-World" pauses.
3. Concurrent collectors like ZGC require extra CPU cycles and memory barriers to clean alongside running application threads.
