# Inheritance vs Composition in Modern Java

### Question
"Favor composition over inheritance" is a classic OOP mantra. How does modern Java (specifically features like Records and Sealed Classes introduced in Java 15-17) change or reinforce this principle? Give a concrete example where you would choose one over the other.

### Analogy
Think of **Inheritance** like biology (a dog *is a* mammal—you get all the DNA, even the parts you don't need). Think of **Composition** like Lego bricks (a car *has a* wheel—you plug in exactly what you need without becoming a wheel). 

### Answer Structure

#### 1. The Core Principle
Inheritance creates a tight coupling (the fragile base class problem) and breaks encapsulation if not designed carefully. Composition allows assembling behaviors dynamically at runtime and keeps components isolated.

#### 2. Modern Java Impact
- **Records (Java 16):** Records do not support inheritance (they are implicitly `final` and extend `java.lang.Record`). This naturally forces developers to use composition if they want to reuse behavior across different record types.
- **Sealed Classes (Java 17):** They restrict which classes can extend them. This makes inheritance *safer* by allowing pattern matching and exhaustive `switch` statements without default branches, taming the wild nature of deep inheritance trees.

#### 3. When to use which?
- **Inheritance (with Sealed Classes):** Modeling a closed domain hierarchy. Example: `sealed interface PaymentMethod permits CreditCard, PayPal, Crypto`. You know exactly what the implementations will be, and it's a structural "IS-A" relationship.
- **Composition (with Records/Classes):** When assembling reusable logic. Example: `record Order(PaymentMethod paymentMethod, Customer customer)`. The order "HAS-A" payment method, delegating logic rather than inheriting it.
