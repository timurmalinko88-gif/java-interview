## Question

Вам нужно сделать 20 независимых HTTP-запросов к стороннему метеорологическому API, собрать их результаты в список и вернуть пользователю.

У вас есть два варианта параллельного выполнения:

* Использовать `urls.parallelStream().map(...).collect(...)`
* Создать список задач через `CompletableFuture.supplyAsync(...)` и объединить их.

Какой вариант вы выберете для production-кода и почему? Что произойдет "под капотом" в обоих случаях?

## Answer

Правильный выбор для HTTP-запросов (I/O-bound задач) — **`CompletableFuture` с кастомным пулом потоков**.

**Почему не `parallelStream()`?**

`parallelStream` по умолчанию использует общий `ForkJoinPool.commonPool()`. Размер этого пула равен количеству логических ядер процессора минус 1 (например, на 4-ядерном сервере там будет 3 потока). HTTP-запрос — это I/O операция. Поток, отправляющий запрос, просто блокируется в ожидании ответа сети.

Если вы отправите 20 запросов через `parallelStream`, вы моментально забьете весь `commonPool` заблокированными потоками. Это повлияет на всё остальное приложение — любые другие параллельные стримы в других частях системы просто встанут в очередь и будут ждать, пока сеть ответит.

**Почему `CompletableFuture`?**

В метод `CompletableFuture.supplyAsync(supplier, executor)` можно передать кастомный `ExecutorService` (например, настроенный на 50-100 потоков специально для I/O операций). Это позволяет HTTP-запросам выполняться в изолированном пуле, не блокируя общесистемный `commonPool`. Кроме того, `CompletableFuture` предоставляет богатый API для комбинирования асинхронных результатов (например, `allOf()`) и обработки таймаутов/ошибок.

### Ключевые моменты

* `parallelStream` использует `ForkJoinPool.commonPool()`, который оптимизирован для вычислительных (CPU-bound) задач, а не для ожидания (I/O).
* Блокировка потоков в `commonPool` вызывает деградацию всего приложения (Thread Starvation).
* Для сетевых запросов и работы с БД (I/O-bound) необходимо использовать асинхронность с отдельными, специально настроенными пулами потоков.
* `CompletableFuture` позволяет явно указать `Executor` и легко комбинировать результаты независимых вызовов.

## Analogy

`commonPool` — это общие кассы в супермаркете. Если вы отправите туда людей, которые будут 10 минут искать мелочь в кошельке (HTTP-запрос), то вся очередь в магазине встанет. Кастомный пул в `CompletableFuture` — это выделение отдельного окна "только для долгих операций", чтобы не мешать основному потоку покупателей (CPU-операциям).
