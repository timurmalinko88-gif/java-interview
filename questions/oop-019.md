# Question: Defensive Copying and Encapsulation

You are designing an immutable `Person` class that holds a name and a list of hobbies. 
Explain how you would ensure true immutability in Java, especially when dealing with mutable dependencies like collections or dates. How do Java 14+ Records change this paradigm?

## Answer

### Life Analogy
Imagine giving someone a copy of your house keys (defensive copying) versus giving them the master key (passing a reference). If they lose the copy, your main locks are still safe. If they lose the master, you have a security breach (encapsulation broken).

### Key Concepts

*   **Shallow Immutability vs Deep Immutability:** Simply making fields `final` or using an interface doesn't make an object deeply immutable if those fields are references to mutable objects (like `List`, `Date`, or arrays).
*   **Defensive Copying (Constructor):** When accepting a mutable object via a constructor, create a copy of it before assigning it to the internal field.
*   **Defensive Copying (Getters):** When returning a mutable object via a getter, return a copy (or an unmodifiable view) so the caller cannot alter the internal state.
*   **Java Records:** Records provide boilerplate-free shallow immutability. However, they do **not** automatically perform defensive copying. If a Record contains a `List`, the list itself can still be mutated unless you explicitly make it unmodifiable in a compact constructor.

### Example Implementation

```java
import java.util.List;

public record Person(String name, List<String> hobbies) {
    
    // Compact constructor to enforce defensive copying
    public Person {
        // Create an unmodifiable copy to prevent external mutation
        hobbies = List.copyOf(hobbies); 
    }
}
```

### Key Points to Look For in an Answer
1. Clearly distinguishes between shallow and deep immutability.
2. Explains that assigning a `List` directly allows external code to modify the object's internal state.
3. Mentions defensive copying in both constructors (incoming data) and getters (outgoing data).
4. Suggests using `List.copyOf()` or `Collections.unmodifiableList()`.
5. Correctly notes that Java Records are only shallowly immutable and still require compact constructors for deep immutability with collections.
