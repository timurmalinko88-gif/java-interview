Question

During a load test of an API Gateway running on Tomcat, you observe a slow but steady increase in heap memory utilization. A heap dump analysis reveals thousands of instances of UserContext objects that are never reclaimed.

Analyze the code snippet below. Identify the structural design error, explain the interaction between ThreadLocalMap entries, weak references, and thread pools, and provide the corrected version.

public class UserContextHolder {
    private static final ThreadLocal<UserContext> contextHolder = new ThreadLocal<>();

    public static void setContext(UserContext context) {
        contextHolder.set(context);
    }

    public static UserContext getContext() {
        return contextHolder.get();
    }
}

// In Tomcat Request Filter:
public class SecurityFilter implements Filter {
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) 
            throws IOException, ServletException {
        
        String token = ((HttpServletRequest) req).getHeader("X-Auth-Token");
        UserContext context = TokenParser.parse(token); // Allocation
        
        UserContextHolder.setContext(context);
        
        chain.doFilter(req, res); // Invoke application downstream
    }
}


Life Analogy

Imagine a hotel where guests rent rooms, and every room has a personal desk drawer (ThreadLocal).

Tomcat uses a pool of workers. Instead of a guest checking out and the hotel demolishing the room, the room is kept, and a new guest checks in the next day.

When a guest checks in, they put their valuable passport (UserContext) inside the room's desk drawer.

The guest departs (request ends), but they forget to empty the desk drawer because they are in a rush.

Because the hotel worker never clears the room between guests, the desk drawer stays locked with the passport inside. As hundreds of guests use the same room over a weekend, old passports pile up inside, eventually filling the entire hotel with forgotten luggage until the floor collapses (OOM).

Deep Dive: ThreadLocal and Thread Pool Interactions

The code above suffers from a classic ThreadLocal Memory Leak when run inside an application server like Tomcat, Jetty, or a custom thread pool framework.

1. The Core Trap: Thread Reuse

Application servers use a Thread Pool (such as Tomcat's worker threads) to handle HTTP requests. When a request completes, the thread is not destroyed. Instead, it goes back to the pool to wait for the next request.

Request 1 (Assigned to Thread "http-nio-exec-1"):
   --> Sets UserContext in ThreadLocalMap of thread "http-nio-exec-1"
   --> Request finishes. Thread goes back to pool. 
   --> UserContext stays pinned to Thread's internal ThreadLocalMap!

Request 2 (Assigned to Thread "http-nio-exec-1"):
   --> Overwrites or keeps reference.
   --> Over time, every thread in the pool holds strong references to dynamic objects.


2. The Illusion of Weak References in ThreadLocalMap

ThreadLocal instances store data inside a custom hash map called ThreadLocalMap, which is owned by the java.lang.Thread object itself, not the ThreadLocal class.

The entries of this map extend WeakReference:

static class Entry extends WeakReference<ThreadLocal<?>> {
    Object value; // <-- Strong Reference to the actual data!
    
    Entry(ThreadLocal<?> k, Object v) {
        super(k);
        value = v;
    }
}


Key is Weak: The key (the ThreadLocal object) is wrapped in a WeakReference. If the UserContextHolder class is unloaded or the ThreadLocal reference is garbage collected, the key becomes null.

Value is Strong: The value (the UserContext object) is a Strong Reference.

The GC Block: Even when the key becomes null, the value cannot be garbage collected because the Thread object (which is actively running in the pool) holds a strong reference to the ThreadLocalMap, which contains the Entry, which contains the value.

The value will only be cleaned up if the thread dies, or if the thread executes subsequent ThreadLocal operations (set(), get(), or remove()) which perform opportunistic scans to clean up orphan entries.

Remediation: Corrected Implementation

To prevent this memory leak, you must guarantee that the ThreadLocal is cleaned up when the request execution block terminates. This is achieved by enclosing the downstream execution in a try-finally block inside the Filter.

Corrected UserContextHolder:

public class UserContextHolder {
    private static final ThreadLocal<UserContext> contextHolder = new ThreadLocal<>();

    public static void setContext(UserContext context) {
        contextHolder.set(context);
    }

    public static UserContext getContext() {
        return contextHolder.get();
    }

    // Explicitly clean up the thread-local allocation!
    public static void clearContext() {
        contextHolder.remove();
    }
}


Corrected SecurityFilter:

public class SecurityFilter implements Filter {
    public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain) 
            throws IOException, ServletException {
        
        String token = ((HttpServletRequest) req).getHeader("X-Auth-Token");
        UserContext context = TokenParser.parse(token);
        
        try {
            UserContextHolder.setContext(context);
            chain.doFilter(req, res); // Downstream processes execute with context active
        } finally {
            // Guaranteed to execute even if downstream throws an Exception
            UserContextHolder.clearContext();
        }
    }
}


Key Points (Must Cover for Full Credit)

Thread Pool Retention: Explain that thread pools reuse threads indefinitely, meaning any data stored in a thread's ThreadLocalMap persists past the request lifetime.

Weak Key vs. Strong Value: Clarify that while the keys in ThreadLocalMap are weak references, the values are strong references and will leak if not removed manually.

The Tomcat Context: Relieve how Tomcat threads hold onto UserContext instances long after HTTP connections terminate.

Try-Finally Pattern: Implement the explicit .remove() call inside a finally block of a filter or interceptor to guarantee cleanup.
