Question

Calculate and compare the exact memory footprint required to store 1,000,000 primitive integers in three different structures on a 64-bit HotSpot JVM with Compressed OOPs enabled (and 8-byte object alignment):

A primitive array: int[]

An ArrayList<Integer> (fully compacted, with no empty trailing array capacity)

A standard LinkedList<Integer>

Provide the step-by-step mathematical calculations for each structure, taking into account object headers, reference sizes, and alignment padding. Explain the architectural hardware consequences of these memory layouts on CPU L1/L2 cache efficiency.

Life Analogy

Imagine you need to store 1,000 eggs (integers) in a warehouse.

Primitive Array (int[]) is like a molded egg carton. It has exactly 1,000 fitted slots. The carton itself has a tiny packaging overhead (the object header), but the eggs sit directly next to each other in perfect, dense order. You can fit thousands of eggs in a tiny shelf space.

ArrayList<Integer> is like buying 1,000 luxury individual egg cups, and then placing all 1,000 cups inside a larger cardboard shipping crate. Each individual cup (wrapper object) consumes space, has its own decorative stand (object header), and the crate has to hold 1,000 individual slots just to hold the cups.

LinkedList<Integer> is the absolute worst: it is like 1,000 individual storage lockers scattered randomly across different cities. Inside Locker 1, you find a single egg cup (wrapped integer) and two written letters containing GPS coordinates pointing to Locker 2 and the locker before it. Finding your eggs requires traveling across the state over and over, ignoring the highway system entirely.

Under the Hood: JVM Memory Calculations (64-bit with Compressed OOPs)

Core Rules for Calculation:

Compressed OOPs: All object references (pointers) are 4 bytes (instead of 8 bytes).

Standard Object Header: Consists of 12 bytes (8-byte Mark Word + 4-byte compressed Klass Word).

Array Header: Consists of 16 bytes (8-byte Mark Word + 4-byte Klass Word + 4-byte array length field).

8-Byte Object Alignment: Every object size in memory must be padded up to a multiple of 8 bytes.

Structure 1: Primitive Array (int[] arr = new int[1_000_000])

A primitive array is represented as a single contiguous object on the heap.

Array Object Header: 16 bytes

Element Payload: $1,000,000 \times 4 \text{ bytes (size of a primitive int)} = 4,000,000 \text{ bytes}$

Total Size: $16 + 4,000,000 = 4,000,016 \text{ bytes}$

Alignment Padding: The value $4,000,016$ is already a multiple of 8 ($4,000,016 \div 8 = 500,002$). No padding is needed.

Total Memory Footprint: $\mathbf{4,000,016\text{ bytes}} \approx \mathbf{4.0\text{ MB}}$

Structure 2: ArrayList<Integer> (Compacted)

An ArrayList consists of the ArrayList wrapper object itself, which points to an internal backing array object (Object[]), which in turn contains references pointing to individual Integer objects on the heap.

Step A: The ArrayList Wrapper Object

Object Header: 12 bytes

Fields:

int size (actual size): 4 bytes

Object[] elementData (reference to backing array): 4 bytes (Compressed OOP)

int modCount (inherited modification tracker): 4 bytes

Subtotal: $12 + 4 + 4 + 4 = 24 \text{ bytes}$ (already 8-byte aligned).

Step B: The Backing Reference Array (Object[])

Array Object Header: 16 bytes

Payload (references): $1,000,000 \times 4 \text{ bytes (Compressed OOPs)} = 4,000,000 \text{ bytes}$

Subtotal: $16 + 4,000,000 = 4,000,016 \text{ bytes}$ (already 8-byte aligned).

Step C: The 1,000,000 Integer Objects

Each individual integer is boxed into its own java.lang.Integer object:

Object Header: 12 bytes

Payload field: int value: 4 bytes

Subtotal per Integer: $12 + 4 = 16 \text{ bytes}$ (already 8-byte aligned).

Total for 1,000,000 Integers: $1,000,000 \times 16 \text{ bytes} = 16,000,000 \text{ bytes}$.

Combined Total for ArrayList:

$$\text{Total} = 24 \text{ (Wrapper)} + 4,000,016 \text{ (Backing Array)} + 16,000,000 \text{ (Objects)} = \mathbf{20,000,040\text{ bytes}} \approx \mathbf{20.0\text{ MB}}$$

Structure 3: LinkedList<Integer>

A LinkedList consists of the LinkedList wrapper object, which references individual node objects. Every Node contains a reference to the data (Integer), a reference to the next node, and a reference to the previous node.

Step A: The LinkedList Wrapper Object

Object Header: 12 bytes

Fields:

int size: 4 bytes

Node first (reference): 4 bytes

Node last (reference): 4 bytes

int modCount (inherited): 4 bytes

Subtotal: $12 + 4 + 4 + 4 + 4 = 28 \text{ bytes}$.

Aligned wrapper: Padded to $\mathbf{32 \text{ bytes}}$.

Step B: The 1,000,000 Node Objects

Each node in a doubly linked list is a separate heap object:

Object Header: 12 bytes

Fields:

E item (reference to Integer): 4 bytes

Node next (reference to next Node): 4 bytes

Node prev (reference to previous Node): 4 bytes

Subtotal per Node: $12 + 4 + 4 + 4 = 24 \text{ bytes}$ (already 8-byte aligned).

Total for 1,000,000 Nodes: $1,000,000 \times 24 \text{ bytes} = 24,000,000 \text{ bytes}$.

Step C: The 1,000,000 Integer Objects

Same as computed in the ArrayList step:

$1,000,000 \times 16 \text{ bytes} = 16,000,000 \text{ bytes}$.

Combined Total for LinkedList:

$$\text{Total} = 32 \text{ (Wrapper)} + 24,000,000 \text{ (Nodes)} + 16,000,000 \text{ (Integers)} = \mathbf{40,000,032\text{ bytes}} \approx \mathbf{40.0\text{ MB}}$$

Architectural Performance Consequences

Comparing the structures highlights why raw memory consumption dictates execution speed on modern hardware:

Primitive Array Memory Layout (Contiguous):
[ 16B Header ][ Int 1 ][ Int 2 ][ Int 3 ][ Int 4 ][ Int 5 ]  <-- Fits cleanly in CPU Cache Line

ArrayList Memory Layout (Referential Pointer Chase):
[ Object Array References ] ---> [ Integer 1 Object (16B) ] (Scattered across heap)
                            ---> [ Integer 2 Object (16B) ] (Scattered across heap)


Cache Line Utilization (64-Byte Bounds):

Modern CPUs retrieve memory in 64-byte chunks (Cache Lines).

With int[], a single cache line load pulls exactly 16 consecutive integers into the L1 CPU Cache instantly. Iterating through the array is incredibly fast because of predictable memory strides and automatic CPU prefetching.

With ArrayList and LinkedList, loading a cache line only retrieves reference pointers or the node metadata wrapper. The actual Integer values are scattered across different addresses on the heap. Accessing each element requires a pointer dereference (pointer chasing), causing constant L1/L2 Cache Misses.

Garbage Collection Overhead:

An int[] is a single object for the GC to scan.

An ArrayList<Integer> contains 1,000,002 separate objects to track.

A LinkedList<Integer> contains 2,000,002 separate objects to track.

This millions-fold object count increase dramatically prolongs GC scanning phases (e.g., G1 root scanning and concurrent marking), driving CPU waste.

Primitive Array Footprint: Must calculate $4\text{ MB}$ exactly.

ArrayList Footprint: Detail wrapper, backing pointer array, and individual boxed Integers to calculate $20\text{ MB}$ exactly.

LinkedList Footprint: Account for the node objects (prev/next/item fields) and Integers to reach $40\text{ MB}$ exactly.

Cache Locality and Pointers: Contrast the contiguous memory layout of primitive arrays (which maps perfectly to 64-byte CPU cache lines) with pointer-chasing in linked structures that degrades cache efficiency.
