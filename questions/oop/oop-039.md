# Leaky Immutability and Liskov Substitution

## Context
A developer was tasked with creating an immutable `TaskBatch` class for a multi-threaded task processor. Another developer subclassed it to create a `StrictTaskBatch` that enforces validation. The code compiles under Java 17/21.

## Code Review Snippet
```java
import java.util.List;
import java.util.ArrayList;

class TaskBatch {
    private final String batchId;
    private final List<String> tasks;

    public TaskBatch(String batchId, List<String> tasks) {
        this.batchId = batchId;
        this.tasks = tasks; // Optimization: avoid copying large lists
    }

    public String getBatchId() {
        return batchId;
    }

    public List<String> getTasks() {
        return tasks; 
    }

    public void displayBatchInfo() {
        System.out.println("Batch: " + batchId + " | Tasks: " + tasks.size());
    }
}

class StrictTaskBatch extends TaskBatch {
    
    public StrictTaskBatch(String batchId, List<String> tasks) {
        super(batchId, tasks);
        if (tasks.isEmpty()) {
            throw new IllegalArgumentException("Strict batches cannot be empty");
        }
    }

    @Override
    public void displayBatchInfo() {
        throw new UnsupportedOperationException("Display is disabled for secure strict batches.");
    }
}
```
Review the above code for OOP and design flaws. Point out the specific bugs related to immutability and SOLID principles, and explain how to refactor them.

## Analogy
- **Leaky Immutability**: It’s like giving someone a "read-only" view of your bank account, but accidentally handing them your debit card PIN along with the statement.
- **LSP Violation**: It's like buying a Universal Remote that claims to control any TV, but pressing the 'Volume' button makes it self-destruct because it decided "secure TVs shouldn't have volume control."

## Expected Answer Structure
- Identify mutability leaks in the constructor and getter.
- Identify the SOLID violation in the subclass.
- Provide the corrected code snippets.

---ANSWER---

1. **Constructor Mutability Leak**: Assigning `this.tasks = tasks;` directly means the caller still holds a reference to the list. If the caller mutates the original list, the "immutable" `TaskBatch` changes. Must use a defensive copy (e.g., `new ArrayList<>(tasks)` or `List.copyOf(tasks)`).
2. **Getter Mutability Leak**: Returning `tasks` directly in `getTasks()` allows the caller to call `.add()` or `.remove()` on the internal list. Must return `Collections.unmodifiableList(tasks)` or use `List.copyOf` (which returns an unmodifiable list in modern Java).
3. **Liskov Substitution Principle (LSP) Violation**: `StrictTaskBatch` overrides `displayBatchInfo()` to throw an `UnsupportedOperationException`. A subclass must be substitutable for its base class. Code relying on `TaskBatch` will crash unexpectedly when passed a `StrictTaskBatch`.
4. **Fix Strategy**: Make `TaskBatch` final or seal it (`sealed class TaskBatch permits ...`) to tightly control inheritance. Extract the display logic into a separate interface (`Displayable`) if some batches shouldn't be displayed, rather than inheriting and disabling behavior.
