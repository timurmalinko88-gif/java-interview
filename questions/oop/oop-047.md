# The Identity Crisis
Review the following Java 21 code. The developer wanted to use `Employee` objects as keys in a `HashMap`, but they are reporting that `map.get(new Employee("E123", "Alice"))` returns `null` even after putting it in the map. Find and explain the bugs.

```java
import java.util.Objects;

public class Employee {
    private final String id;
    private final String name;

    public Employee(String id, String name) {
        this.id = id;
        this.name = name;
    }

    // Developer wrote this to check equality
    public boolean equals(Employee other) {
        if (this == other) return true;
        if (other == null) return false;
        return Objects.equals(id, other.id);
    }
}
```

---ANSWER---

### Life Analogy
Imagine trying to pick up your coat at a cloakroom. The attendant is trained to match "Any Item" (Object) with your ticket. But you attached a label that says "Only compare this to other Coats" (overloaded `equals`). The attendant gets confused by the strict instructions and assumes they don't match, so you never get your coat back.

### Code Review Findings
1. **Overloading instead of Overriding:** The `equals` method takes an `Employee` instead of `Object`. The `HashMap` calls `equals(Object)`, which defaults to `Object.class`'s reference equality.
2. **Missing `@Override`:** If the developer had used `@Override`, the compiler would have caught the wrong signature.
3. **Missing `hashCode()`:** Even if `equals` was correct, `hashCode()` is not overridden. `HashMap` uses the hash code to find the bucket. Two distinct instances will have different default hash codes, so the map won't even check `equals()`.

### Mandatory Evaluation Points
1. Identifies the wrong parameter type in `equals` (`Employee` instead of `Object`).
2. Explains the contract between `equals()` and `hashCode()`.
3. Identifies the missing `hashCode()` implementation.
