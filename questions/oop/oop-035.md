You are tasked with designing a Distributed Pricing Engine for a ride-sharing application (like Uber/Lyft). 
The engine must calculate the final price of a ride based on multiple dynamic factors: base fare, distance, time of day, traffic conditions, weather, and user loyalty tier.

How would you model the core pricing logic using Object-Oriented principles (Design Patterns) to ensure it is highly extensible? 
Furthermore, how do you map this OOP design into a distributed architecture where some factors (like weather) require calling external, potentially slow services?

---ANSWER---



- Proposes a clear behavioral pattern (Strategy, Chain of Responsibility, or Decorator) to avoid massive `if-else` blocks for pricing logic.
- Separates I/O (fetching data) from Domain Logic (calculating the price). Pure OOP functions should not make network calls.
- Discusses asynchronous data gathering (Scatter-Gather pattern) to handle latency.
- Mentions statelessness of the rule executors to ensure thread safety and horizontal scalability.

### Life Analogy
Designing a pricing engine is like a restaurant kitchen line. The dish (the price) passes through multiple stations (chefs). The base chef cooks the meat, the sauce chef adds flavor based on what's available (weather), and the expediter applies a VIP discount at the end. You want to be able to add a new chef (e.g., a "surge pricing" chef) without rewriting the recipe for the whole kitchen.

### Object-Oriented Design (Core Logic)
- **Pattern**: **Chain of Responsibility** combined with **Strategy**.
- **Model**:
  - `PriceContext`: An object holding the immutable ride request details and a mutable running total or a list of applied adjustments.
  - `PricingRule` (Interface): Exposes a method `void apply(PriceContext context)`.
  - Concrete Implementations: `BaseFareRule`, `SurgePricingRule`, `WeatherModifierRule`, `LoyaltyDiscountRule`.
- **Extensibility**: To add a new pricing factor, you simply implement a new `PricingRule` and inject it into the chain (Open/Closed Principle).

### Distributed Architecture & Trade-offs
- **Handling Slow External Calls**: The `WeatherModifierRule` needs external data. Making synchronous HTTP calls inside a domain object is an anti-pattern.
  - *Solution*: **Enrichment Phase before Execution**. The system should gather all context (fetch weather, fetch traffic) *asynchronously* (e.g., using `CompletableFuture` or reactive streams) *before* passing the hydrated `PriceContext` to the OOP rules engine.
- **Statelessness**: The pricing rules must be completely stateless singletons to allow horizontal scaling of the service.
- **Resilience**: If the weather service is down, the enrichment phase should fall back to a default strategy (e.g., no weather modifier) rather than failing the entire price calculation.
