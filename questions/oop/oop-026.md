# Java Records and the Illusion of Immutability
In Java 14+, `record` was introduced as a concise way to create immutable data carriers. However, a developer complains that a record they created is somehow having its state changed at runtime, violating encapsulation. 

Explain how a Java `record` can be mutated, why this happens, and how to properly enforce absolute immutability.

## Analogy
Think of a Java `record` like a glass safe. The safe itself (the record) cannot be altered or broken into. However, if you store a remote control (a mutable object) inside that glass safe, someone outside can still point a laser at it, press a button, and change the channel on your TV. The safe didn't change, but the state of what it controls did.

## Explanation

**The Core Nuance:**
Java records guarantee **shallow immutability**. The fields of a record are implicitly `final`, meaning the *reference* cannot be reassigned to point to a different object. However, if the field references a **mutable object** (like a `List`, `Date`, or a custom class), the internal state of that object can still be modified.

**How the Leak Happens:**
If you pass a mutable list into a record, the caller retains a reference to that list. If the caller adds an item to the list, the record reflects that change.

**The Fix:**
To enforce deep immutability, you must perform **defensive copying** in the compact constructor and ensure accessors return unmodifiable views.

### Example Fix
```java
public record ShoppingCart(List<String> items) {
    // Compact constructor for defensive copying
    public ShoppingCart {
        items = List.copyOf(items); // creates an unmodifiable deep copy
    }
}
```

## Mandatory Key Points for a Correct Answer
1. **Shallow vs. Deep Immutability:** Records only provide shallow immutability (final references).
2. **Mutable References:** Fields pointing to objects like arrays, `ArrayList`, or `java.util.Date` can be mutated by external code holding the reference.
3. **Defensive Copying:** The compact constructor must be used to defensively copy mutable inputs (e.g., `List.copyOf()`).
4. **Immutable Returns:** Any custom accessor methods for mutable types must also return defensive copies or unmodifiable wrappers.
