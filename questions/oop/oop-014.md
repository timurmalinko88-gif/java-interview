In modern Java (17+), how does the combination of `sealed` interfaces and Interface Segregation Principle (ISP) change domain modeling? Discuss the trade-offs of using `default` methods in a sealed interface versus traditional abstract classes.

---ANSWER---

1. **Exhaustiveness:** Sealed interfaces allow exhaustive `switch` expressions (pattern matching), eliminating the need for default/fallback branches.
2. **ISP Synergy:** You can have narrow, focused sealed interfaces rather than bloated ones, keeping implementations strictly controlled but lightweight.
3. **Default Methods vs Abstract Classes:** Default methods don't hold state (no instance fields), whereas abstract classes can. Using sealed interfaces with default methods allows multiple inheritance of behavior while strictly controlling the hierarchy.

* **Pattern Matching:** Explain how sealing guarantees exhaustiveness in domain modeling (Algebraic Data Types).
* **Trade-offs:** Compare abstract classes (stateful, single inheritance) with sealed interfaces + default methods (stateless, multiple inheritance, strict hierarchy).

### Life Analogy
Think of a **Sealed Interface** like an exclusive VIP club. Only specifically named members (permitted classes) can enter. **Interface Segregation** is like giving each member only the specific drink tickets they need, rather than an all-you-can-drink pass that half of them won't use.
