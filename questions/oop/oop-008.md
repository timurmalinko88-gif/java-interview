# Question: The Leaky Immutable Object

A junior developer was tasked with creating a strictly immutable `Configuration` object to pass safely between multiple threads. They wrote the code below. 

Despite declaring fields as `final`, instances of this class are having their internal state modified concurrently, causing unpredictable runtime behavior. 
Point out the vulnerabilities in this class and how to fix them to achieve true immutability.

```java
import java.util.Date;
import java.util.List;
import java.util.ArrayList;

public final class Configuration {
    private final String configName;
    private final Date createdAt;
    private final List<String> allowedEndpoints;

    public Configuration(String configName, Date createdAt, List<String> allowedEndpoints) {
        this.configName = configName;
        this.createdAt = createdAt;
        this.allowedEndpoints = allowedEndpoints;
    }

    public String getConfigName() {
        return configName;
    }

    public Date getCreatedAt() {
        return createdAt;
    }

    public List<String> getAllowedEndpoints() {
        return allowedEndpoints;
    }
}
```

---ANSWER---

- **Defensive Copying on Input:** The constructor accepts mutable objects (`Date`, `List`) and assigns them directly. The caller retains a reference and can mutate them externally.
- **Defensive Copying on Output:** The getters return the exact references to the internal mutable objects. Anyone calling `getAllowedEndpoints().add("malicious-url")` will mutate the internal state.
- **Alternative Types:** Acknowledge that `Date` is obsolete and `java.time.Instant` or `LocalDateTime` (which are inherently immutable) should be used instead.

### Bug 1: Leaking State via Getters
`getAllowedEndpoints()` returns the internal list reference. 
**Fix:** Return an unmodifiable view: `return Collections.unmodifiableList(allowedEndpoints);` or a deep copy. Similarly, `getCreatedAt()` should return `new Date(createdAt.getTime())`.

### Bug 2: Leaking State via Constructor
The constructor stores references passed from the caller. If the caller alters `allowedEndpoints` after creating the `Configuration`, the internal state changes.
**Fix:** Create a new copy of the data inside the constructor: `this.allowedEndpoints = new ArrayList<>(allowedEndpoints);` and `this.createdAt = new Date(createdAt.getTime());`.

### Modern Java Improvements
- Replace `java.util.Date` with `java.time.Instant` which is immutable.
- Use `List.copyOf(allowedEndpoints)` in the constructor (Java 10+), which creates an unmodifiable copy efficiently.
- Consider making this a `record` (Java 14+) combined with defensive copying in a compact constructor.

### Life Analogy
Passing references to mutable internal state is like handing a stranger the master key to your house instead of giving them a photocopy of the blueprint. You wanted them to just look at the design, but now they can walk in and rearrange your living room.
