# The Deceptive Processor
Review the following Java code managing order items. Identify the **three OOP and behavioral flaws**. Focus specifically on Encapsulation and the SOLID principles.

```java
import java.util.ArrayList;
import java.util.List;

public class OrderProcessor {
    private final List<String> items = new ArrayList<>();

    public void addItem(String item) {
        if (item != null) {
            items.add(item);
        }
    }

    public List<String> getItems() {
        return items;
    }
}

class VIPOrderProcessor extends OrderProcessor {
    @Override
    public void addItem(String item) {
        if (item.length() < 3) {
            throw new IllegalArgumentException("VIP items must be premium");
        }
        super.addItem(item);
    }
}
```

### Life Analogy
Imagine a **Fake Door**. It looks exactly like a real door (Inheritance), so you assume you can push it open (Base Class behavior). But when you push it, it shocks you (throws an Exception). If a subclass looks like its parent, it must behave predictably like its parent.

---ANSWER---

The code suffers from poor encapsulation and a severe violation of OOP design principles.

**Bug 1: Liskov Substitution Principle (LSP) Violation**
- `OrderProcessor.addItem` accepts any non-null string.
- `VIPOrderProcessor.addItem` strengthens the preconditions by throwing an `IllegalArgumentException` if the string is less than 3 characters.
- If a client relies on the base class abstraction (`OrderProcessor p = new VIPOrderProcessor();`), their program will crash unexpectedly when passing valid data like "PC". 

**Bug 2: Encapsulation Leak (Leaking Mutable State)**
- `getItems()` returns the internal reference to the mutable `items` list.
- A caller can do `processor.getItems().clear();` or `processor.getItems().add(null);`, entirely bypassing the validation logic in `addItem()`.
- **Fix:** Return an unmodifiable view: `return Collections.unmodifiableList(items);` or a defensive copy.

**Bug 3: NullPointerException in Subclass**
- The base class explicitly protects against `null` items (it just ignores them).
- The subclass calls `item.length()` *before* checking for null or deferring to `super.addItem()`.
- If `item` is null, `VIPOrderProcessor` throws a `NullPointerException`, completely changing the base class's null-handling contract.

### Key Mandatory Points for a Correct Answer
1. **LSP Violation:** Must explain that subclasses cannot strengthen preconditions (make rules stricter) without breaking the contract expected by users of the base class.
2. **Encapsulation / Defensive Copying:** Must point out that returning the raw `List` allows external modification of internal state.
3. **Null Handling mismatch:** Must identify the NPE risk introduced in the subclass that was safely handled in the parent class.
