Review the following `UserConfiguration` class. Point out the bugs related to object-oriented design and explain how to fix them.

```java
import java.util.List;
import java.util.ArrayList;

public class UserConfiguration {
    private String username;
    private List<String> permissions;

    public UserConfiguration(String username, List<String> permissions) {
        this.username = username;
        this.permissions = permissions;
    }

    public List<String> getPermissions() {
        return this.permissions;
    }
    
    public void setUsername(String username) {
        this.username = username;
    }
}
```

## Life Analogy
Imagine giving someone your home address (a copy of the data) versus giving them the physical master key to your house (a mutable reference). If you give the key, they can rearrange your furniture without asking. Encapsulation means giving out copies or read-only views, not the master key.

---ANSWER---

1. **Constructor Leak:** The constructor assigns the passed `List` reference directly, allowing the caller to modify the list from outside.
2. **Getter Leak:** `getPermissions()` returns the internal list reference, allowing external code to clear or add permissions without the class knowing.
3. **Fix:** Use defensive copying (e.g., `new ArrayList<>(permissions)`) in the constructor, and return an unmodifiable view (e.g., `Collections.unmodifiableList()`) or a copy in the getter.

---ANSWER---

* **Identify the Leak:** Point out that `List` is mutable and references are being shared.
* **Explain the Impact:** Explain how a malicious or careless caller could bypass business logic by altering the list directly.
* **Propose the Fix:** Show the updated constructor and getter using defensive copying.
