The Liskov Substitution Principle (LSP) states that objects of a superclass should be replaceable with objects of its subclasses without breaking the application. 

Beyond the classic "Square inheriting from Rectangle" example, how does Java enforce LSP at the compiler level regarding method signatures? Additionally, give an example of how you can still violate LSP in Java even if the code compiles perfectly.

---ANSWER---

### Life Analogy
LSP is like replacing a standard lightbulb with an LED bulb. As long as the base fits the socket and it runs on the same voltage (the contract), the room lights up. If the LED bulb requires a completely different dimmer switch to avoid catching fire, it violates the substitution principle.

### Compiler Enforcement
The Java compiler enforces LSP through method overriding rules:
1. **Contravariance of arguments**: You cannot make overridden method arguments more restrictive (Java actually requires exact type matches for overriding, otherwise it's overloading).
2. **Covariance of return types**: An overridden method can return a more specific subclass, but never a broader type.
3. **Exception Handling**: An overridden method cannot throw new or broader *checked* exceptions. It can only throw the same exceptions, narrower subclasses of them, or none at all.

### Logical Violations (Compiles but breaks LSP)
You can violate LSP via *behavioral* changes that the compiler cannot catch:
- **Preconditions**: A subclass method throwing an `IllegalArgumentException` for an input that the parent class accepts.
- **Postconditions**: A subclass failing to modify state in a way the parent class promised (e.g., an `InMemoryCache` returns data immediately, but an `AsyncDatabaseCache` returns null and fetches data in the background).
- **Invariants**: A subclass allowing a previously immutable field to be mutated.
