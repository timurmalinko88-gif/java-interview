# Question: The Rectangle-Square Dilemma

A junior developer submitted this code to represent shapes in a drawing application. 
Review the code. Identify any architectural or behavioral bugs, specifically regarding SOLID principles.

```java
class Rectangle {
    protected int width;
    protected int height;

    public void setWidth(int width) {
        this.width = width;
    }

    public void setHeight(int height) {
        this.height = height;
    }

    public int getArea() {
        return width * height;
    }
}

class Square extends Rectangle {
    @Override
    public void setWidth(int width) {
        this.width = width;
        this.height = width;
    }

    @Override
    public void setHeight(int height) {
        this.width = height;
        this.height = height;
    }
}

public class ShapeProcessor {
    public static void scaleWidth(Rectangle rect, int multiplier) {
        rect.setWidth(rect.width * multiplier);
    }

    public static void main(String[] args) {
        Rectangle rect = new Square();
        rect.setWidth(5);
        rect.setHeight(10);
        
        System.out.println("Expected area: 50, Actual: " + rect.getArea());
    }
}
```

---ANSWER---

1. Identifies the violation of the Liskov Substitution Principle (LSP).
2. Explains the behavioral bug: setting the height mutates the width, breaking the caller's expectations.
3. Identifies the poor encapsulation of `protected` fields.
4. Proposes a structural fix using interfaces (IS-A `Shape`) instead of concrete inheritance.
5. Mentions immutability as a modern solution to prevent state mutation bugs.

### Code Review Findings

1.  **Liskov Substitution Principle (LSP) Violation:** The `Square` class violates LSP. LSP states that objects of a superclass should be replaceable with objects of its subclasses without breaking the application. 
2.  **Contract Violation:** `ShapeProcessor.main` expects a `Rectangle` to behave like a rectangle (changing height shouldn't change width). But because it's actually a `Square`, `setHeight(10)` also sets the width to 10. The resulting area is 100, not 50.
3.  **Encapsulation Breach:** The fields `width` and `height` are `protected` and accessed directly in `ShapeProcessor.scaleWidth(rect.width * multiplier)`. This is bad practice; it bypasses getters.

### The Fix
Squares and Rectangles share properties, but structurally, a Square should not inherit from Rectangle if it mutates behavior in a way that breaks the Rectangle contract. 

Solution 1: Abstract common behavior into a `Shape` interface.
Solution 2: Make the shapes immutable (using Java Records) so mutation isn't an issue.

```java
interface Shape {
    int getArea();
}

record Rectangle(int width, int height) implements Shape {
    public int getArea() { return width * height; }
}

record Square(int side) implements Shape {
    public int getArea() { return side * side; }
}
```

### Life Analogy
It's like replacing a standard power outlet with a dimmer switch. They look similar and both provide electricity, but if you plug a refrigerator into the dimmer switch, it will malfunction because it violates the expected contract of providing constant voltage.
