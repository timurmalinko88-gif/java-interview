# The Liskov Contract Breach
Review the following E-commerce code. The architecture relies on polymorphism to process various types of payments. The system works perfectly with `CreditCardProcessor`, but occasionally crashes in production when `CryptoProcessor` is injected dynamically by the DI container.

Identify the object-oriented design violations and bugs in this code, and explain how to fix them.

```java
import java.util.List;

public class Order {
    private double amount;
    private List<String> items;
    private String status = "PENDING";
    
    public Order(double amount, List<String> items) {
        this.amount = amount;
        this.items = items;
    }
    
    public double getAmount() { return amount; }
    public List<String> getItems() { return items; }
    public void setStatus(String status) { this.status = status; }
}

public class PaymentProcessor {
    /**
     * Processes the order payment.
     * @param order the order to process
     */
    public void process(Order order) {
        // Base processing logic
        order.setStatus("PAID");
    }
}

public class CryptoProcessor extends PaymentProcessor {
    @Override
    public void process(Order order) {
        // BUG 1
        if (order.getAmount() < 100) {
            throw new IllegalArgumentException("Crypto payments require a $100 minimum.");
        }
        
        order.setStatus("PAID");
        
        // BUG 2
        order.getItems().clear(); // Free up memory after processing
    }
}
```

## Analogy
Substituting objects should be like substituting a standard lightbulb with an LED lightbulb. They both fit the same socket and provide light. If the LED bulb requires you to rewire your house before it turns on, or randomly melts the lampshade (unexpected side effect), it has violated the basic contract of being a "lightbulb."

## Explanation

**Bug 1: Liskov Substitution Principle (LSP) Violation**
The `CryptoProcessor` strengthens the preconditions of the base class. The base `PaymentProcessor.process()` contract accepts any valid `Order`. By throwing an unexpected `IllegalArgumentException` for amounts under 100, the subclass violates the expectations of the client code calling it. 
*Fix:* Design the system so validation rules are explicit in the interface (e.g., `boolean canProcess(Order o)`), or handle the limitation gracefully without breaking the polymorphic contract.

**Bug 2: Unintended Side Effects (Encapsulation/State Mutation)**
The `CryptoProcessor` clears the `items` list of the order. This is a severe side effect that the base class does not do. Client code expecting the order to remain intact after processing will fail or corrupt data.
*Fix:* Remove the `clear()` call. Furthermore, `Order.getItems()` should return an unmodifiable list (e.g., `Collections.unmodifiableList(items)`) to protect its internal state from external processors.

## Mandatory Key Points for a Correct Answer
1. **LSP Violation:** Explicitly name the Liskov Substitution Principle. Explain that subclasses cannot strengthen preconditions (throw new unexpected exceptions for standard inputs).
2. **Polymorphic Breakdown:** The client cannot safely substitute `PaymentProcessor` with `CryptoProcessor` without writing `instanceof` checks, which defeats the purpose of polymorphism.
3. **Side Effects & Encapsulation:** Point out that modifying the domain entity's internal list (`clear()`) is a dangerous side effect.
4. **Defensive Design:** Suggest returning an unmodifiable list from `Order` to strictly enforce encapsulation.
