# The Fragile Subclass
Review the following Java 21 code. The developer wanted to create a basic hierarchy for company personnel and allow them to be compared and stored in collections. 

Find and explain the **three bugs** related to OOP principles and Java core contracts.

```java
import java.util.Objects;

public class Employee {
    private String name;
    
    public Employee(String name) { 
        this.name = name; 
    }
    
    public String getName() { 
        return name; 
    }
    
    @Override
    public boolean equals(Object obj) {
        if (obj instanceof Employee) {
            return this.name.equals(((Employee) obj).name);
        }
        return false;
    }
}

class Manager extends Employee {
    private String department;
    
    public Manager(String name, String department) { 
        super(name); 
        this.department = department; 
    }
    
    @Override
    public boolean equals(Object obj) {
        if (obj instanceof Manager) {
            return super.equals(obj) && this.department.equals(((Manager) obj).department);
        }
        return false;
    }
}
```

---ANSWER---

The code contains critical flaws that will cause unpredictable behavior, especially when these objects are used in standard collections like `HashSet` or `HashMap`.

**Bug 1: Violation of the `equals()` Symmetry Contract**
- If we have `Employee e = new Employee("Alice");` and `Manager m = new Manager("Alice", "Sales");`.
- `e.equals(m)` evaluates to `true` (since `m` is an instance of `Employee` and names match).
- `m.equals(e)` evaluates to `false` (since `e` is not an instance of `Manager`).
- **Fix:** Use `getClass()` instead of `instanceof` for exact class matching, or make the `equals` method `final` in the base class.

**Bug 2: Missing `hashCode()` Implementation**
- Whenever `equals()` is overridden, `hashCode()` MUST be overridden. 
- Two objects that are equal according to `equals()` must have the same hash code. Without this, storing `Manager` objects in a `HashMap` will result in lost data or duplicate entries.
- **Fix:** Implement `@Override public int hashCode()` in both classes using `Objects.hash(...)`.

**Bug 3: NullPointerException Risk**
- In `Employee.equals()`, if `this.name` is null, `this.name.equals(...)` will throw a `NullPointerException`.
- **Fix:** Use `Objects.equals(this.name, other.name)` to handle potential nulls safely.

### Key Mandatory Points for a Correct Answer
1. **Symmetry violation:** Explicitly demonstrate how `e.equals(m) != m.equals(e)`.
2. **hashCode contract:** Mention that missing `hashCode` breaks Java collections (Hash-based).
3. **Null safety:** Point out the naive `.equals` call on a potentially null field.
