# Immutability Leaks in Java Records
The team is migrating older DTOs to Java Records to guarantee immutability. The following Java 21 record represents an employee's assigned access roles. During testing, developers noticed that the roles can still be modified after the `EmployeeProfile` object is created. 
Review the code, explain the vulnerability, and provide a fix.

```java
import java.util.List;
import java.util.ArrayList;

public record EmployeeProfile(String employeeId, List<String> roles) {
    
    // Custom constructor to validate input
    public EmployeeProfile {
        if (employeeId == null || employeeId.isBlank()) {
            throw new IllegalArgumentException("ID cannot be blank");
        }
        if (roles == null) {
            roles = new ArrayList<>();
        }
    }
    
    public boolean hasAdminAccess() {
        return roles.contains("ADMIN");
    }
}
```

---ANSWER---

#### 1. The Core Issue: Shallow Immutability
Java Records provide *shallow* immutability. The fields themselves are `final` (the reference cannot point to a new list), but the object the field points to (the `List<String>`) is still inherently mutable if it's a standard `ArrayList`. 

#### 2. The Bugs Identified
- **Leaking external reference in constructor:** If a caller passes an `ArrayList` to the constructor, they retain a reference to it. The caller can then add or remove roles from the list outside the record, mutating the record's internal state.
- **Leaking internal reference in getter:** Because records automatically generate accessor methods, calling `profile.roles()` returns the exact reference to the mutable list, allowing anyone to modify it (`profile.roles().add("ADMIN")`).

#### 3. The Fix
Use defensive copying in the compact constructor to ensure the record encapsulates an unmodifiable copy of the list.

```java
import java.util.List;

public record EmployeeProfile(String employeeId, List<String> roles) {
    public EmployeeProfile {
        if (employeeId == null || employeeId.isBlank()) {
            throw new IllegalArgumentException("ID cannot be blank");
        }
        // Defensive copy and unmodifiable wrapper
        roles = (roles == null) ? List.of() : List.copyOf(roles);
    }
    
    public boolean hasAdminAccess() {
        return roles.contains("ADMIN");
    }
}
```
*(Note: `List.copyOf()` returns an unmodifiable list and safely copies the elements, preventing both external and internal mutation leaks.)*

- [ ] Identify that the constructor accepts and stores a mutable reference directly.
- [ ] Identify that the auto-generated getter `roles()` exposes the mutable reference.
- [ ] Provide the fix using defensive copying, preferably via `List.copyOf()` or `Collections.unmodifiableList()`.

### Life Analogy
Making a Record is like locking your front door. But if your Record holds a mutable `List`, it's like leaving the back door wide open. You gave someone the actual keys to your house (the reference to the list) instead of a photocopy of your floor plan.

### Key Points
- [ ] Explain that Java Records only guarantee shallow immutability (fields are final, but referenced objects can be mutable).
