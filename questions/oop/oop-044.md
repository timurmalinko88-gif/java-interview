# Liskov Substitution Principle (LSP) in Practice
The Liskov Substitution Principle states that objects of a superclass should be replaceable with objects of its subclasses without affecting the correctness of the program. 
In the Java standard library, `java.util.Collections.unmodifiableList(...)` returns a `List` that throws an `UnsupportedOperationException` when `add()` is called. 
Does this violate the Liskov Substitution Principle? Defend your reasoning and discuss the trade-offs of this API design choice.

### Analogy
Imagine a universal remote control (the Interface). The remote has a "Record" button. If you buy a TV (the Subclass) that doesn't support recording, pressing the button does nothing or shows an error. Did the TV break the promise of the remote, or is the remote's design flawed by assuming all TVs can record?

---ANSWER---

#### 1. Does it violate LSP?
**Yes, strictly speaking.** The `List` interface contract (via `Collection`) defines the `add()` method. A consumer programming against the `List` interface expects to be able to add elements (or at least expects the post-condition that the collection contains the element, barring capacity limits). Throwing `UnsupportedOperationException` changes the expected behavior and breaks the contract.

#### 2. Why did Java design it this way?
It was a pragmatic trade-off. 
- **Interface Bloat:** If Java adhered strictly to LSP, they would have needed a massive hierarchy of interfaces: `ReadableList`, `MutableList`, `AppendOnlyList`, etc. 
- **Backwards Compatibility:** Adding immutable collections later in Java's life meant they had to reuse the existing `List` interface to remain compatible with older APIs.

#### 3. Modern Alternatives
Modern languages and libraries address this cleanly by separating concerns:
- **Kotlin:** Distinguishes between `List` (read-only) and `MutableList` (read-write) at the interface level.
- **Java Vavr/Guava:** Provide purely functional immutable collections that don't implement the mutable methods at all.
