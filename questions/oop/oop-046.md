# Designing a Bulletproof Immutable Class
Many beginners think making a class immutable just means marking fields `private` and removing setters. 
Explain what else is required to make a Java class truly immutable, especially when it contains mutable fields like `java.util.Date` or `List<String>`.

---ANSWER---

### Life Analogy
Imagine a museum displaying a rare painting (your class). Putting it behind glass (private fields) keeps people from touching it. But if you accidentally leave a backdoor open that lets the painter sneak in and alter the canvas (returning a direct reference to a mutable object), it's not truly protected. You must provide a *photocopy* (defensive copy) to anyone who asks for it.

### Key Points
- **Defensive Copying in Constructor:** When accepting mutable objects, clone or copy them before assigning to the field.
- **Defensive Copying in Getters:** Never return the original reference of a mutable field; return a copy.
- **Final Class & Fields:** Mark the class `final` to prevent subclassing (which could override methods and break immutability), and mark fields `final` to ensure they are assigned only once.

### Mandatory Evaluation Points
1. Mentions `final` class and `final` fields.
2. Identifies the danger of mutable fields (e.g., `Date`, `ArrayList`).
3. Explains defensive copying in constructors.
4. Explains defensive copying in getter methods.
