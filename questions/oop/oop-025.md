# OOP at Scale: Distributed Payment Processing
You are designing a distributed Payment Processing Pipeline. The system must handle different payment methods (Credit Card, Crypto, PayPal), each requiring vastly different third-party API flows. Furthermore, each payment transitions through several phases (Pending, Authorized, Captured, Failed).

How do you apply classical OOP design patterns (like Strategy and State) to this system while accounting for the trade-offs of a distributed microservices environment (e.g., node failures, network partitions)?

### Life Analogy
Think of a **Global Shipping Network**. 
- The *Strategy* is the routing instruction (Fly vs. Sail vs. Drive). In a local program, the manager just tells a worker what to do. In a distributed world, the instructions must travel *on the barcode of the box* so any facility knows how to handle it.
- The *State* is the tracking status. You can't just change the status in one warehouse's ledger; you must broadcast the update to a central tracking system so a crashed truck doesn't leave the package in limbo forever.

---ANSWER---

When migrating OOP patterns to a distributed architecture, memory references become network calls, and synchronous operations become asynchronous events.

**1. Strategy Pattern for Payment Methods**
- **Local OOP:** You inject a `PaymentStrategy` interface into a processor class.
- **Distributed:** Strategies become separate bounded contexts or microservices (e.g., `CryptoService`, `StripeService`).
- **Trade-off:** Instead of dynamic dispatch via an interface, you use an **Event Router / API Gateway**. A generic `PaymentOrchestrator` inspects the payment type and publishes an event to a specific Kafka topic (e.g., `process.crypto`). 
- *Crucial Distributed Aspect:* Because network calls fail, the "Strategy" execution must be wrapped in retries and require an **Idempotency Key** to ensure a retry doesn't double-charge the user.

**2. State Pattern for Payment Lifecycle**
- **Local OOP:** An object holds a `State` interface reference and changes its internal behavior based on whether it is `Pending` or `Captured`.
- **Distributed:** The State pattern evolves into the **Saga Pattern** or a central State Machine (like AWS Step Functions). 
- **Trade-off:** In memory, state changes are atomic. Over a network, a service might crash after charging the card but before updating the database.
- *Crucial Distributed Aspect:* State transitions must be **Event-Sourced** or tied to an Outbox Pattern. If moving from `Authorized` to `Failed`, a compensating transaction must be fired asynchronously.

### Key Mandatory Points for a Correct Answer
1. **Decoupling Strategy via Messaging:** Must explain that local interface dispatch becomes message routing (pub/sub or RPC) to specialized processor services.
2. **Idempotency:** A senior candidate must state that distributed "strategies" can execute multiple times (at-least-once delivery) and need idempotency keys.
3. **Distributed State Machine:** Must identify that state transitions cannot rely on local DB locks; they require distributed sagas, compensating transactions, or an event-driven orchestrator.
4. **Resilience / Partial Failures:** Must address how the system recovers if the node applying the strategy crashes mid-execution (e.g., dead-letter queues, timeouts).
