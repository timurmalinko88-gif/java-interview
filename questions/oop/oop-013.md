Review this caching implementation. Does it violate any core OOP principles? Identify any bugs and propose a better design.

```java
import java.util.HashMap;
import java.util.Map;

public class SimpleCache {
    protected Map<String, String> cache = new HashMap<>();

    public void put(String key, String value) {
        cache.put(key, value);
    }

    public String get(String key) {
        return cache.get(key);
    }
}

public class ReadOnlyCache extends SimpleCache {
    @Override
    public void put(String key, String value) {
        throw new UnsupportedOperationException("Cannot write to a ReadOnlyCache");
    }
}
```

## Life Analogy
Imagine ordering a decaf coffee, but instead of receiving coffee, the barista throws the mug at you (throws an Exception). If you advertise an interface (a coffee shop), you must honor its basic contract. You can't violently reject expected operations without breaking trust.

---ANSWER---

1. **LSP Violation:** `ReadOnlyCache` alters the expected behavior of `SimpleCache.put()` by throwing a runtime exception. Code expecting a `SimpleCache` will crash when given a `ReadOnlyCache`.
2. **State Exposure:** `protected Map` breaks encapsulation, allowing subclasses to mess with the internal state directly (and it's not thread-safe).
3. **Alternative Design:** Favor Composition or extract a common `Cache` interface with only a `get()` method, then have `WritableCache` extend it with `put()`.

---ANSWER---

* **Identify LSP:** Clearly state the Liskov Substitution Principle and how the `UnsupportedOperationException` violates it.
* **Identify Encapsulation/Concurrency Issues:** Mention the protected `HashMap` and lack of synchronization.
* **Refactor:** Provide a quick interface-based hierarchy (`ReadableCache` <- `WritableCache`) that safely models the domain.
