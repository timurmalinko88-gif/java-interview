# Question: Multi-Channel Notification Router

You need to design a Notification Service for an e-commerce platform. Currently, the system sends notifications via Email. Next sprint, they want to add SMS. In the future, they plan to add Push Notifications and Slack messages. 
Different channels have different requirements (e.g., SMS has a strict 160-character limit, Email requires an HTML template).

How would you design the object-oriented architecture for this system so that adding a new channel does not require changing the core routing logic? Detail the interfaces and patterns you would use.

## Life Analogy
Think of a multi-tool delivery network. The logistics manager (the Router) doesn't care if the package goes by Drone, Bike Courier, or Freight Truck. They just stamp "Deliver To X" and hand it to the specific transportation department (the Strategy). Each department independently figures out fuel, weight limits, and routes.

---ANSWER---

- **Strategy Pattern:** Defining a common `NotificationSender` interface that Email, SMS, and Push classes implement.
- **Open/Closed Principle:** The core router should be closed for modification (no massive `if-else` or `switch` statements for channel types) but open for extension (adding new sender implementations).
- **Factory/Registry Pattern:** Using a registry (often populated via Dependency Injection, like a Map of ChannelType to Strategy) to dynamically select the right sender at runtime.
- **Handling Constraints:** Managing channel-specific formatting (like SMS limits) cleanly without polluting the core interface.

---ANSWER---

### 1. Core Interfaces (Strategy)
Create a generic interface `NotificationSender`:
```java
public interface NotificationSender {
    void send(NotificationRequest request);
    ChannelType getChannelType();
}
```
Implementations (`EmailSender`, `SmsSender`) will handle their specific API integrations.

### 2. The Factory / Registry
Instead of a `switch` statement, use a registry to route requests:
```java
@Service
public class NotificationRouter {
    private final Map<ChannelType, NotificationSender> senders;

    public NotificationRouter(List<NotificationSender> senderList) {
        this.senders = senderList.stream()
            .collect(Collectors.toMap(NotificationSender::getChannelType, s -> s));
    }

    public void route(NotificationRequest request) {
        NotificationSender sender = senders.get(request.getChannel());
        if (sender == null) throw new UnsupportedChannelException();
        sender.send(request);
    }
}
```

### 3. Handling Channel Constraints (The Nuance)
How do we handle the 160-character SMS limit without forcing the Email sender to care about it? 
We should use a generic `NotificationRequest` payload, but allow each `NotificationSender` to utilize a specific `MessageFormatter` or `ContentAdapter` to extract and truncate/format what it needs before dispatching. Validation of channel-specific limits should reside in the concrete Strategy, keeping the Liskov Substitution Principle intact.
