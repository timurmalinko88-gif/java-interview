С выходом Java 15+ в языке появились `sealed` классы и интерфейсы. 
До их появления, если мы хотели ограничить иерархию наследования, мы могли сделать класс `final` (вообще запретить наследование) или сделать конструктор package-private (разрешить наследование только внутри текущего пакета).

Зачем потребовались `sealed` классы? Как они улучшают доменное моделирование (Domain-Driven Design) и безопасность кода в сочетании с новой конструкцией `switch` (Pattern Matching)?

---

---ANSWER---

`sealed` классы позволяют строго контролировать иерархию типов, явно указывая в контракте класса, кто имеет право от него наследоваться (ключевое слово `permits`).

**Ограничения старых подходов:**
- `final` запрещает наследование полностью. Мы не можем выразить идею "У класса X есть только наследники Y и Z".
- Package-private конструкторы позволяют наследование только в одном пакете, но это не выражено явно в API (контракте) класса. Плюс, вы физически вынуждены держать всех наследников в одной папке, что ломает структуру крупных проектов.

**Преимущества `sealed` классов:**
1. **Явное моделирование домена:** Мы можем описать алгебраические типы данных (Algebraic Data Types). Например: `public sealed interface PaymentResult permits Success, Failed, Pending {}`. Разработчик, читающий код, сразу видит все возможные исходы операции.
2. **Exhaustiveness (Исчерпываемость) в `switch`:** Это главная фича. В сочетании с *Pattern Matching for switch* (Java 21), компилятор знает **всех** возможных наследников `sealed` интерфейса. 

Если вы напишете:
```java
return switch (paymentResult) {
    case Success s -> "OK";
    case Failed f -> "Error";
    case Pending p -> "Wait";
};
```
Вам **не нужна** ветка `default`. Более того, если завтра кто-то добавит в `PaymentResult` новый статус `Cancelled`, проект просто не скомпилируется, пока вы не добавите обработку `Cancelled` во все `switch` по проекту. Это спасает от критических багов в рантайме.

### Life Analogy
Представьте клубную карту. 
- `final` класс — это именная карта, которой можете пользоваться только вы.
- Package-private конструктор — картой могут пользоваться люди, живущие с вами в одной квартире (но это не написано на самой карте).
- `sealed` класс — это карта, на которой сзади четко напечатан список лиц (permits), имеющих право ей воспользоваться (Вы, Супруга, Ребенок). Охранник на входе (компилятор) точно знает всех в лицо и никого больше не пустит.

### Key Points
- `sealed` классы/интерфейсы ограничивают список разрешенных наследников через `permits`.
- Наследники `sealed` класса должны быть объявлены как `final`, `sealed` или `non-sealed`.
- Они идеально подходят для моделирования бизнес-состояний и ограниченных наборов (похоже на Enum, но наследники могут иметь собственное состояние и разные поля).
- Компилятор обеспечивает exhaustiveness-проверку в конструкциях `switch` без блока `default`.
