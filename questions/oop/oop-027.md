# The Equality Trap in Inheritance
Review the following code. The developer intended to create an entity hierarchy where a `Manager` is a specialized `Employee`. They use these objects in a `HashSet` to ensure no duplicates. However, duplicates are appearing, and behavior is inconsistent.

Identify the bugs in this implementation and explain how to fix them.

```java
import java.util.Objects;

public class Employee {
    private String id;
    
    public Employee(String id) {
        this.id = id;
    }
    
    // BUG 1 hides here
    public boolean equals(Employee other) {
        if (other == null) return false;
        return Objects.equals(this.id, other.id);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(id);
    }
}

class Manager extends Employee {
    private String department;
    
    public Manager(String id, String department) {
        super(id);
        this.department = department;
    }
    
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Manager manager)) return false;
        // BUG 2 hides here
        return Objects.equals(department, manager.department);
    }
    
    @Override
    public int hashCode() {
        return Objects.hash(department); // BUG 3 hides here
    }
}
```

## Explanation

**Bug 1: Overloading instead of Overriding (`Employee.equals`)**
`Employee` defines `equals(Employee other)` instead of `equals(Object other)`. The `HashSet` calls `equals(Object)`. Because the method signature doesn't match the `Object` class contract, Java uses the default reference equality check for `Employee` objects.
*Fix:* Change the parameter to `Object` and add `@Override`.

**Bug 2: Ignoring Parent State (`Manager.equals`)**
The `Manager` class successfully overrides `equals`, but it only checks the `department` field. It completely ignores the `id` from the parent class. Two managers in the same department but with different IDs will be evaluated as equal.
*Fix:* Call `super.equals(o)` before evaluating the subclass fields.

**Bug 3: Breaking the HashCode Contract (`Manager.hashCode`)**
Similar to `equals`, the `hashCode` in `Manager` only hashes the `department`. It ignores the parent's state. Two different managers in the same department will cause constant hash collisions.
*Fix:* Include the parent's hash code, e.g., `Objects.hash(super.hashCode(), department)`.

2. **Super Implementation:** Subclasses must invoke `super.equals()` to ensure inherited state is included in the equality check.
3. **HashCode Contract:** `hashCode` must be updated to reflect the exact same fields used in `equals`, including the parent's state.

### Life Analogy
Checking identity without inheritance rules is like checking if two passports are identical by only looking at the expiration date, completely ignoring the photo and the name. Both layers of the identity card must match.

### Key Points
1. **Method Signature Match:** Point out that `equals(Employee)` is overloading, not overriding. The `@Override` annotation prevents this mistake.
