# The Pitfalls of Subclassing and Equality

## Context
A developer is implementing a 2D geometry library. They created a `Point` class and then extended it to create a `ColorPoint` class. This code compiles successfully in Java 17/21.

## Code Review Snippet
```java
import java.util.Objects;
import java.util.HashSet;
import java.util.Set;

class Point {
    private final int x;
    private final int y;

    public Point(int x, int y) {
        this.x = x;
        this.y = y;
    }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Point)) return false;
        Point p = (Point) o;
        return p.x == this.x && p.y == this.y;
    }

    @Override
    public int hashCode() {
        return Objects.hash(x, y);
    }
}

class ColorPoint extends Point {
    private final String color;

    public ColorPoint(int x, int y, String color) {
        super(x, y);
        this.color = color;
    }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof ColorPoint)) return false;
        ColorPoint cp = (ColorPoint) o;
        return super.equals(o) && cp.color.equals(this.color);
    }
    
    // Developer note: Didn't override hashCode because super.hashCode() 
    // already hashes the coordinates, and that's good enough for distribution.
}
```
Review the above code. Identify the bugs, explain why they occur, and propose a solution to fix the design.

## Analogy
Comparing objects with inheritance is like comparing a standard Passport to a Passport with a Visa. If the standard Passport only checks ID numbers, it will say it's identical to the Visa Passport. But the Visa Passport checks both the ID and the Visa, so it rejects the standard one. The relationship isn't mutual!

## Expected Answer Structure
- Identify the **Symmetry Violation** in `equals()`.
- Identify the **hashCode Contract Violation**.
- Propose a structural fix (e.g., Composition over Inheritance).

---ANSWER---

1. **Symmetry Bug**: `Point.equals(ColorPoint)` can return `true`, but `ColorPoint.equals(Point)` will return `false`. This breaks the `equals` contract (`x.equals(y)` must equal `y.equals(x)`).
2. **HashCode Bug**: `ColorPoint` fails to override `hashCode`. If two `ColorPoint` objects have the same coordinates but different colors, they might be placed in the same hash bucket, degrading `HashMap`/`HashSet` performance, but more critically, breaking the rule that unequal objects should ideally have different hash codes (though strictly, equal objects MUST have the same hash code—if we fix `equals`, we MUST fix `hashCode`).
3. **The "instanceof" Trap**: Using `getClass()` instead of `instanceof` fixes symmetry but breaks Liskov Substitution Principle if we just want a simple subclass.
4. **Architectural Fix**: Use **Composition over Inheritance**. `ColorPoint` should *contain* a `Point` as a private field rather than extending it, providing its own distinct `equals` and `hashCode`.
