# Question: The Fragile Base Class

Review the following Java 17 code. The developer wanted to create a `HashSet` that keeps track of how many elements have been added to it over its lifetime. 

Spot the bugs and explain why this approach is flawed.

```java
import java.util.Collection;
import java.util.HashSet;

public class InstrumentedHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    public InstrumentedHashSet() {
        super();
    }

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);
    }

    public int getAddCount() {
        return addCount;
    }
    
    public static void main(String[] args) {
        InstrumentedHashSet<String> s = new InstrumentedHashSet<>();
        s.addAll(java.util.List.of("Snap", "Crackle", "Pop"));
        System.out.println("Elements added: " + s.getAddCount()); 
    }
}
```

---ANSWER---

### Life Analogy
It's like hiring a subcontractor to paint your house, but you didn't realize they subcontract the interior painting to the same company you just hired for the exterior. You end up paying twice for the same rooms!

### Code Review Findings

1.  **Double Counting Bug:** The output of the `main` method will be `6`, not `3`. The `HashSet.addAll` method internally calls the `add` method for each element. Since `addAll` is overridden to add the size, and `super.addAll` internally calls the overridden `add` method, `addCount` gets incremented twice per element.
2.  **Inheritance Violation (Coupling):** This is a classic example of why inheritance breaks encapsulation. The `InstrumentedHashSet` depends on the *implementation details* of `HashSet` (specifically, that `addAll` calls `add`). If a future Java release changes this internal behavior, our class will silently break.

### The Fix: Composition over Inheritance
Instead of extending `HashSet`, use composition (the Decorator pattern).

```java
import java.util.Collection;
import java.util.Set;

public class InstrumentedSet<E> {
    private final Set<E> s;
    private int addCount = 0;

    public InstrumentedSet(Set<E> s) {
        this.s = s;
    }

    public boolean add(E e) {
        addCount++;
        return s.add(e);
    }

    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return s.addAll(c);
    }

    public int getAddCount() {
        return addCount;
    }
}
```

### Key Points to Look For in an Answer
1. Identifies the double-counting bug in `addAll`.
2. Explains *why* the bug occurs (internal method dispatch of `HashSet`).
3. Highlights that inheritance breaks encapsulation because the subclass depends on the superclass's internal implementation.
4. Suggests Composition (Decorator pattern) as the correct architectural solution.
