На код-ревью вам прислали реализацию неизменяемой коллекции (Immutable Collection). Разработчик решил унаследоваться от стандартного `java.util.ArrayList`, чтобы переиспользовать его логику хранения, но переопределил методы изменения, чтобы защитить коллекцию.

Посмотрите на код. Какой базовый принцип объектно-ориентированного дизайна (из SOLID) здесь грубо нарушен? К каким багам в production это приведет? Предложите правильное архитектурное решение.

```java
import java.util.ArrayList;
import java.util.Collection;

public class ImmutableList<E> extends ArrayList<E> {
    
    public ImmutableList(Collection<? extends E> c) {
        super(c);
    }

    @Override
    public boolean add(E e) {
        throw new UnsupportedOperationException("Эта коллекция неизменяемая!");
    }

    @Override
    public boolean remove(Object o) {
        throw new UnsupportedOperationException("Эта коллекция неизменяемая!");
    }
    
    // ... переопределение других методов мутации ...
}
```

---

---ANSWER---

Этот код грубо нарушает **Принцип подстановки Барбары Лисков (Liskov Substitution Principle - LSP)**, букву "L" в SOLID.

Принцип гласит: объекты базового класса должны быть заменяемы объектами его наследников без изменения корректности программы. 

Класс `ArrayList` в своем контракте (имплементируя интерфейс `List`) обещает, что метод `add()` успешно добавит элемент. Если мы передадим экземпляр нашего `ImmutableList` в метод, который принимает `ArrayList` или `List` и попытается добавить туда элемент (например, метод фильтрации или агрегации), программа упадет в рантайме с `UnsupportedOperationException`. Это обман ожиданий вызывающего кода.

*(Кстати, именно поэтому методы вроде `Collections.unmodifiableList()` в Java часто критикуют за дизайн, хотя они и стали стандартом де-факто до появления `List.of()` в Java 9).*

**Как исправить:**
Не использовать наследование (`extends`). Если мы хотим создать неизменяемую коллекцию, мы должны использовать **Композицию**. Класс не должен расширять `ArrayList`, он должен реализовать интерфейс `Iterable` (или создать собственный узкий интерфейс `ReadOnlyList`), а внутри себя инкапсулировать `ArrayList` или обычный массив, предоставляя наружу только методы чтения (`get`, `size`, `iterator`).

### Life Analogy
Представьте, что вы заказали такси (базовый класс). Контракт такси гласит: "откройте дверь, сядьте, мы поедем". Приезжает "Инкассаторская машина" (наследник такси, потому что тоже машина). Вы дергаете ручку, а она бьет вас током и кричит "UnsupportedOperationException: Двери заблокированы!". Инкассаторская машина нарушила ожидаемый контракт такси, хотя формально является автомобилем.

### Key Points
- Нарушение LSP приводит к непредсказуемому поведению и скрытым багам в рантайме (Runtime Exceptions).
- Выбрасывание `UnsupportedOperationException` в переопределенных методах — классический признак нарушения LSP.
- Наследование ради переиспользования кода (как в примере с ArrayList) — это антипаттерн. Для переиспользования нужно применять композицию.
- В современных версиях Java для неизменяемых коллекций следует использовать фабричные методы `List.of()`, которые возвращают скрытые реализации без публичного нарушения иерархии мутабельных списков.
