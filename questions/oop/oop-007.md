# Question: The Disappearing User

Review the following `User` class. A developer wrote this class intending to store `User` objects in a `HashSet` to quickly deduplicate them based on their email. 

However, when they add duplicate users to the set, the duplicates are not being filtered out. Furthermore, sometimes a user cannot be found in the set even though they were just added. Identify the bugs in this code.

```java
import java.util.Objects;
import java.util.Set;
import java.util.HashSet;

public class User {
    private String email;
    private String status;

    public User(String email, String status) {
        this.email = email;
        this.status = status;
    }

    public void setStatus(String status) {
        this.status = status;
    }

    // Attempting to override equality
    public boolean equals(User other) {
        if (this == other) return true;
        if (other == null) return false;
        return Objects.equals(email, other.email);
    }

    @Override
    public int hashCode() {
        return Objects.hash(email, status);
    }
    
    public static void main(String[] args) {
        Set<User> users = new HashSet<>();
        User u1 = new User("test@example.com", "ACTIVE");
        User u2 = new User("test@example.com", "ACTIVE");
        
        users.add(u1);
        users.add(u2);
        
        System.out.println("Set size: " + users.size()); 
        
        u1.setStatus("INACTIVE");
        System.out.println("Contains u1? " + users.contains(u1)); 
    }
}
```

---ANSWER---

- **Overloading vs. Overriding:** The `equals` method takes a `User` instead of an `Object`. It doesn't override `Object.equals(Object)`, so `HashSet` uses reference equality.
- **Missing `@Override`:** Mention that `@Override` would have caught the `equals` signature bug at compile time.
- **Mutable Fields in HashCode:** The `hashCode` uses `status`, which is mutable. Changing the status after insertion changes the hash bucket, effectively losing the object in the `HashSet`.

### Bug 1: Incorrect `equals` Signature
The method signature is `public boolean equals(User other)`. The `HashSet` calls `equals(Object)`. Because the signatures don't match, this is *method overloading*, not *overriding*. 
**Fix:** Change to `public boolean equals(Object other)`, cast to `User` after an `instanceof` check. Always use the `@Override` annotation to let the compiler catch this.

### Bug 2: HashCode and Equality Mismatch
The `equals` method only checks the `email` field, but the `hashCode` method hashes both `email` and `status`. The contract between `equals` and `hashCode` mandates that if two objects are equal, their hash codes must be identical. By including `status` in the hash but not in equality, this contract is broken.

### Bug 3: Mutable HashCode
The `status` field is used in `hashCode()`, but it can be mutated via `setStatus()`. If the object is mutated after being placed in a hash-based collection, its hash code changes, but it remains in the original bucket. Subsequent lookups (like `contains`) will fail. 
**Fix:** Remove `status` from the `hashCode` calculation, or make `status` immutable if it truly defines the object's identity.

### Life Analogy
Using a mutable field in `hashCode` is like using a person's hair color for their permanent passport ID. If they dye their hair (mutate the field), the border agents can no longer find their record in the system. 
Overloading instead of overriding `equals` is like a bouncer checking the VIP list but looking at a completely different book because you didn't ask using the exact required phrase.
