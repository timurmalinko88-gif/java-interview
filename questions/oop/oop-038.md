# Default Methods vs Abstract Classes in Modern Java
Before Java 8, the rule was simple: interfaces defined contracts (no behavior), and abstract classes provided partial implementations. With the introduction of `default` methods in interfaces, and later private methods in interfaces (Java 9), this line has blurred. 

If you are designing a core domain model for a new application today, how do you decide whether to use an interface with `default` methods or an `abstract` class to share common behavior? Furthermore, how does Java resolve the "Diamond Problem" if a class implements two interfaces with the same default method signature?

## Analogy
Think of an **Abstract Class** as a biological family tree: a child shares DNA (state) and inherent traits (methods) with their parent, but can only have one biological father (single inheritance). 
Think of an **Interface with default methods** as a job certification (e.g., First Aid, Forklift Operator). You can hold many certifications (multiple inheritance of behavior), and they come with standard operating procedures (default methods), but they don't give you your physical DNA (state).

## Expected Answer Structure
- Contrast **state** handling in both constructs.
- Discuss **inheritance limits** (single vs. multiple).
- Explain the **Diamond Problem resolution** rules in Java.
- Provide a clear heuristic for when to choose which.

---ANSWER---

1. **State is the Decider**: Abstract classes can have instance state (non-static, non-final fields) and constructors. Interfaces cannot hold instance state (all fields are implicitly `public static final`).
2. **Inheritance Limits**: A class can implement multiple interfaces but extend only one abstract class. Default methods allow multiple inheritance of *behavior*, but not *state*.
3. **Diamond Problem Resolution**: If a class implements two interfaces that provide the same default method, the compiler forces the class to override the method and explicitly choose which implementation to use (e.g., `InterfaceA.super.method()`), preventing ambiguity.
4. **Design Heuristic**: Use interfaces for defining capabilities/roles (e.g., `Flyable`, `Auditable`) and default methods for convenience/backward compatibility. Use abstract classes when establishing a strong "is-a" identity that fundamentally shares mutable state or requires complex construction logic.
