# Walk Me Through Diagnosing an `OutOfMemoryError` Using a Heap Dump

## The Analogy

A heap dump is like a **freeze-dried snapshot of your warehouse at the moment it ran out of floor space**. Every box (object) is labeled with its type and size. Every forklift path (reference chain) is frozen in place. Your job as the analyst is to find which shelf (GC root) is hoarding boxes that should have been shipped out long ago.

---

## What a Good Answer Covers

### 1. Capturing the dump (before or at crash)

```bash
# Option A: automatic dump on OOM (always enable in production)
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/var/dumps/

# Option B: manual dump of a running process
jmap -dump:live,format=b,file=heap.hprof <pid>

# Option C: via jcmd (preferred, lower risk)
jcmd <pid> GC.heap_dump /var/dumps/heap.hprof
```

> **Tip:** The `live` option in `jmap` forces a full GC first, which removes unreachable objects — this makes the dump smaller and focuses analysis on truly retained objects.

### 2. Opening the dump — tool choice

| Tool | Strength |
|------|----------|
| **Eclipse MAT** | Best for large dumps; "Leak Suspects" report; OQL query |
| **VisualVM** | Simpler UI; good for medium dumps |
| **JDK Mission Control (JMC)** | Integrates with JFR events |
| **IntelliJ Profiler** | Convenient for dev environments |

### 3. First-look workflow in Eclipse MAT

**Step 1 — Run "Leak Suspects" report**
MAT automatically groups retained heap by accumulation point. It highlights the single object (or class) that retains the most bytes transitively.

**Step 2 — Check the Dominator Tree**
The Dominator Tree shows which objects "own" the most retained heap. If one `HashMap` dominates 800 MB out of 1 GB, that's your suspect.

```
Example Dominator Tree output:
  java.util.HashMap @ 0x7f... — retained 823 MB (82%)
    └── Entry[] @ 0x7f...
          └── com.acme.CacheEntry (×4,200,000 instances)
```

**Step 3 — Find the GC root path**
Right-click the suspect → "Path to GC Roots" → exclude weak/soft/phantom references.
This reveals the exact reference chain from a GC root (static field, thread stack, JNI) to the leaking object.

```
GC Root → Thread "main"
  → com.acme.RequestContext.cache (static field)
    → java.util.HashMap
      → CacheEntry × 4.2M
```

**Step 4 — Use OQL for targeted queries**

```sql
-- Find all byte[] larger than 1 MB
SELECT * FROM byte[] b WHERE b.@retainedHeapSize > 1048576

-- Count instances per class
SELECT toString(classof(s).name), count(*) 
FROM java.lang.Object s 
GROUP BY classof(s).name
```

### 4. Interpreting common OOM patterns

| Symptom in dump | Likely cause |
|-----------------|--------------|
| Millions of `char[]` or `byte[]` backing `String` | Unbounded string cache or message accumulation |
| Large `Entry[]` in `HashMap`/`ConcurrentHashMap` | Unbounded cache without eviction |
| Many instances of a domain class | Batch job loading too much data into memory at once |
| `ClassLoader` instances with thousands of classes | Metaspace leak (check non-heap separately) |
| `Finalizer` queue backed up | Objects with `finalize()` overrides; finalizer thread can't keep up |

### 5. After analysis — validate the fix

```bash
# Before fix: confirm leak is contained to a specific class
# After deploying fix: re-capture dump under the same load
# Check that retained heap of the suspect class no longer grows

# Also monitor live with:
jstat -gcutil <pid> 5000   # GC activity every 5 seconds
jcmd <pid> GC.heap_info    # current heap regions
```

## Key Points (Mandatory for Full Credit)

1. **Always enable `-XX:+HeapDumpOnOutOfMemoryError` in production** — post-mortem dumps are your only evidence after a crash.
2. **"Retained heap" is the key metric**, not "shallow heap" — retained heap = everything that would be freed if this object were deleted.
3. **Dominator Tree + Path to GC Roots** is the canonical two-step to find both *what* leaks and *why* it is still alive.
4. **The `live` flag on `jmap` reduces noise** by forcing a GC before dumping, but it pauses the JVM — use `jcmd` in production to reduce risk.
5. **OQL lets you write targeted queries** when visual browsing is insufficient for large or complex heaps.
