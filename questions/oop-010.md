# Question: Resilient Payment State Machine

You are designing a centralized Payment Gateway that routes transactions to 3rd-party providers (Stripe, PayPal, Adyen). 
In a distributed environment, network partitions happen, APIs timeout, and asynchronous webhooks are delayed. 

We need to design a system that:
1. Applies OOP principles (specifically State pattern and Polymorphism) to handle the complex lifecycle of a payment (Pending, Authorized, Captured, Failed, Refunded).
2. Prevents the "Anemic Domain Model" anti-pattern—meaning the business rules for state transitions should not be scattered across helper services.
3. Gracefully handles idempotency and provider-specific failure nuances without violating the Liskov Substitution Principle (LSP).

Sketch the core class design and explain how you model the distributed state transitions.

## Life Analogy
Imagine a restaurant kitchen. The order ticket (Payment) goes through distinct stations: Prep, Grill, Plate (States). You can't plate a raw steak (invalid transition). The Head Chef (Gateway) manages the flow but delegates the actual cooking to specialized Sous-Chefs (Provider Adapters). If a Sous-Chef drops a pan (Timeout), the Head Chef has a standard protocol (Compensating Action) to recover, regardless of who dropped it.

## Key Points for a Correct Answer
- **State Pattern / DDD:** The `Payment` entity should control its own state transitions (e.g., `payment.capture()`) rather than a `PaymentService` modifying a dumb data object.
- **Liskov Substitution (LSP) in Distributed Failures:** All payment providers must conform to the same interface contract, even for errors. A specific `StripeException` leaking into the core router violates LSP. Providers must map their errors to standard domain exceptions.
- **Idempotency Handling:** Adapters must encapsulate idempotency keys to ensure retries of transitions (like `Authorize`) don't double-charge if a timeout occurs.
- **Anti-Corruption Layer:** Using interfaces to isolate the core Domain from the 3rd-party APIs.

## Answer Structure

### 1. Rich Domain Model (State Transitions)
Avoid setters like `payment.setStatus(CAPTURED)`. Instead, embed business rules in the domain entity:
```java
public class Payment {
    private PaymentState state;
    
    public void authorize() {
        if (!state.canTransitionTo(AUTHORIZED)) throw new InvalidStateException();
        this.state = AUTHORIZED;
    }
}
```
Use the **State Pattern** (often implemented via enums or classes in Java) to define valid transition graphs.

### 2. Abstracting the Providers (Anti-Corruption Layer)
Define a strictly enforced `PaymentProvider` interface. 
```java
public interface PaymentProvider {
    ProviderResult authorize(Payment context, IdempotencyKey key);
}
```
**The Nuance:** The interface must not leak HTTP concepts (like 503 statuses) or provider-specific SDK classes. The implementations (`StripeAdapter`) translate provider concepts into domain concepts (`ProviderResult.TIMEOUT`, `ProviderResult.DECLINED`).

### 3. Handling Distributed Timeouts (LSP)
If a timeout occurs, the system doesn't know if the money moved. 
The core orchestrator shouldn't have `if (provider instanceof PayPal)` logic for recovery. The interface should enforce a unified capability, such as a `reconcile(TransactionId)` method, which the abstract orchestrator calls via a background Cron job or asynchronous queue to fetch the definitive remote state and update the local State Machine.
