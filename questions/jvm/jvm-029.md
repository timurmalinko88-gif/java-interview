# Code Review: What Is Causing Mysterious Latency Spikes Unrelated to GC?

## Context

A high-throughput REST service processes ~50,000 requests/second. GC logs show only short minor GCs (<5 ms). Yet p99 latency occasionally spikes to 2–3 seconds. The team rules out network and database — the spikes happen even on trivial endpoints that touch no external system. A senior engineer mutters: "It smells like a safepoint problem." Review the code below and identify what forces unexpectedly long JVM safepoints.

## Code Under Review

```java
import java.util.concurrent.atomic.AtomicLong;

public class MetricsAggregator {

    private static final int BUFFER_SIZE = 10_000_000; // 10 million slots
    private final long[] latencyBuckets = new long[BUFFER_SIZE]; // (A)
    private final AtomicLong writeIndex = new AtomicLong(0);

    /** Called on every request completion — extremely hot path */
    public void record(long latencyNanos) {
        int slot = (int)(writeIndex.getAndIncrement() % BUFFER_SIZE);
        latencyBuckets[slot] = latencyNanos;
    }

    /**
     * Called by a background reporter thread every 10 seconds.
     * Bug area: this loop is the safepoint problem.
     */
    public double computeAverage() {                              // (B)
        long sum = 0;
        for (int i = 0; i < BUFFER_SIZE; i++) {                  // (C)
            sum += latencyBuckets[i];
        }
        return (double) sum / BUFFER_SIZE;
    }

    // ---- Separate subsystem ----

    /**
     * Periodically counts active sessions.
     * Bug area 2: counted loop on a large collection.
     */
    public static long countActive(int[] sessionFlags) {          // (D)
        long count = 0;
        for (int i = 0; i < sessionFlags.length; i++) {          // (E)
            if (sessionFlags[i] == 1) count++;
        }
        return count;
    }
}
```

## Bugs / Issues to Find

### Bug 1 — Counted `int`-indexed loop over a huge primitive array blocks safepoint polls (lines C, E)

The JVM inserts **safepoint poll checks** at loop back-edges — but historically (and still in many JIT-compiled loop shapes), **counted loops with an `int` index over a primitive array** are compiled by the C2 JIT into a tight loop **without safepoint polls** when the loop body is simple enough for auto-vectorization.

This means:
- The `computeAverage()` loop iterates **10 million times** without a single safepoint check.
- If a GC (or any other safepoint-required operation) is requested while this loop is running, **every other thread waits** at their nearest safepoint poll — potentially for hundreds of milliseconds until the 10M-iteration loop finishes.

```
GC log (with -XX:+PrintSafepointStatistics):
vmop         [threads: total initially_running wait_to_block]
G1EvacuatePause  [312   1  308]   // 308 threads waiting for 1 thread
              [time: spin=1847ms sync=1850ms ... ]
```

The 1847 ms "spin" is other threads waiting for the reporter thread to exit its tight loop.

**Fix A — Use `long` index to break the counted-loop optimization:**

```java
// A long index prevents the C2 "counted loop" vectorization that omits safepoint polls
public double computeAverage() {
    long sum = 0;
    for (long i = 0; i < BUFFER_SIZE; i++) {   // long index = safepoint polls restored
        sum += latencyBuckets[(int) i];
    }
    return (double) sum / BUFFER_SIZE;
}
```

**Fix B — Add an explicit safepoint via a JVM-cooperative construct:**

Breaking the loop into chunks with a `Thread.yield()` or `Thread.sleep(0)` at intervals forces a safepoint check:

```java
public double computeAverage() throws InterruptedException {
    long sum = 0;
    for (int i = 0; i < BUFFER_SIZE; i++) {
        sum += latencyBuckets[i];
        if ((i & 0xFFFF) == 0) Thread.onSpinWait(); // hint; or yield() for hard poll
    }
    return (double) sum / BUFFER_SIZE;
}
```

**Fix C (architectural) — Reduce the loop's scope:**

Process only the last N records actually written rather than all 10 M slots.

### Bug 2 — `sessionFlags` array size is unbounded (line D/E)

Same root cause: if `sessionFlags.length` is also in the millions, the same safepoint-omission applies. Additionally, the array is passed by reference with no size contract, so callers may pass arbitrarily large arrays.

**Fix:** Document and enforce a maximum size, or use `Arrays.stream(sessionFlags).filter(f -> f == 1).count()` which uses the stream infrastructure and cooperates with safepoints naturally (though with boxing overhead trade-off).

## How to Diagnose Safepoint Issues

```bash
# Enable safepoint statistics (Java 11+)
-Xlog:safepoint*=debug:file=safepoint.log

# Java 8
-XX:+PrintGCApplicationStoppedTime
-XX:+PrintSafepointStatistics
-XX:PrintSafepointStatisticsCount=1

# JFR (preferred, low overhead)
jcmd <pid> JFR.start duration=60s filename=recording.jfr settings=profile
# Then look for: jdk.SafepointBegin, jdk.SafepointStateSynchronization events
```

In JFR output, look for `jdk.SafepointStateSynchronization` events where `timeToSpin` is large — that identifies the thread holding everyone up.

---ANSWER---

1. **Safepoints are not just GC** — deoptimization, biased lock revocation, class redefinition, and heap dumps also require all threads to reach a safepoint.
2. **C2 JIT may omit safepoint polls in counted `int`-indexed loops** over primitive arrays — a loop iterating millions of times with no poll is a "safepoint black hole."
3. **Using a `long` loop index** is the simplest fix; it breaks the counted-loop pattern and forces C2 to re-insert polls.
4. **Diagnosis tool:** `-Xlog:safepoint*=debug` (Java 11+) or JFR `SafepointStateSynchronization` events reveal time-to-spin and the guilty thread.
5. **Latency spikes with clean GC logs** should immediately raise suspicion of safepoint delays, not just garbage collection.

### Life Analogy

A JVM safepoint is like a **fire drill in an office building**: every thread must stop what it's doing and walk to a marked spot before the drill (GC, deoptimization, etc.) can begin. If one employee is in the middle of a long, uninterruptible task — like photocopying 100,000 pages — everyone else waits at the assembly point until that one employee finishes and joins them. The drill can't start until the very last person arrives.
