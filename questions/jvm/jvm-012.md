# GC Algorithm Trade-offs

## Context
Your team recently upgraded a legacy reporting application from Java 8 to Java 17. The application runs heavy batch processing jobs at night and serves a web UI during the day. 

After the upgrade, the team noticed two things:
1. The web UI is much more responsive and users report fewer "freezes".
2. The nightly batch jobs, which used to take 2 hours, now take 2 hours and 40 minutes.

No code changes were made, only the Java version was updated.
Explain why updating the Java version caused this specific change in performance. What JVM defaults changed, and how do they explain both the UI improvement and the batch job degradation?

---ANSWER---

- **Parallel GC:** Closing the entire office every Friday for 4 hours to do a deep clean. Nobody can work during this time, but the overall cleaning is highly efficient (high throughput).
- **G1GC:** Hiring a janitor who quietly cleans rooms as people work, occasionally asking someone to step aside for a minute. The office never fully closes (low latency), but people are slightly distracted, meaning their overall daily output drops a bit (lower throughput).

### 2. The Core Explanation
- **Default GC Change:** In Java 8, the default Garbage Collector was the **Parallel GC** (Throughput Collector). In Java 9 and above (including 17), the default is **G1GC** (Garbage-First Collector).
- **Impact on the UI (Latency):** G1GC is designed to minimize "Stop-The-World" (STW) pauses. It works concurrently to clean up the heap, which prevents the web UI from freezing for long periods. Hence, the UI feels more responsive.
- **Impact on Batch Jobs (Throughput):** Parallel GC pauses all application threads to collect garbage but utilizes all CPU cores strictly for GC, making the actual cleanup incredibly fast and maximizing overall application throughput. G1GC steals CPU cycles from the application threads to do concurrent background work. For a heavy batch job, this overhead slows down the total processing time.

2.  **Define G1GC's Goal:** Must explain that G1GC prioritizes **low latency** (shorter pauses).
3.  **Define Parallel GC's Goal:** Must explain that Parallel GC prioritizes **high throughput** (overall execution speed).
4.  **Map to the Scenario:** Connect the UI improvement to low latency, and the batch degradation to reduced throughput.

### 4. Follow-up / Solution
To fix the batch job, you could configure the JVM to use the Parallel GC explicitly for the batch processing environment by passing the flag: `-XX:+UseParallelGC`.
