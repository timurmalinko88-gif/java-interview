# Where does a String live — String Pool or Heap, and why does it matter?
Explain the difference between a `String` created with a literal (`"hello"`) and one created with `new String("hello")`. Where does each live in memory? What happens when you call `.intern()` on the `new`-created instance?

As a bonus: what changed about the String Pool's physical location between Java 7 and Java 8+?

---ANSWER---

1. **String literals are automatically interned into the String Pool**  
   When the compiler sees `"hello"`, it looks up the pool; if the string already exists, the same reference is returned. No duplicate object is created.

2. **`new String("hello")` always creates a new heap object**  
   Forces creation of a brand-new `String` on the heap, even if an identical string already lives in the pool.

3. **`==` vs `.equals()` behavior differs**  
   Pool strings share references, so `"hello" == "hello"` is `true`, but `new String("hello") == new String("hello")` is `false`. Always use `.equals()` for content comparison.

4. **`.intern()` returns the canonical pool reference**  
   Calling `str.intern()` on a heap `String` looks up (or inserts) the value in the pool and returns the pooled reference.

5. **Pool moved from PermGen to Heap in Java 7**  
   Before Java 7, the String Pool lived in **PermGen** (fixed size, prone to `OutOfMemoryError: PermGen space`). From Java 7 onward it was moved to the **main Heap**, allowing the pool to grow dynamically and be garbage-collected.

### Memory layout

| Creation style | Location | GC-eligible? | Reference equality with same literal |
|---|---|---|---|
| `"hello"` (literal) | String Pool (Heap since Java 7) | Yes (if no live refs) | `true` |
| `new String("hello")` | Regular Heap | Yes | `false` |
| `new String("hello").intern()` | Pool (returns pool ref) | Yes | `true` |

### Code illustration

```java
String a = "hello";
String b = "hello";
String c = new String("hello");
String d = c.intern();

System.out.println(a == b);       // true  — same pool reference
System.out.println(a == c);       // false — c is a fresh heap object
System.out.println(a == d);       // true  — intern() returned pool ref
System.out.println(a.equals(c));  // true  — same content
```

### Why it matters in practice

- **Memory efficiency**: Heavy use of `new String(...)` in loops can create thousands of duplicate heap objects.
- **Security**: `String` immutability + pooling makes it safe as a `HashMap` key and in class loading.
- **PermGen vs Metaspace**: Java 8 removed PermGen entirely, eliminating pool-related OOM errors.

- Assuming interned strings are immortal — the pool **can** be garbage-collected when no live references remain.
- Believing `intern()` is always a performance win — on very large pools it can cause hash-table contention.

### Life Analogy

Think of the **String Pool as a hotel's message board** where identical notes are pinned once and shared by everyone who needs them. Using `new String(...)` is like writing the same note on your private notepad — it contains the same text but is a separate physical copy nobody else shares.

### Key Points

- Using `==` for string comparison in general — it only works for literals due to pooling coincidence.
