# Zero-Latency Bidding Engine

## Context
You are tasked with designing the core memory architecture for a Real-Time Bidding (RTB) advertising engine. 

**Requirements & Constraints:**
- Throughput: 100,000 requests per second.
- Latency SLA: Every request must be processed in under 15 milliseconds.
- State: The system must hold about 50GB of hot campaign data (budgets, user targeting profiles) in memory to evaluate bids.
- Current Architecture: A single Java monolith with a 64GB Heap using G1GC. 

**The Problem:**
The current system routinely breaches the 15ms SLA because Garbage Collection "Stop-The-World" pauses occasionally take 200-500ms, causing massive timeouts and lost revenue.
As a Senior Engineer, how would you redesign the memory management and system architecture to meet the strict 15ms SLA while handling 50GB of state? Discuss at least three different approaches, weighing the trade-offs of each.

---ANSWER---

### Life Analogy
**The Mega-Warehouse vs Distribution Centers:** Storing 50GB in one JVM heap is like having one massive mega-warehouse. Doing inventory (Garbage Collection) requires shutting down the whole warehouse, losing business. You can solve this by: 
1. Using magic shelves that self-inventory (ZGC/Shenandoah).
2. Moving inventory to an adjacent building the managers don't track (Off-heap).
3. Splitting into 10 smaller local distribution centers (Sharding).

### 2. Approach 1: Modern Low-Latency Garbage Collectors (ZGC / Shenandoah)
Instead of redesigning the architecture, upgrade the JVM and switch to ZGC (Java 15+) or Shenandoah. 
- **Pros:** Zero code changes required. ZGC guarantees sub-millisecond GC pauses regardless of heap size. It performs GC concurrently with application threads.
- **Cons:** Throughput penalty. Concurrent GC steals CPU cycles from the application. It also requires a larger heap headroom (e.g., you might need 100GB of RAM to comfortably manage a 50GB live set to prevent allocation stalls).

### 3. Approach 2: Off-Heap Memory (ByteBuffer / Chronicle Map)
Move the 50GB of state out of the JVM heap and into native OS memory using Java NIO `DirectByteBuffer` or libraries like Chronicle Map.
- **Pros:** The JVM GC only sees a few small pointer objects. A 50GB cache becomes invisible to G1GC, completely eliminating STW pauses. Extremely fast (microsecond latency).
- **Cons:** High development complexity. You must manually serialize/deserialize Java objects to bytes. You must manually manage memory (preventing memory leaks natively). 

### 4. Approach 3: Distributed In-Memory Grid (Redis / Hazelcast)
Remove the state from the application entirely and put it in a dedicated caching layer like Redis or an Ignite cluster.
- **Pros:** Stateless JVMs. You can scale the application nodes infinitely. The JVM heap stays tiny.
- **Cons:** Network I/O latency. A network hop to Redis takes 0.5ms - 2ms. If processing a bid requires 10 cache lookups, network latency will breach the 15ms SLA. Requires data modeling changes (e.g., Lua scripts in Redis) to minimize round-trips.

### Key Points
1.  **Acknowledge Large Heap Physics:** Must state that large JVM heaps naturally suffer from longer STW pauses under traditional collectors like G1GC.
2.  **Evaluate ZGC/Shenandoah:** Must mention modern concurrent collectors as a viable, low-effort solution, but identify the CPU/throughput tradeoff.
3.  **Evaluate Off-Heap:** Must understand that off-heap memory escapes GC management but requires manual serialization overhead.
4.  **Evaluate Distributed Cache:** Must identify network latency as the primary risk when moving state to an external system like Redis.
5.  **Final Recommendation:** A strong candidate will likely propose **ZGC** first for lowest engineering cost, falling back to **Off-heap (Chronicle Map)** if the throughput penalty of ZGC is unacceptable.
