# The Hidden Handle Leak
Review the following Java 21 method. It processes a massive log file line by line to count how many times a specific error code appears. The code compiles and works for small files, but causes unpredictable application crashes and high off-heap memory usage when run frequently on large files. 

Identify the bugs and explain how to fix them.

```java
import java.nio.file.Files;
import java.nio.file.Path;
import java.io.IOException;

public class LogAnalyzer {
    
    public long countErrorOccurrences(Path logFilePath, String errorCode) throws IOException {
        long count = 0;
        
        // Count matching lines
        count = Files.lines(logFilePath)
                     .filter(line -> line.contains(errorCode))
                     .count();
                     
        return count;
    }
}
```

---ANSWER---

### 1. The Core Bug (Resource Leak)
The `Files.lines()` method returns a `Stream<String>` that encapsulates an underlying I/O resource (a file channel/reader). Unlike standard collection streams (like `List.stream()`), I/O streams must be explicitly closed to release the file handle back to the operating system.

### 2. JVM & OS Impact
If the file handle is not closed:
* The OS eventually runs out of file descriptors (`Too many open files` exception).
* The underlying native memory buffers associated with the open file might not be reclaimed immediately by the GC, causing indirect memory pressure.
* The GC *might* eventually close it via a cleaner/finalizer when the Stream object is collected, but relying on GC for OS resource management is a critical anti-pattern.

### 3. The Fix
Wrap the Stream in a `try-with-resources` block so that `close()` is automatically invoked regardless of success or exceptions.

```java
public long countErrorOccurrences(Path logFilePath, String errorCode) throws IOException {
    try (var lines = Files.lines(logFilePath)) {
        return lines.filter(line -> line.contains(errorCode))
                    .count();
    }
}
```

2. Streams backed by I/O must always be wrapped in a **try-with-resources** block.
3. Failure to do so leads to file handle leaks, potentially crashing the application or blocking other processes from reading/writing the file.

### Life Analogy
Imagine turning on the garden hose to wash your car, but when you're done, you just walk away without turning off the spigot. The water keeps running, wasting resources until the city shuts off your water supply.

### Key Points
1. `Files.lines()` wraps native OS resources (file descriptors) that are NOT purely managed by standard JVM heap garbage collection.
