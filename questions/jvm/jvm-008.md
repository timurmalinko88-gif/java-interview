# GC Generations: why does the JVM split the heap, and when does the split hurt you?
Describe the generational heap model used by HotSpot JVM. Why are objects separated into **Young** and **Old** generations? Explain what a **Minor GC**, **Major GC**, and **Full GC** are, and under what conditions each fires.

Then describe a **concrete scenario** where the generational assumption breaks down and causes GC pressure — and how you would diagnose and fix it.

---ANSWER---

1. **Generational hypothesis: most objects die young**  
   The vast majority of objects become unreachable within milliseconds of allocation. Separating them into Young gen lets the GC reclaim them cheaply without scanning the full heap.

2. **Young generation structure: Eden + two Survivor spaces (S0/S1)**  
   New objects land in Eden. Minor GC copies live objects to a Survivor space (copying collector — no fragmentation). Objects that survive enough GC cycles are promoted to Old gen via a **tenuring threshold**.

3. **Minor GC vs Major GC vs Full GC**  
   - **Minor GC**: collects Young gen only; fast (ms range); triggered when Eden fills.  
   - **Major GC**: collects Old gen; slower; triggered when Old gen fills.  
   - **Full GC**: collects entire heap + Metaspace; stop-the-world; triggered by promotion failure, explicit `System.gc()`, or Metaspace exhaustion.

4. **When the generational assumption breaks: premature promotion / humongous objects**  
   Long-lived temporary objects (e.g., objects cached inside a request scope that outlasts multiple Minor GCs) get promoted to Old gen prematurely. Old gen then fills quickly, causing frequent Full GCs. Similarly, G1GC treats objects > 50% of a region as **humongous** and allocates them directly in Old gen, bypassing Young gen entirely.

5. **Diagnosis and remediation**  
   - Use `-Xlog:gc*` (Java 9+) or `-verbose:gc` to detect frequent Full GCs.  
   - Use heap profilers (JFR, async-profiler, Eclipse MAT) to identify what is being promoted.  
   - Tune `-XX:MaxTenuringThreshold`, region sizes (`-XX:G1HeapRegionSize`), or redesign object lifetimes (object pooling, smaller scopes, off-heap for large blobs).

### Heap regions overview

```
[ Young Generation           ] [ Old Generation ] [ Metaspace ]
  Eden | Survivor0 | Survivor1
   ^                                ^
   new allocations              promoted objects
```

### GC cycle summary

| GC type | Scope | Typical pause | Trigger |
|---|---|---|---|
| Minor GC | Young gen | < 50 ms | Eden full |
| Major GC | Old gen | 100 ms – seconds | Old gen full |
| Full GC | Entire heap + Metaspace | Seconds | Promotion failure, explicit `System.gc()`, Metaspace OOM |

### Concrete failure scenario

```java
// Anti-pattern: storing large objects in a request-scoped "context"
// that lives longer than a single Young GC cycle
public class RequestContext {
    // This 5 MB report buffer survives multiple Minor GCs
    // and gets promoted to Old gen on every request
    private byte[] reportBuffer = new byte[5 * 1024 * 1024];
}
```

**Fix**: use a `ThreadLocal` pool, allocate off-heap with `ByteBuffer.allocateDirect()`, or stream the data instead of buffering.

### Diagnosis commands

```bash
# Enable GC logging (Java 9+)
java -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m MyApp

# Analyse heap dump with Eclipse MAT
jmap -dump:format=b,file=heap.hprof <pid>
```

## Common Mistakes

- Confusing **Major GC** (Old gen only) with **Full GC** (entire heap). G1GC blurs this distinction — it has concurrent Mixed GC phases that collect both generations incrementally.
- Assuming `System.gc()` is safe to call in production — it triggers a Full GC and can cause multi-second pauses.
- Ignoring Metaspace as a trigger: dynamic class generation (reflection proxies, lambda desugaring, Groovy scripts) can exhaust Metaspace and trigger Full GCs even when the heap is fine.

### Life Analogy

Imagine a **restaurant kitchen**: most dishes (young objects) are used briefly, washed, and put back — a fast dishwasher cycle (Minor GC) handles them constantly. Long-term equipment like heavy pots (old objects) rarely needs cleaning and goes through a deep-clean cycle (Major GC) infrequently. The problem arises when someone keeps pulling a pan from the fast cycle onto the permanent shelf prematurely — suddenly the permanent shelf fills up and requires a full kitchen shutdown (Full GC) to reorganise.
