# Code Review: Find the Metaspace Leak

## Context

A team runs a plugin system that compiles and loads user-supplied Groovy scripts at runtime. After several hours in production, the service crashes with:

```
java.lang.OutOfMemoryError: Metaspace
```

A colleague says: "We call `unloadPlugin()` when users replace a script, so memory should be freed." Review the code below and identify what is actually wrong.

## Code Under Review

```java
import javax.script.*;
import java.util.*;
import java.util.concurrent.ConcurrentHashMap;

// PluginRegistry.java
public class PluginRegistry {

    // Bug area 1
    private static final Map<String, ScriptEngine> engineCache =
            new ConcurrentHashMap<>();                                  // (A)

    private static final ScriptEngineManager MANAGER =
            new ScriptEngineManager();

    public static void loadPlugin(String pluginId, String scriptSource)
            throws ScriptException {
        ScriptEngine engine = MANAGER.getEngineByName("groovy");       // (B)
        engine.eval(scriptSource);
        engineCache.put(pluginId, engine);                             // (C)
    }

    public static void unloadPlugin(String pluginId) {
        engineCache.remove(pluginId);                                  // (D)
        // Bug area 2: caller believes this is sufficient
    }

    public static Object invokePlugin(String pluginId, String fn,
                                      Object... args) throws Exception {
        ScriptEngine engine = engineCache.get(pluginId);
        if (engine == null) throw new IllegalStateException("Not loaded");
        Invocable inv = (Invocable) engine;
        return inv.invokeFunction(fn, args);
    }
}
```

```java
// PluginWatcher.java  — monitors script files for hot-reload
public class PluginWatcher {

    // Bug area 3
    private static final List<ScriptEngine> auditLog = new ArrayList<>(); // (E)

    public static void onScriptEvaluated(ScriptEngine engine) {
        auditLog.add(engine);    // (F) — "for debugging, keeps last N engines"
    }
}
```

## Bugs to Find

### Bug 1 — `ScriptEngineManager` reuse creates a shared ClassLoader (line B)

`MANAGER.getEngineByName("groovy")` returns a Groovy engine whose `GroovyClassLoader` is held by the shared `ScriptEngineManager`. All compiled script classes are registered in **that one ClassLoader**, which is never discarded. Calling `engineCache.remove()` removes the engine reference but does **not** release the `GroovyClassLoader` or the class metadata it anchors in Metaspace.

**Fix:** Create a **new** `ScriptEngineManager` per plugin (or per reload) so each plugin owns an isolated `GroovyClassLoader` that can be GC'd independently.

```java
// Correct isolation pattern
public static void loadPlugin(String pluginId, String scriptSource)
        throws ScriptException {
    // Fresh manager = fresh GroovyClassLoader = isolatable Metaspace chunk
    ScriptEngineManager localManager = new ScriptEngineManager();
    ScriptEngine engine = localManager.getEngineByName("groovy");
    engine.eval(scriptSource);
    engineCache.put(pluginId, engine);
}
```

### Bug 2 — `unloadPlugin` does not null out local references (line D)

`engineCache.remove()` is correct, but if any other part of the codebase holds a strong reference to the engine (e.g., method-local variable, returned value sitting on a thread stack), the ClassLoader cannot be collected. This is not immediately visible here but compounds Bug 3.

### Bug 3 — `auditLog` is a **static strong reference** to every engine ever created (line E/F)

`PluginWatcher.auditLog` is a `static` `ArrayList` that accumulates every `ScriptEngine` ever evaluated. Because each `ScriptEngine` holds a reference to its `GroovyClassLoader`, **no class metadata can ever be GC'd** as long as the JVM runs. This is the primary cause of the OOM.

**Fix options:**
- Use a bounded `Deque` with a capacity limit and **weak references**: `new WeakReference<>(engine)`.
- Store only metadata (plugin ID, timestamp), not the engine itself.

```java
// Correct audit approach
private static final Deque<String> auditLog = new ArrayDeque<>(100);

public static void onScriptEvaluated(String pluginId, Instant time) {
    if (auditLog.size() >= 100) auditLog.poll();
    auditLog.offer(pluginId + " @ " + time);
}
```

## How to Confirm with JVM Tools

```bash
# 1. Watch Metaspace growth — should plateau after unloads
jstat -gcmetacapacity <pid> 2000

# 2. Capture heap dump and inspect ClassLoader retention
jmap -dump:format=b,file=heap.hprof <pid>
# Open in Eclipse MAT -> "Leak Suspects" -> look for GroovyClassLoader instances

# 3. Count loaded classes over time
jcmd <pid> VM.classloader_stats
```

---ANSWER---

1. **Metaspace holds class metadata** — it is freed only when the defining `ClassLoader` is GC'd, not when objects of that class are GC'd.
2. **A ClassLoader is GC'd only when it has zero strong references** — a single static field holding any related object blocks the entire Metaspace chunk.
3. **`ScriptEngineManager` re-use shares a `GroovyClassLoader`** — every new script adds to the same unreleasable Metaspace region.
4. **Static collections are the most common cause** of ClassLoader leaks — `static List`, `static Map`, logging caches, etc.
5. **Diagnosis path**: `jstat -gcmetacapacity` for growth trend, heap dump + MAT for ClassLoader reference chain.

### Life Analogy

Metaspace is like a library's card catalogue for class blueprints. Imagine you rent a room in the library (a ClassLoader) to store your own book index cards. When you "leave" (dereference the ClassLoader), the library should reclaim your room — but only if nobody else in the building still has a reference to even one of your cards. If the head librarian keeps a global list with one of your cards in it, your entire room is stuck forever.
