Explain the Java ClassLoader hierarchy. What is the **parent-delegation model**, why does it exist, and what happens if you try to load `java.lang.String` with your own custom ClassLoader?


---ANSWER---

### ClassLoader Hierarchy (top to bottom)

| Level | ClassLoader | What it loads |
|---|---|---|
| 1 | **Bootstrap ClassLoader** (native, C++) | `rt.jar` / `java.*` core classes (Java 8) or `java.base` module (Java 9+) |
| 2 | **Platform ClassLoader** (formerly Extension) | JDK extension / platform modules |
| 3 | **Application (System) ClassLoader** | Classes on the `-classpath` / module path |
| 4 | **Custom ClassLoader** (user-defined) | Application plugins, hot-reload, etc. |

### Parent-Delegation Model — How it Works

```
CustomClassLoader.loadClass("com.example.Foo")
  delegates up to AppClassLoader
    delegates up to PlatformClassLoader
      delegates up to BootstrapClassLoader
        NOT FOUND -> returns to Platform -> returns to App
          AppClassLoader finds and loads it
```

### Why Does It Exist?

1. **Security** — Prevents malicious code from shadowing trusted JDK classes (e.g., a rogue `java.lang.String`).
2. **Consistency** — Guarantees that `java.lang.Object` is always the same class across the entire JVM; two classes loaded by different loaders are considered different types even if the bytecode is identical.
3. **Avoids duplicate loading** — If a parent already loaded a class, the child won't load it again.

### What Happens with a Custom `java.lang.String`?

```java
// Attempting to define java.lang.String in a custom ClassLoader
public class EvilLoader extends ClassLoader {
    @Override
    public Class<?> loadClass(String name) throws ClassNotFoundException {
        if (name.equals("java.lang.String")) {
            // defineClass() will throw java.lang.SecurityException:
            // "Prohibited package name: java.lang"
        }
        return super.loadClass(name);
    }
}
```

- The JVM **prohibits** defining classes in `java.*` packages from non-bootstrap loaders.
- Attempting to call `defineClass()` for `java.lang.String` throws a `SecurityException`.
- Even before that: since the delegation model is followed, the parent (Bootstrap) will always find and return the real `java.lang.String` first — your custom version is never even attempted.

### Key Nuance: Breaking Delegation (SPI Pattern)

Some frameworks (JDBC, JNDI, OSGi) intentionally **break** delegation with a **Context ClassLoader** to let lower-level code load higher-level implementations:

```java
Thread.currentThread().setContextClassLoader(pluginLoader);
```

This is a deliberate escape hatch — and a common source of ClassLoader leaks in application servers.


---ANSWER---

1. Three built-in layers: Bootstrap -> Platform -> Application.
2. Delegation always goes **up** first; loading only happens on the way **back down**.
3. `java.*` packages are protected — `defineClass()` throws `SecurityException`.
4. Two classes loaded by **different** ClassLoaders are **not equal** even with identical bytecode — this is why `ClassCastException` can appear in plugin systems.
5. The Thread Context ClassLoader is the standard workaround for SPI / service-loader patterns.

### Life Analogy

Think of a chain of managers in a company. When a new task comes in, the intern doesn't just start doing it — they first pass it up to the team lead, who passes it to the manager, who passes it to the director. Only if the director says "I don't know how to do this" does the task get delegated *back down* the chain. Java ClassLoaders work exactly the same way: every load request bubbles up to the top before anyone at a lower level attempts to handle it.
