# Code Review: The Immortal Cache
Review the following Java 21 code for a user cache. Identify the bugs and explain how they impact JVM memory management. How would you fix them?

```java
import java.util.HashMap;
import java.util.Map;

public class UserSessionCache {
    // Stores active sessions
    private static final Map<String, UserSession> sessionCache = new HashMap<>();

    public void addSession(String sessionId, UserSession session) {
        sessionCache.put(sessionId, session);
    }

    public UserSession getSession(String sessionId) {
        return sessionCache.get(sessionId);
    }
}

class UserSession {
    private byte[] data = new byte[1024 * 1024]; // 1MB session data
    private String username;

    public UserSession(String username) {
        this.username = username;
    }

    // Clean up resources when the object is garbage collected
    @Override
    protected void finalize() throws Throwable {
        System.out.println("Session for " + username + " is being cleaned up.");
        super.finalize();
    }
}
```

---ANSWER---

The provided code has several major flaws relating to memory management that will quickly lead to an `OutOfMemoryError`.

### Life Analogy
Imagine a hotel that registers every guest in a massive logbook. Even after the guests check out and leave the city, their names are never erased from the logbook, and their rooms are never reassigned. Eventually, the hotel turns away new guests because it thinks it's completely full. Furthermore, the hotel manager insists on personally singing a farewell song (`finalize()`) to every departing guest, drastically slowing down the checkout process.

### Bugs & Issues Identified

1. **Unbounded Static Cache (Memory Leak)**:
   - **Bug**: `sessionCache` is a `static HashMap`. Sessions are added via `addSession` but never removed.
   - **Impact**: Because the map is strongly reachable (tied to the class literal via a static field), the `UserSession` objects (each taking 1MB) will never be garbage collected, causing an `OutOfMemoryError`.
   - **Fix**: Use a cache with an eviction policy (e.g., Caffeine/Guava cache with expiration), or manually remove sessions upon logout. `WeakHashMap` could be used if keys are weakly referenced elsewhere, though string keys often make this tricky.

2. **Use of Deprecated `finalize()` method**:
   - **Bug**: The `UserSession` class overrides `finalize()`.
   - **Impact**: `finalize()` is deprecated in Java 9+ (and marked for removal in Java 18+). Objects with finalizers are placed in a finalization queue, requiring at least two GC cycles to be reclaimed. If the finalizer thread falls behind, memory can build up rapidly. 
   - **Fix**: Remove the `finalize()` method entirely. If explicit cleanup of native resources is ever needed, use `AutoCloseable` with a `try-with-resources` block or the `java.lang.ref.Cleaner` API (Java 9+).

### Mandatory Key Points
1. **Static Collections**: Explain that static variables are GC roots; anything referenced by them lives forever unless explicitly removed.
2. **Missing Eviction**: Point out the absence of a `remove` mechanism or eviction policy.
3. **Finalizer Drawbacks**: Explain why `finalize()` is harmful (performance degradation, delayed GC, deprecated in modern Java).
