# The Silent Memory Eater
You are reviewing a simple Java application that pulls user data from a database and caches it in a `static HashMap` to improve performance. The application runs fine in testing, but in production, it throws an `OutOfMemoryError: Java heap space` after about three days of continuous operation. 

Explain why this happens, how the JVM Garbage Collector interacts with this code, and propose a reliable fix.

## Analogy
Think of your computer's memory like a public library. If visitors take out books and never return them (because they put them in a permanent display case near the door), eventually, the library runs out of books. No matter how hard the librarian (Garbage Collector) tries to clean up, they aren't allowed to touch the permanent display.

---ANSWER---

### 1. The Root Cause
The `static HashMap` creates a strong, permanent reference to every object placed inside it. Because static variables belong to the Class object (and are tied to the ClassLoader's lifecycle), they exist for the lifetime of the application. 

### 2. How the Garbage Collector Reacts
The Garbage Collector (GC) tracks references starting from "GC Roots" (like static variables, active threads, and local variables on the stack). 
* Since the `HashMap` is a static GC Root, any object referenced by it is considered "alive."
* Even if the user logs out and their data is no longer needed, the map still holds the reference.
* The GC runs continuously but cannot free up this space, eventually filling the entire heap.

### 3. Proposed Solutions
* **Use a specialized caching library:** Libraries like Caffeine or Ehcache provide time-based (TTL) or size-based eviction policies.
* **Weak References:** Using a `WeakHashMap` can allow keys to be garbage collected if there are no other strong references to them elsewhere in the application, though this requires careful design to ensure the keys aren't held unnecessarily.

## Mandatory Key Points
1. Static variables act as **GC Roots**, meaning their referenced objects are immune to garbage collection for the application's lifespan.
2. The application suffers from a **Memory Leak** (specifically, unintentional object retention, as Java doesn't have true pointer leaks like C++).
3. The fix requires an **eviction policy** (TTL or size limit) rather than an unbounded standard collection.
