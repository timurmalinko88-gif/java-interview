# Encapsulation vs. Abstraction in the Era of Java Records
A common definition states that encapsulation hides data, while abstraction hides complexity. With the introduction of `record` in Java 14+, data is often completely exposed via public accessors, yet developers still consider this good OOP design in many contexts. 

Explain the difference between encapsulation and abstraction. Then, discuss how Java `record`s fit into these concepts—do they violate encapsulation by exposing all their data?

- **Encapsulation** is the hood locked shut. You aren't allowed to manually inject fuel into the engine cylinders; the system protects its own state to prevent you from breaking it.
- **Abstraction** is the steering wheel and pedals. You don't need to understand the combustion engine to drive the car; you just use a simplified interface.

## Expected Answer Structure
- Define **Encapsulation** (bundling data and methods, restricting direct access to protect invariants).
- Define **Abstraction** (exposing only essential features of an object, hiding background details).
- Explain the role of **Java Records** as transparent data carriers.
- Analyze if records violate encapsulation.

---ANSWER---

1. **Clear Distinction**: Encapsulation is about *protection and boundaries* (invariants), while abstraction is about *simplification and interfaces*.
2. **Records Intent**: Records are explicitly designed to be *transparent carriers for immutable data*, meaning the data *is* the interface.
3. **Immutability as Protection**: Records do not violate encapsulation because their fields are `final`. Since the data cannot be changed after construction, there are no internal invariants to protect against mutation. 
4. **Shallow vs Deep**: A candidate should note that records only provide *shallow* immutability. If a record holds a mutable object (like a `List`), encapsulation can still be broken if a defensive copy isn't made.
