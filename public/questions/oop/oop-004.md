During a code review, you were sent an implementation of an Immutable Collection. The developer decided to inherit from the standard `java.util.ArrayList` to reuse its storage logic, but overridden the mutation methods to protect the collection.

Look at the code. Which basic principle of object-oriented design (from SOLID) is grossly violated here? What production bugs will this lead to? Propose a correct architectural solution.

```java
import java.util.ArrayList;
import java.util.Collection;

public class ImmutableList<E> extends ArrayList<E> {
    
    public ImmutableList(Collection<? extends E> c) {
        super(c);
    }

    @Override
    public boolean add(E e) {
        throw new UnsupportedOperationException("This collection is immutable!");
    }

    @Override
    public boolean remove(Object o) {
        throw new UnsupportedOperationException("This collection is immutable!");
    }
    
    // ... overriding other mutation methods ...
}
```

---ANSWER---

This code grossly violates the **Liskov Substitution Principle (LSP)**, the letter "L" in SOLID.

The principle states: objects of a base class should be replaceable with objects of its descendants without altering program correctness. 

The `ArrayList` class in its contract (by implementing the `List` interface) promises that the `add()` method will successfully add an element. If we pass an instance of our `ImmutableList` into a method that takes an `ArrayList` or `List` and attempts to add an element there (for example, a filtering or aggregation method), the program will crash in runtime with an `UnsupportedOperationException`. This is a betrayal of the caller code's expectations.

*(By the way, this is exactly why methods like `Collections.unmodifiableList()` in Java are often criticized for their design, even though they became a de-facto standard before the introduction of `List.of()` in Java 9).*

**How to fix:**
Do not use inheritance (`extends`). If we want to create an immutable collection, we should use **Composition**. The class should not extend `ArrayList`; it should implement the `Iterable` interface (or create its own narrow `ReadOnlyList` interface), and encapsulate an `ArrayList` or a regular array inside itself, exposing only read methods to the outside (`get`, `size`, `iterator`).

- Throwing `UnsupportedOperationException` in overridden methods is a classic sign of an LSP violation.
- Inheritance for the sake of code reuse (as in the ArrayList example) is an anti-pattern. Composition should be used for reuse.
- In modern Java versions, for immutable collections, you should use factory methods like `List.of()`, which return hidden implementations without publicly violating the hierarchy of mutable lists.

### Life Analogy
Imagine you ordered a taxi (base class). The taxi contract says: "open the door, sit down, we will go". A "Cash-in-transit vehicle" arrives (descendant of a taxi, because it's also a car). You pull the handle, but it shocks you with electricity and yells "UnsupportedOperationException: Doors are locked!". The cash-in-transit vehicle violated the expected taxi contract, although it is technically a car.

### Key Points
- Violating LSP leads to unpredictable behavior and hidden bugs in runtime (Runtime Exceptions).
