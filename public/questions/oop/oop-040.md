# OOP at Scale: Distributed Domain Aggregates

## Context
In monolithic applications, OOP principles like Encapsulation and Data Hiding are straightforward because everything shares the same JVM and memory space. 
You are designing an e-commerce platform transitioning to a microservices architecture. You have three primary domains: **Orders**, **Payments**, and **Inventory**.
How do you apply OOP encapsulation and invariants when an object's behavior spans multiple distributed services? Specifically:
1. How do you model an `Order` using Domain-Driven Design (DDD) principles (Aggregates and Aggregate Roots) to enforce encapsulation?
2. If an `Order` requires the `Inventory` to be decremented and a `Payment` to be processed, how do you maintain system consistency without distributed transactions (Two-Phase Commit), while preserving the encapsulation of each domain? Discuss the trade-offs of your approach.

**Encapsulation (Aggregates)**: You don't walk into the Accounting department and directly modify their spreadsheets (direct data access). You submit a formal request to the Accounting Manager (Aggregate Root).
**Consistency (Eventual vs Immediate)**: If Sales promises a product, they send a memo to the Warehouse. The product isn't instantly teleported (Immediate Consistency/2PC); the Warehouse processes the memo asynchronously (Eventual Consistency/Events). If they are out of stock, they send a failure memo back, and Sales issues a refund (Compensating Transaction).

## Expected Answer Structure
- Define **Aggregate** and **Aggregate Root** in the context of the `Order`.
- Explain how encapsulation applies at the architectural boundaries (Microservices).
- Introduce the **Saga Pattern** or **Domain Events** for distributed consistency.
- Discuss handling failures (Compensating transactions) and the trade-offs between consistency, availability, and complexity.

---ANSWER---

1. **Aggregate Roots as Encapsulation Boundaries**: An `Order` is an Aggregate Root. External components cannot directly modify an `OrderItem` or an `OrderShippingAddress`. They must invoke business methods on the `Order` root (e.g., `order.addItem(...)`), which guarantees local invariants (like calculating totals).
2. **No Shared Database / State**: To respect OOP encapsulation at scale, microservices must not share a database. The Order service owns order data; Inventory owns stock data. Direct database joins are a violation of macro-level encapsulation.
3. **Event-Driven Communication (Sagas)**: To coordinate without 2PC, the `Order` aggregate emits a Domain Event (e.g., `OrderCreatedEvent`). The Payment and Inventory services listen to this event, perform their localized Aggregate updates, and emit their own events (`InventoryReserved`, `PaymentProcessed`).
4. **Eventual Consistency & Compensation**: The candidate must acknowledge the trade-off: giving up immediate ACID consistency for distributed eventual consistency. If `Payment` fails after `Inventory` is reserved, a compensating event (`PaymentFailed`) must trigger a reversal in the Inventory service.
5. **Outbox Pattern (Bonus/Strong Answer)**: Mentioning the Transactional Outbox pattern ensures that saving the Aggregate state and publishing the Domain Event occur atomically, preventing "ghost" state changes where the database updates but the message broker fails.

Think of microservices like distinct corporate departments.
