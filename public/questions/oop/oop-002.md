Explain the difference between method Overloading and Overriding using an example. Which of these concepts refers to static polymorphism (compile-time) and which to dynamic (run-time)?

Imagine the situation: you wrote a `print(Object o)` method and a `print(String s)` method. If you call `print(null)`, which one will execute and why?

---ANSWER---

**Overloading** is the creation of multiple methods in a single class with the same name, but a **different parameter list** (different number or types of arguments). This is *static polymorphism (Compile-time)*. The compiler itself decides which method to call based on the types of passed variables when writing the code.

**Overriding** is replacing the implementation of a base class method in a subclass. The name, parameters, and return type (or covariant type) must match. This is *dynamic polymorphism (Run-time)*. The JVM decides which method to call based on the actual object type in memory during program execution (Late Binding).

**Answer to the `print(null)` problem:**
The `print(String s)` method will be called. 
During overloading, the Java compiler always looks for the **most specific (narrow) type** that fits the passed argument. Since `null` can be cast to any reference type, and `String` is a more specific descendant of `Object`, the compiler will choose the method with `String`. 
*(Note: if there were `print(String)` and `print(Integer)` methods, the compiler would throw an "ambiguous method call" error, as they are on the same level of hierarchy).*

- **Overriding:** You assign the task "Make a report" to your intern (descendant). You expect him to do it just like you (the task interface is the same), but he does it his own way, using Excel instead of Word. The decision on exactly how the report will be made is taken the moment a specific person starts the work (Run-time).

- Overriding — same parameters, different classes (parent-child), Run-time (Late Binding).
- With Overloading, the compiler chooses the most specific matching type.
- The `@Override` annotation protects against typo errors when overriding.

### Life Analogy
- **Overloading:** You have a "Screwdriver" tool. Depending on which screw you apply it to (Phillips or flathead), you use the correct bit. The tool is one, but works differently depending on the "input parameters". And you decide this before starting the work (Compile-time).

### Key Points
- Overloading — different parameters, one class, Compile-time (Early Binding).
