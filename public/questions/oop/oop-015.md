Design the Object-Oriented model for a **Distributed Rate Limiter** used across a microservice architecture. Focus on the core domain classes, how state is encapsulated, and the trade-offs regarding consistency vs. latency (CAP theorem). 

---ANSWER---

1. **Domain Entities:** Clear separation between `RateLimiterPolicy` (rules), `TokenBucket` / `Counters` (state), and the `LimiterStrategy` (algorithm like Sliding Window vs Token Bucket).
2. **Storage Interface:** Abstraction of the storage layer (e.g., `RateLimitRepository`) so Redis or Hazelcast can be injected, keeping the domain logic pure.
3. **Concurrency/Atomicity:** Discussion on how the OO model handles atomic operations (e.g., Lua scripts in Redis represented as an atomic `consume()` method).
4. **Trade-offs:** Explicitly weighing local in-memory caching (faster, less consistent) versus synchronous remote state (slower, strictly consistent).

* **Requirements & Algorithm:** Choose an algorithm (e.g., Token Bucket) and justify it.
* **Class Diagram / Interfaces:** Define the core Java interfaces (`RateLimiter`, `Policy`, `StateStore`).
* **Distributed Handling:** Explain how the `StateStore` implementation interacts with the network (e.g., Redis).
* **Trade-off Analysis:** Detail the CAP theorem implications of the chosen state synchronization method.

### Life Analogy
Imagine multiple ticket gates at a stadium (distributed nodes). If each gate checks with a central office for every fan (Strong Consistency), the line moves too slowly (High Latency). If each gate keeps its own tally and syncs every minute (Eventual Consistency), they might let too many people in. The design must balance the speed of the line with the strictness of the capacity limit.
