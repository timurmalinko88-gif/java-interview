# Modernizing OOP: Inheritance vs. Composition
Your team is migrating an old Java 8 system to Java 21. The old system relies heavily on deep, complex inheritance trees (e.g., `BaseEntity -> AbstractUser -> RegisteredUser -> PremiumUser`). 

How would you leverage modern Java features (specifically Records, Sealed Classes, and Pattern Matching) to refactor this codebase? Compare this approach to traditional inheritance.

---ANSWER---

Modern Java pushes developers strongly toward **Composition over Inheritance** and **Data-Oriented Programming**. 

**1. Taming the Hierarchy with Sealed Classes**
- Deep inheritance allows unknown, endless subclassing, breaking encapsulation.
- **Java 17+ Sealed Classes** (`sealed class User permits Registered, Premium`) restrict exactly which classes can inherit. 
- This gives the compiler complete knowledge of the hierarchy, preventing rogue subclasses.

**2. Immutable Data Carriers with Records**
- Traditional subclasses often add mutable state, making them thread-unsafe and hard to test.
- **Java 16+ Records** (`record PremiumUser(String id, int points)`) represent transparent, immutable data carriers. They replace bulky POJOs and enforce immutability, which pairs perfectly with composition.

**3. Replacing the Visitor Pattern with Pattern Matching**
- In traditional OOP, interacting with different subclasses required complex polymorphic method overrides or the Visitor design pattern.
- **Java 21 Pattern Matching for `switch`** allows exhaustive checks. Because the hierarchy is sealed, the compiler ensures every subtype is handled without needing a `default` case.

### Key Mandatory Points for a Correct Answer
1. **Sealed Classes:** Must explain that they restrict inheritance boundaries and guarantee exhaustiveness.
2. **Records:** Must identify them as a tool for creating immutable, boilerplate-free data carriers (composition-friendly).
3. **Pattern Matching:** Must mention how `switch` expressions replace polymorphic dispatch or Visitor patterns by allowing safe, exhaustive type checking.
4. **Trade-offs:** Mention that traditional inheritance tightly couples data and behavior, while modern Java separates them for better maintainability and thread-safety.

Modern Java OOP (Composition + Sealed Classes) is like using **standardized shipping containers**. You define exactly what types of containers exist (sealed), pack immutable data inside them (records), and snap them together seamlessly (pattern matching).

### Life Analogy
Traditional Inheritance is like **building extensions onto a single house**. Eventually, the foundation cracks under the weight, and changing the first floor breaks the third floor.
