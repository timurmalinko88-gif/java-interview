# Distributed OOP: Aggregates and Event Sourcing
In a traditional monolithic Java application, Object-Oriented principles dictate that an object encapsulates its state and exposes methods to change that state (e.g., `BankAccount.withdraw()`). 

However, you are tasked with designing a highly concurrent, distributed system for a Bank across multiple microservices. Instead of traditional database CRUD operations, the system must use **Event Sourcing** and **Domain-Driven Design (DDD)**.

How do you map traditional OOP concepts (like Encapsulation, State, and Behaviors) to an Event-Sourced architecture? Specifically, explain how a `BankAccount` object ensures its invariants (e.g., balance cannot go below zero) when its state is fundamentally a stream of distributed events.

Distributed Event Sourcing is like a bank ledger. You never erase anything. If you want to know your balance, you read the ledger from page one, adding deposits and subtracting withdrawals. To protect the ledger, a designated teller (the Aggregate Root) verifies every new transaction before it is permanently written down.

## Explanation

**1. The Aggregate Root (Encapsulation Boundary)**
In distributed systems, the OOP concept of an "object" evolves into a DDD **Aggregate Root**. The `BankAccount` is the aggregate root. Its job is to enforce business invariants (rules). External services cannot modify the bank account directly; they must send a Command to the aggregate.

**2. Commands vs. Events (Behavior vs. State)**
*   **Commands (Intent):** E.g., `WithdrawMoneyCommand`. This triggers a behavior/method on the Aggregate Root.
*   **Events (Fact):** E.g., `MoneyWithdrawnEvent`. If the command succeeds, the Aggregate generates an immutable event. 

**3. Maintaining State without Mutable Fields**
Instead of calling `this.balance -= amount`, the Aggregate does two things:
1. Checks the invariant: `if (this.balance - amount < 0) throw Exception;`
2. Yields an event: `return new MoneyWithdrawnEvent(amount);`

The actual state of the object is reconstructed by replaying past events. The `BankAccount` has an `apply(Event e)` method that blindly updates the internal state for subsequent checks, but no business logic happens in the `apply` method.

**4. Concurrency and Consistency**
Because state is distributed, you protect invariants across concurrent requests using **Optimistic Concurrency Control**. Every aggregate instance has a version number derived from the event stream. When the aggregate saves new events to the Event Store, it includes the expected version. If another node appended an event first, the version mismatches, the save fails, and the command is retried.

2. **Commands vs Events:** Clearly separate behaviors (Commands targeting the Aggregate) from state changes (Immutable Events emitted by the Aggregate).
3. **State Rehydration:** Explain that the object's internal state is built by replaying historical events, not by direct database updates.
4. **Invariant Protection:** The business logic (e.g., balance checks) runs *before* the event is generated.
5. **Concurrency Strategy:** Mention optimistic locking/versioning on the Event Store to prevent race conditions in a distributed environment.

### Life Analogy
Traditional OOP is like keeping a pencil tally on a whiteboard; you erase the old number and write the new one. You only see the current state.
