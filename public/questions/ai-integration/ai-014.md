# Prompt Caching & Tokens

When building enterprise applications with Large Language Models (LLMs), understanding how models process text—and how you pay for that processing—is critical. The fundamental unit of computation and billing in the LLM ecosystem is the **Token**. Managing tokens efficiently directly impacts application performance and operational costs. 

### What is a Token?

A token is a fragment of a word. LLMs do not read text character-by-character or word-by-word. They use a "Tokenizer" to break text down into tokens. 
* As a general rule of thumb for English text, **1 token ≈ 4 characters** or **0.75 words**.
* For example, the word "hamburger" might be split into three tokens: `ham`, `bur`, `ger`.
* Code, JSON, and non-English languages often require significantly more tokens per character.

### Context Windows and Billing

Every LLM has a **Context Window** limit (e.g., 128k tokens for GPT-4o). This limit includes the sum of both the Input (Prompt) and the Output (Generation). If you send a document that exceeds the limit, the API will reject the request.

Furthermore, commercial APIs charge based on token usage, usually distinguishing between:
1. **Input Tokens:** The text you send to the model (cheaper).
2. **Output Tokens:** The text the model generates (more expensive).

In applications like Retrieval-Augmented Generation (RAG) or AI Agents, you repeatedly send large amounts of context (system prompts, chat history, retrieved documents) in every request, causing costs and latency to skyrocket.

### Prompt Caching

To solve the inefficiency of repeatedly processing the same input tokens, modern LLM providers (like Anthropic and OpenAI) introduced **Prompt Caching**.

Prompt caching allows the LLM provider to temporarily store the computed state of a large prefix of your prompt. If you send a subsequent request with the exact same prefix, the provider skips re-computing it, resulting in drastically lower latency and reduced input costs (often discounted by 50-90%).

#### How it works in practice

Prompt caching relies on static prefixes. 
Consider a RAG prompt:
`[System Instructions] + [Large Book Text] + [User Chat History] + [New User Question]`

To leverage caching, the unchanging parts (`System Instructions` and `Large Book Text`) must be placed at the very beginning of the prompt. If you change even one character in the prefix, the cache is busted.

#### Java Implementation Considerations

While the caching mechanism happens entirely on the provider's servers, Java developers must structure their requests to utilize it. In frameworks like Spring AI, this involves structuring the `SystemMessage` properly.

```java
import org.springframework.ai.chat.messages.SystemMessage;
import org.springframework.ai.chat.messages.UserMessage;
import org.springframework.ai.chat.prompt.Prompt;
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.stereotype.Service;

import java.util.List;

@Service
public class CachingChatService {

    private final ChatClient chatClient;
    private final String massiveKnowledgeBase; // Assume this is a 50k token string

    public CachingChatService(ChatClient chatClient) {
        this.chatClient = chatClient;
        this.massiveKnowledgeBase = loadHugeDocument();
    }

    public String askQuestion(String userQuestion) {
        // The SystemMessage contains the static data. 
        // Providers will automatically cache this large block if sent repeatedly.
        SystemMessage systemMsg = new SystemMessage(
            "You are an expert. Use this context: " + massiveKnowledgeBase
        );
        
        // The UserMessage changes on every request, so it goes at the end.
        UserMessage userMsg = new UserMessage(userQuestion);

        Prompt prompt = new Prompt(List.of(systemMsg, userMsg));
        
        return chatClient.prompt(prompt).call().content();
    }
}
```

### Local Token Counting in Java

To prevent hitting context window limits before making an API call, you can count tokens locally. Java libraries like `jtokkit` (for OpenAI's tiktoken) allow you to calculate token usage offline.

```java
import com.knuddels.jtokkit.Encodings;
import com.knuddels.jtokkit.api.Encoding;
import com.knuddels.jtokkit.api.EncodingRegistry;
import com.knuddels.jtokkit.api.ModelType;

public class TokenCounter {
    public static void main(String[] args) {
        EncodingRegistry registry = Encodings.newDefaultEncodingRegistry();
        Encoding encoding = registry.getEncodingForModel(ModelType.GPT_4);
        
        String text = "Hello, how many tokens is this sentence?";
        int tokenCount = encoding.countTokens(text);
        
        System.out.println("Tokens: " + tokenCount);
    }
}
```
By understanding tokens and leveraging prompt caching, developers can build AI applications that are both performant and economically viable.
