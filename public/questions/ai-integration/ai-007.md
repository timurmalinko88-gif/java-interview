# Function Calling (@Tool) in LangChain4j

Modern Large Language Models (LLMs) like GPT-4 and Claude 3 are incredibly powerful, but by default, they are isolated. They cannot browse the web, read your corporate database, or trigger external APIs. **Function Calling** (also known as Tool Calling) bridges this gap, allowing the LLM to interact with external systems. 

In the Java ecosystem, **LangChain4j** provides an elegant, annotation-driven approach to Function Calling using the `@Tool` annotation.

### How Function Calling Works

1. **Definition:** You provide the LLM with a prompt and a list of available "Tools" (functions), including their names, descriptions, and expected parameters.
2. **Decision:** The LLM analyzes the user's prompt. If it determines it needs data from a tool to answer, instead of returning a text response, it returns a JSON object specifying which tool to call and with what arguments.
3. **Execution:** The application intercepts this JSON, executes the corresponding local Java method, and passes the result back to the LLM.
4. **Final Answer:** The LLM uses the result of the tool execution to generate a final natural language response for the user.

### Implementing @Tool in LangChain4j

LangChain4j abstracts away the complex JSON serialization and request-response looping. You simply annotate standard Java methods.

#### 1. Define the Tools

Create a Java class containing the methods you want the LLM to access. Annotate them with `@Tool`. The Javadoc or the annotation value itself acts as the description sent to the LLM.

```java
import dev.langchain4j.agent.tool.P;
import dev.langchain4j.agent.tool.Tool;
import org.springframework.stereotype.Component;

@Component
public class CustomerServiceTools {

    // The Tool description is critical. The LLM reads this to know WHEN to use the tool.
    @Tool("Retrieves the current order status for a given customer and order ID.")
    public String getOrderStatus(
            @P("The unique customer ID") String customerId, 
            @P("The specific order ID") String orderId) {
        
        // In reality, this would query a database or external REST API
        if ("123".equals(customerId) && "ORD-99".equals(orderId)) {
            return "Order ORD-99 is currently SHIPPED and arriving tomorrow.";
        }
        return "Order not found.";
    }
    
    @Tool("Cancels an order if it has not been shipped yet.")
    public String cancelOrder(@P("The specific order ID") String orderId) {
        return "Order " + orderId + " has been successfully cancelled.";
    }
}
```

#### 2. Create the AI Service

LangChain4j uses declarative interfaces (AI Services) to bind everything together.

```java
import dev.langchain4j.service.SystemMessage;
import dev.langchain4j.service.spring.AiService;

@AiService
public interface CustomerAssistant {
    
    @SystemMessage({
        "You are a helpful customer support assistant.",
        "Use the provided tools to look up order information or perform actions."
    })
    String chat(String userMessage);
}
```

#### 3. Execution Configuration

When configuring the application (or relying on LangChain4j Spring Boot Starter), the framework automatically wires the tools into the ChatModel requests.

```java
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class ChatController {

    private final CustomerAssistant assistant;

    public ChatController(CustomerAssistant assistant) {
        this.assistant = assistant;
    }

    @GetMapping("/chat")
    public String chat(@RequestParam String message) {
        // If the user says: "Where is my order ORD-99 for customer 123?"
        // 1. LLM detects it needs getOrderStatus tool.
        // 2. LangChain4j executes getOrderStatus("123", "ORD-99").
        // 3. Result is sent back to LLM.
        // 4. LLM replies: "Your order is shipped and arriving tomorrow!"
        return assistant.chat(message);
    }
}
```

### Best Practices

1. **Clear Descriptions:** The `@Tool` and `@P` descriptions are essentially prompts. Make them unambiguous so the LLM doesn't guess incorrectly.
2. **Error Handling:** Tools should gracefully handle exceptions and return meaningful string messages (e.g., "Error: Database timeout, try again"). The LLM will read this error and apologize to the user.
3. **Security:** Never expose destructive tools (like `dropDatabase()`) without human-in-the-loop approval, as LLMs can be manipulated via Prompt Injection.
