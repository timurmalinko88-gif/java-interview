# Corporate AI assistant use-case
System Design: Architect an internal "HR AI Assistant" for a large corporation using Java. The assistant must answer questions based on private PDF employee handbooks, book time off via an internal REST API, and ensure employees can only access policies relevant to their specific department.

---ANSWER---

Designing an enterprise-grade AI Assistant requires combining RAG (for knowledge retrieval), Agents/Tools (for taking actions), and robust Security mechanisms. Here is an architectural blueprint using a Java framework like Spring AI or LangChain4j.

**1. Data Ingestion Pipeline (Asynchronous)**
To answer questions about PDF handbooks, we need a RAG ingestion pipeline.
*   **Document Loading:** Use an S3 or local filesystem `DocumentLoader` to read the PDFs.
*   **Chunking:** Use a `TokenTextSplitter` to break the PDFs into 500-token chunks with overlap to respect LLM context limits.
*   **Embedding & Storage:** Send chunks to an Embedding Model (e.g., Azure OpenAI or local Ollama for privacy) and store them in a Vector Database like `pgvector` or Qdrant.
*   **Crucial Step (Metadata):** During ingestion, every chunk must be tagged with metadata indicating its department (e.g., `{"department": "Engineering"}`). This is vital for security.

**2. The AI Agent Orchestrator (Synchronous)**
The core of the assistant is an Agent loop (ReAct pattern) powered by a Chat Model (e.g., GPT-4). The Agent has access to two main capabilities:
*   **Capability 1: RAG Tool.** A Java method annotated with `@Tool` that performs similarity search against the Vector DB.
*   **Capability 2: Action Tool.** A Java method annotated with `@Tool` that makes REST API calls to the internal HR system to book time off.

**3. Security & Access Control (Department Filtering)**
We cannot rely on the LLM to enforce security; it can be tricked via prompt injection. Security must happen at the data retrieval layer.
*   When the user sends a query, the Spring Boot application authenticates the user (via Spring Security / OAuth2) and retrieves their department claim from their JWT.
*   When the Agent calls the RAG Tool, the Java code dynamically constructs a **Metadata Filter**. It forces the Vector DB query to *only* return documents where `department == user.department` OR `department == 'Global'`. The LLM physically never sees policies from other departments.

**4. The Execution Flow**
1.  User (Engineering Dept) types: *"What is the vacation policy and book me next Friday off."*
2.  The framework prepends the Chat Memory and sends it to the LLM Agent.
3.  **Thought/Action 1:** The LLM decides it needs the policy first. It calls the RAG Tool.
4.  The Java RAG Tool executes a similarity search for "vacation policy", *strictly filtered* by `department=Engineering`. It returns the text to the LLM.
5.  **Thought/Action 2:** The LLM reads the policy, confirms the user can take time off, and calls the `bookTimeOff` Tool with arguments `{"date": "next Friday"}`.
6.  The Java Tool executes the internal REST call to the HR system and returns "Success".
7.  **Final Response:** The LLM generates the final response: *"Engineering gets 20 days off. I have successfully booked next Friday for you."*

### Examples
```java
// Tool implementation showing Security/Filtering at the Java level
@Service
public class HrTools {
    private final VectorStore vectorStore;
    private final RestClient hrRestClient;

    // Tool 1: RAG with Security
    @Tool("Search for HR policies regarding benefits, rules, and vacation.")
    public String searchPolicies(String query) {
        // Get user context injected by Spring Security
        String userDept = SecurityContextHolder.getContext().getAuthentication().getAuthorities().toString();
        
        // Construct strict metadata filter
        FilterExpression filter = new FilterExpressionBuilder()
            .in("department", userDept, "Global").build();
            
        List<Document> docs = vectorStore.similaritySearch(
            SearchRequest.query(query).withFilterExpression(filter)
        );
        return docs.stream().map(Document::getContent).collect(Collectors.joining("\n"));
    }

    // Tool 2: Action Tool
    @Tool("Book time off for the current user.")
    public String bookTimeOff(String dateStr) {
        String userId = SecurityContextHolder.getContext().getAuthentication().getName();
        // Call internal API
        ResponseEntity<String> response = hrRestClient.post()
            .uri("/api/pto/book").body(new BookingReq(userId, dateStr)).retrieve().toEntity(String.class);
        return response.getBody();
    }
}
```

### Life Analogy
Imagine the AI is a receptionist.
You ask: "What are the rules for breaks, and tell my manager I'm taking one."
The receptionist (Agent) doesn't know the rules. First, they check your ID badge (Authentication). Your badge says "Engineering". 
They go to the filing cabinet (Vector DB). The cabinet has a physical lock that only opens drawers marked "Engineering" or "Global" (Metadata Filtering). They pull the rules (RAG) and read them.
Then, they pick up the phone and call your manager to log the break (Action Tool). Finally, they tell you everything is sorted.

### Key Points
- A complete architecture requires an async Ingestion Pipeline and a sync Agent loop.
- Agents act as orchestrators, deciding when to search for knowledge (RAG) and when to execute actions (Tools).
- Security (who can read what) MUST be enforced at the Vector DB level using Metadata Filtering based on the user's authenticated identity.
- Never rely on Prompt Engineering alone to hide sensitive data; the LLM must be physically denied access to unauthorized documents.
