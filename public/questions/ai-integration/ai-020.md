# Corporate AI assistant use-case

During Java technical interviews focused on AI, interviewers often present a system design scenario: *"Design a Corporate AI Assistant that can answer questions based on our internal Confluence wiki and Jira tickets, securely."* 

This question tests your ability to synthesize various AI patterns—specifically RAG, Function Calling, memory, and security—into a cohesive enterprise architecture using Java tools like Spring AI or LangChain4j.

### High-Level Architecture Design

A robust Corporate AI Assistant requires several interconnected systems:

1. **The Ingestion Pipeline (Asynchronous):** Continuously syncs Confluence and Jira data into a searchable format.
2. **The Vector Store:** Stores document embeddings for semantic search.
3. **The Orchestration Layer (Java Backend):** Handles user requests, enforces security, and manages the LLM interactions.
4. **The Action Layer (Tools/Agents):** Allows the AI to perform actions (e.g., creating a ticket) rather than just answering questions.

### Step-by-Step System Flow

#### 1. Data Ingestion & Chunking
Nightly batch jobs (or webhook triggers) use Spring Integration or custom services to fetch data from Confluence/Jira APIs.
* **Processing:** The raw HTML/Markdown is cleaned.
* **Chunking:** `TokenTextSplitter` divides large wiki pages into 500-token chunks with overlap.
* **Embedding:** A local model (via Ollama) or cloud model converts text to vectors.
* **Storage:** Vectors and metadata (Author, Date, Document ID) are stored in a dedicated Vector DB like **Milvus** or **pgvector**.

#### 2. The User Request & Security (Guardrails)
A user asks: *"What is the status of the new payment gateway API?"*
* The request hits the Spring Boot backend.
* **Authentication/Authorization:** Spring Security identifies the user. The system *must* append this user's permissions to the context (e.g., ensuring they are allowed to see financial project tickets).

#### 3. Retrieval (RAG)
* The user's query is embedded into a vector.
* The `VectorStore` executes a similarity search. 
* **Crucial:** The search uses Metadata Filtering: `WHERE permission_level <= USER_LEVEL`. This prevents the AI from leaking confidential executive documents to an intern.

#### 4. The Orchestration (Spring AI / LangChain4j)
* **Chat Memory:** The system retrieves the last 5 messages from Redis `ChatMemoryStore` to maintain conversation context.
* **Prompt Assembly:** The Orchestrator combines the System Prompt, the Chat History, the Retrieved Documents, and the User Question.

#### 5. Function Calling (Agentic Actions)
Suppose the user instead asked: *"Create a Jira bug ticket for the payment gateway failure."*
* The LLM recognizes it needs to take action based on the tools provided in its prompt.
* It returns a Function Call request for `createJiraTicket(title, description)`.
* The Java backend intercepts this, validates the user has write permissions, calls the Jira API, and returns the new Ticket ID back to the LLM.

#### 6. Generation & Streaming Response
* The LLM generates the final answer based on the retrieved context or the tool execution result.
* The Spring WebFlux controller streams the response back to the corporate frontend via Server-Sent Events (SSE), ensuring a fast, ChatGPT-like user experience.

### Key Considerations to Mention in an Interview

* **Data Privacy:** Highlight that for corporate data, using private endpoints (Azure OpenAI) or local models (Ollama) is required to prevent proprietary code/data from training public models.
* **Hallucination Prevention:** Emphasize the use of strict System Prompts ("Answer ONLY using the provided context") and potential RAG Evaluation metrics in CI/CD.
* **Cost Management:** Mention Prompt Caching for large system instructions and choosing appropriate models (e.g., using a cheaper model for routing/intent detection and a larger model for complex synthesis).
