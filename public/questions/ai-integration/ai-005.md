# Document Loaders & Splitters (Token/Character)

In any Retrieval-Augmented Generation (RAG) pipeline, the quality of the LLM's response is directly tied to the quality of the data ingested. Before text can be converted into vector embeddings and stored in a database, it must go through a critical preprocessing pipeline involving **Document Loaders** and **Document Splitters** (Chunking). In the Java ecosystem, frameworks like Spring AI and LangChain4j provide robust tools for this workflow.

### 1. Document Loaders

Document Loaders are responsible for extracting raw text and metadata from various data sources (files, databases, APIs) and converting them into a unified `Document` object structure.

Common loaders include:
* **TextLoaders:** For plain `.txt`, `.md`, or `.csv` files.
* **PdfLoaders:** For parsing complex PDFs. Libraries often wrap Apache PDFBox to extract text while maintaining some structural context.
* **Web/HTML Loaders:** For scraping and parsing web pages (often utilizing libraries like JSoup).
* **Office Document Loaders:** For Word, Excel, and PowerPoint files.

**Example using Spring AI:**

```java
import org.springframework.ai.reader.pdf.PagePdfDocumentReader;
import org.springframework.ai.document.Document;
import org.springframework.core.io.FileSystemResource;
import java.util.List;

public class LoaderExample {
    public List<Document> loadPdf() {
        FileSystemResource resource = new FileSystemResource("manual.pdf");
        // PagePdfDocumentReader reads the PDF and creates one Document per page
        PagePdfDocumentReader pdfReader = new PagePdfDocumentReader(resource);
        return pdfReader.get();
    }
}
```

### 2. Document Splitters (Chunking)

Large language models have a strict **Context Window Limit** (e.g., 8k, 128k tokens). Furthermore, embedding models usually have much smaller limits (e.g., 8192 tokens for OpenAI, 512 for some local models). Therefore, large documents must be divided into smaller, semantically coherent segments called "chunks". 

This is where Document Splitters come in. Choosing the right splitting strategy is vital; if chunks are too small, they lose context. If they are too large, they dilute the semantic meaning in the vector space.

#### Character-Based Splitters
These splitters divide text based on a fixed number of characters or specific character separators (like newlines `\n` or paragraphs).

* **Pros:** Fast and simple to implement.
* **Cons:** Can arbitrarily cut words in half or sever the semantic connection between two closely related sentences.

#### Token-Based Splitters
Tokens are the sub-word units that LLMs actually process. Token-based splitters use the specific tokenizer of the target model (e.g., `tiktoken` for OpenAI) to split documents based on token counts.

* **Pros:** Ensures you never exceed the exact token limits of the embedding or LLM models. Preserves better semantic boundaries.
* **Cons:** Slower, and requires importing specific tokenizer libraries.

**Example using Spring AI's TokenTextSplitter:**

```java
import org.springframework.ai.transformer.splitter.TokenTextSplitter;
import org.springframework.ai.document.Document;
import java.util.List;

public class SplitterExample {
    public List<Document> processDocuments(List<Document> rawDocuments) {
        // Create a splitter targeting 500 tokens per chunk, 
        // with an overlap of 100 tokens to preserve context between chunks.
        TokenTextSplitter splitter = new TokenTextSplitter(500, 100, 10, 10000, true);
        
        // The splitter takes the raw documents, breaks their content down,
        // and returns a new, larger list of chunked Documents.
        return splitter.apply(rawDocuments);
    }
}
```

### Best Practices

1. **Use Overlap:** Always configure an overlap (e.g., 10-20% of the chunk size). If a critical concept is split right down the middle, overlap ensures that both resulting chunks retain the full context.
2. **Metadata Preservation:** Ensure your splitters carry over the metadata (like filename, page number, author) from the parent Document to all child chunks. This is critical for filtering later in the vector database.
3. **Semantic Chunking:** Advanced techniques attempt to parse the grammatical structure (sentences, markdown headers) to split text at natural logical boundaries rather than arbitrary token counts.
