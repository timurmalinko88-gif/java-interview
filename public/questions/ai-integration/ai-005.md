# Document Loaders & Splitters
In a Java RAG application, why is Document Splitting (Chunking) necessary? Compare the strategies of Token-based splitting versus Character-based splitting.

---ANSWER---

In a Retrieval-Augmented Generation (RAG) architecture, Document Splitting (or Chunking) is a critical preprocessing step that occurs after documents are loaded and before they are embedded. 

**Why Chunking is Necessary:**
1.  **Context Window Limits:** Every LLM has a hard limit on the number of tokens it can process in a single request (the context window). You cannot feed a 500-page PDF directly into the prompt.
2.  **Embedding Accuracy:** Embedding models represent the semantic meaning of text as a vector. If you embed an entire document as a single vector, the specific details get lost in the "average" meaning of the whole text. Smaller chunks yield much more precise and relevant vectors for similarity search.
3.  **Cost and Latency:** Passing massive amounts of irrelevant text to an LLM increases the token cost and significantly degrades response latency. Chunking ensures only the most relevant snippets are sent to the model.

Java frameworks like Spring AI and LangChain4j provide abstractions for `DocumentLoaders` (to read PDFs, Word docs, URLs) and `TextSplitters` (to break them down). The choice of splitter strategy heavily impacts the quality of your RAG system.

**Character-based Splitting:**
This is the simplest approach. The text is divided based on a fixed number of characters. A naive character splitter might cut a word in half, destroying its meaning. Therefore, frameworks use "recursive character text splitters." They try to split on logical boundaries first (like double newlines `\n\n` for paragraphs, then single newlines, then spaces) until the chunk size is under the character limit.
*   *Pros:* Easy to understand, fast to execute, language-agnostic.
*   *Cons:* Characters do not directly map to LLM tokens. You might set a 1000-character limit, but depending on the language or symbols used, this could result in 200 tokens or 800 tokens. This makes it harder to perfectly optimize for the LLM's context window.

**Token-based Splitting:**
This approach splits the text based on the actual tokens used by the specific LLM. It requires integrating a tokenizer (like OpenAI's `tiktoken` algorithm, often implemented via Java libraries like `jtokkit`).
*   *Pros:* Highly precise. You can guarantee that a chunk will be exactly *N* tokens, allowing you to perfectly pack the LLM's context window without exceeding it.
*   *Cons:* Slower than character splitting because the text must be tokenized first. It is also model-specific; the tokens for OpenAI differ from the tokens for Anthropic or Llama.

For both strategies, configuring **Overlap** is crucial. Overlap ensures that a chunk shares a certain number of characters/tokens with the preceding chunk. This prevents critical context from being lost if a sentence is split exactly across the boundary of two chunks.

### Examples
```java
// Spring AI: Token-based Text Splitter using JTokkit
public List<Document> chunkDocuments(List<Document> rawDocuments) {
    // Split into chunks of 500 tokens, with 100 tokens overlapping to preserve context
    TokenTextSplitter splitter = new TokenTextSplitter(500, 100, 5, 10000, true);
    
    // The splitter retains the metadata of the original document in all chunks
    return splitter.apply(rawDocuments);
}

// LangChain4j: Recursive Character Splitter
public List<TextSegment> split(String text) {
    DocumentSplitter splitter = DocumentSplitters.recursive(
            1000, // Max chunk size in characters
            100   // Overlap in characters
    );
    return splitter.split(Document.from(text));
}
```

### Life Analogy
Imagine trying to memorize an encyclopedia. You can't read the whole thing at once (Context Limit). 
Character splitting is like deciding to memorize exactly one page at a time. It's easy to measure, but a page might end in the middle of a sentence, making it hard to understand.
Token splitting is like deciding to memorize exactly 500 complete words at a time. It takes a bit more effort to count the words, but you guarantee you are processing whole, meaningful concepts. 
Overlap is re-reading the last sentence of the previous section to ensure you don't lose the train of thought.

### Key Points
- Chunking is mandatory due to LLM context limits and to improve vector search accuracy.
- Character splitters break text by length and separators (paragraphs, spaces) but don't map perfectly to LLM limits.
- Token splitters use model-specific algorithms to split text precisely by token count, maximizing context window usage.
- Chunk Overlap is vital to prevent context loss at the boundaries between chunks.
