# Reranking in RAG

In a Retrieval-Augmented Generation (RAG) pipeline, the standard retrieval mechanism relies on **Dense Vector Similarity** (e.g., Cosine Similarity). While embedding models are excellent at finding generally related concepts across a massive database quickly, they lack precision. They often return the best documents at rank 4 or 5 instead of rank 1, or they return documents that share words but have opposing meanings.

To solve this precision problem, advanced RAG architectures implement a two-stage retrieval process: **Retrieval** followed by **Reranking**.

### The Two-Stage Architecture

1. **Stage 1: Broad Retrieval (Bi-Encoder / Vector Store)**
   - **Goal:** Fast, broad sweep.
   - **Mechanism:** The user query is converted into a vector. The database returns the Top-K results (e.g., K = 20).
   - **Speed:** Very fast. Scales to billions of documents.
   - **Precision:** Medium.

2. **Stage 2: Reranking (Cross-Encoder)**
   - **Goal:** High precision sorting.
   - **Mechanism:** A specialized, highly accurate machine learning model evaluates the user's query against *each* of the 20 retrieved documents simultaneously, assigning a strict relevance score. It then sorts the top 20, selecting only the Top-N (e.g., N = 3) to send to the LLM.
   - **Speed:** Slower and computationally expensive, which is why it is only run on a small subset (the top 20) of documents.
   - **Precision:** Extremely high.

### Why Cross-Encoders are Better at Scoring

Standard embedding models (Bi-encoders) process the query and the document separately. They never "see" them together until the dot-product is calculated. 

A Reranker (Cross-encoder) processes the query and the document *at the same time* in its neural network attention layers. It can understand deep semantic interactions between the specific words in the query and the specific words in the document, resulting in a much more accurate relevance score.

### Implementing Reranking in Java

In Java, reranking is often implemented as a middle layer between the `VectorStore` query and the `ChatClient` prompt generation. You can use external APIs (like Cohere Rerank) or run local reranking models using ONNX or custom REST wrappers.

Here is a conceptual implementation of how it fits into a Spring AI workflow using an external Reranking API:

```java
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.SearchRequest;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;
import java.util.List;
import java.util.stream.Collectors;

@Service
public class RerankingRagService {

    private final VectorStore vectorStore;
    private final ExternalRerankClient rerankClient; // E.g., Cohere or a local ONNX model service

    public RerankingRagService(VectorStore vectorStore, ExternalRerankClient rerankClient) {
        this.vectorStore = vectorStore;
        this.rerankClient = rerankClient;
    }

    public List<Document> getHighlyRelevantDocuments(String query) {
        
        // STAGE 1: Broad Retrieval
        // Fetch a larger pool of candidates (e.g., 20) using fast vector search
        List<Document> broadResults = vectorStore.similaritySearch(
            SearchRequest.query(query).withTopK(20)
        );

        // Extract just the text for the reranker
        List<String> documentTexts = broadResults.stream()
                .map(Document::getContent)
                .collect(Collectors.toList());

        // STAGE 2: Reranking
        // Send the query AND the 20 texts to the Cross-Encoder model.
        // It returns a list of scores or sorted indices.
        List<RerankedResult> scoredResults = rerankClient.rerank(query, documentTexts);

        // Filter and return only the top 3 highest-scoring documents
        return scoredResults.stream()
                .sorted((a, b) -> Double.compare(b.score(), a.score()))
                .limit(3)
                .map(res -> broadResults.get(res.originalIndex()))
                .collect(Collectors.toList());
    }
}
```

### Conclusion

Reranking adds complexity and slight latency to a RAG pipeline, but it is often the single most impactful architectural change a team can make to eliminate hallucinations and ensure the LLM receives the absolute best context possible.
