# HyDE & Multi-Query Retrieval
Standard RAG often fails when the user's query is poorly phrased or very short. Explain advanced retrieval strategies like HyDE (Hypothetical Document Embeddings) and Multi-Query Retrieval to solve this issue in Java applications.

---ANSWER---

Standard ("naive") RAG takes the user's exact query string, converts it to a vector, and performs a similarity search. This assumes the query itself is semantically similar to the target document. Often, it is not. A user might type a vague, two-word query (e.g., "vacation policy"). The target document is a 5-page legal HR document. The semantic overlap is low, leading to poor retrieval.

Advanced retrieval strategies use an LLM to "fix" or "expand" the query *before* searching the Vector Database.

**1. Multi-Query Retrieval (Query Expansion)**
This strategy assumes that a single phrasing of a question might miss relevant documents due to slight variations in terminology.
*   **The Process:** Before querying the vector store, the Java application sends the user's original query to an LLM with a prompt like: *"Generate 3 different versions of this search query from different perspectives."*
*   **Execution:** If the user asked "How to reset DB", the LLM might generate: 1. "Steps to restart PostgreSQL", 2. "Database reboot instructions", 3. "Clear and initialize DB connection".
*   **Search:** The application performs a vector search for *all 4* queries (the original + the 3 generated). The results are merged, deduplicated, and passed as context to the final generation step. This significantly increases recall.

**2. HyDE (Hypothetical Document Embeddings)**
HyDE attacks the problem that a short question is structurally and semantically different from a long, detailed answer document.
*   **The Process:** The application intercepts the user's query and asks the LLM to write a *hypothetical, fake answer* to that question, without any context.
*   **Execution:** User asks: "vacation policy". The LLM (which might hallucinate) generates a fake document: *"The company offers 20 days of PTO. Employees must request time off via the HR portal two weeks in advance..."*
*   **Search:** The crucial step: The Java application embeds this *fake document* (not the user's short query) and searches the Vector DB. Because the fake document looks, sounds, and structures itself like the real target document, the vector similarity search is vastly more accurate.

**Implementation in Java:**
Frameworks like LangChain4j provide abstractions (like `QueryTransformer` or `RetrievalAugmentor`) that allow you to plug these strategies into the pipeline. Spring AI developers often implement these as separate, explicit LLM calls within their service layer orchestrator before calling the `VectorStore`.

These advanced techniques trade latency (requiring an extra LLM call before the search) for significantly higher retrieval accuracy.

### Examples
```java
// Conceptual Multi-Query Implementation
@Service
public class AdvancedRetrievalService {
    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public List<Document> multiQuerySearch(String userQuery) {
        // 1. Ask LLM to expand query
        String prompt = "Generate 3 alternative phrasings for this search query, separated by commas: " + userQuery;
        String expanded = chatClient.prompt().user(prompt).call().content();
        
        List<String> allQueries = new ArrayList<>(Arrays.asList(expanded.split(",")));
        allQueries.add(userQuery);

        // 2. Search for all variations and deduplicate
        Set<Document> uniqueResults = new HashSet<>();
        for (String q : allQueries) {
            uniqueResults.addAll(vectorStore.similaritySearch(q.trim()));
        }
        
        return new ArrayList<>(uniqueResults);
    }
}
```

### Life Analogy
Standard RAG is like looking for a book on "Baking" by only looking for the word "Baking" in the index.
**Multi-Query** is asking a librarian for synonyms first. They tell you to also look up "Pastry," "Oven techniques," and "Dough preparation," increasing your chances of finding the right section.
**HyDE** is sitting down and writing a fake recipe for a cake yourself, even if you don't know the exact measurements. You take your fake recipe to the librarian and say, "Find me a book that contains a chapter that looks exactly like this." The librarian can match the structure and terminology much better than if you just said "cake."

### Key Points
- Naive RAG fails on short or poorly phrased queries.
- Multi-Query uses an LLM to generate alternative phrasings of the query to cast a wider net.
- HyDE uses an LLM to generate a hypothetical answer, and uses the vector of that answer to find the real document.
- Both techniques improve recall accuracy at the cost of increased latency (an extra LLM call).
