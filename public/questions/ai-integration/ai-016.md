# HyDE & Multi-Query Retrieval

Standard Retrieval-Augmented Generation (RAG) relies on converting a user's query into a vector and finding similar document chunks in a vector database. However, this "naive" approach often fails. A user's short query (e.g., "Kafka timeout error") is semantically very different from the detailed technical documentation detailing the solution. 

To bridge this semantic gap, Advanced RAG techniques modify or expand the query *before* it hits the vector database. Two of the most powerful techniques are **Multi-Query Retrieval** and **HyDE (Hypothetical Document Embeddings)**.

### 1. Multi-Query Retrieval

Users often phrase questions poorly or use specific terminology that doesn't match the documentation. Multi-Query uses an LLM to rewrite the user's single query into multiple different variations from different perspectives. 

**The Workflow:**
1. User asks: "How to fix Spring Boot OOM?"
2. LLM rewrites this into 3 queries:
   - "How to fix Spring Boot OOM?" (Original)
   - "Troubleshooting OutOfMemoryError in Spring Java applications."
   - "Heap space configuration and memory leaks in Spring Boot."
3. The system embeds all three queries, searches the vector DB for each, and combines the deduplicated results.
4. The rich combined context is sent to the final generation LLM.

**Why it works:** It increases the "Recall" (the likelihood of finding the right document) by covering various synonyms and phrasings.

### 2. HyDE (Hypothetical Document Embeddings)

HyDE is a fascinating approach that operates on the premise that a query and its answer exist in different semantic spaces. 

**The Workflow:**
1. The user asks a question.
2. The system prompts an LLM: *"Write a hypothetical, plausible answer to this question. It doesn't matter if the facts are wrong, just mimic the style and structure of a real document."*
3. The LLM generates a "fake" document.
4. **Crucial Step:** The system embeds the *fake document* (not the user's query) and uses that vector to search the database.
5. The retrieved real documents are then used to answer the query accurately.

**Why it works:** It transforms a short, poorly phrased question into a rich, document-like paragraph, closing the semantic gap and matching perfectly with the actual documents in the vector store.

### Implementing Advanced Retrieval in Java

While LangChain4j and Spring AI provide abstractions for standard retrieval, advanced techniques often require custom orchestration using the framework's building blocks.

Here is a conceptual implementation of Multi-Query Retrieval using Spring AI:

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.document.Document;
import org.springframework.ai.vectorstore.SearchRequest;
import org.springframework.ai.vectorstore.VectorStore;
import org.springframework.stereotype.Service;

import java.util.ArrayList;
import java.util.List;
import java.util.stream.Collectors;

@Service
public class AdvancedRetriever {

    private final ChatClient chatClient;
    private final VectorStore vectorStore;

    public AdvancedRetriever(ChatClient chatClient, VectorStore vectorStore) {
        this.chatClient = chatClient;
        this.vectorStore = vectorStore;
    }

    public List<Document> multiQueryRetrieve(String originalQuery) {
        // 1. Generate Query Variations
        String prompt = "You are an AI assistant. Your task is to generate 3 different versions " +
                        "of the given user question to retrieve relevant documents from a vector database. " +
                        "Provide these alternative questions separated by newlines. Question: " + originalQuery;

        String variationsResponse = chatClient.prompt().user(prompt).call().content();
        List<String> queries = List.of(variationsResponse.split("\n"));
        
        List<Document> allRetrievedDocs = new ArrayList<>();

        // 2. Execute parallel searches for each variation (simplified to sequential here)
        for (String query : queries) {
            if (!query.isBlank()) {
                allRetrievedDocs.addAll(
                    vectorStore.similaritySearch(SearchRequest.query(query).withTopK(2))
                );
            }
        }

        // 3. Deduplicate based on Document ID
        return allRetrievedDocs.stream()
                .distinct() // Requires Document to have equals/hashCode implemented correctly
                .collect(Collectors.toList());
    }
}
```

### Trade-offs

The primary cost of Advanced RAG techniques is **Latency and Cost**. Both HyDE and Multi-Query require an additional LLM call *before* the retrieval even happens. This can add seconds to the response time. They are best utilized when accuracy is far more important than raw speed, or when dealing with highly technical, domain-specific corpora where naive search fails.
