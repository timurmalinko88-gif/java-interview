# Local Embeddings models (Ollama)
Why might an enterprise choose to use local embedding models via Ollama instead of a cloud provider like OpenAI? How is this integrated in a Java application?

---ANSWER---

In enterprise Java applications involving Retrieval-Augmented Generation (RAG), embedding models are used to convert proprietary text into vector representations. While cloud providers like OpenAI (e.g., `text-embedding-ada-002` or `text-embedding-3-small`) offer excellent performance, many enterprises choose to run local embedding models using tools like **Ollama**.

**Reasons for Choosing Local Embeddings:**

1.  **Data Privacy and Security:** This is the most common driver. Sending highly sensitive corporate data, PII (Personally Identifiable Information), or proprietary source code to a third-party cloud provider for embedding is often prohibited by enterprise security policies or compliance regulations (GDPR, HIPAA). Local models ensure that data never leaves the corporate network.
2.  **Cost Predictability:** Cloud APIs charge per token. If an application needs to embed massive datasets (e.g., millions of internal documents) initially and keep them updated continuously, the API costs can become astronomical and unpredictable. Running a local model on enterprise hardware incurs a fixed infrastructure cost, effectively making the marginal cost of embedding a new document zero.
3.  **Latency Avoidance:** While cloud APIs are generally fast, they are still subject to network latency, rate limits, and occasional outages. For high-throughput ingestion pipelines, generating embeddings locally can sometimes be faster, avoiding round-trip times and API throttling.
4.  **Specialized Models:** Enterprises can use specific open-source embedding models (like `nomic-embed-text` or `all-minilm`) that are fine-tuned for particular domains, rather than relying on a cloud provider's general-purpose model.

**Integration in Java (via Spring AI or LangChain4j):**

Ollama provides a lightweight runtime to host LLMs and embedding models locally. It exposes a REST API. Integrating this into a Java application is straightforward, as frameworks provide dedicated clients for Ollama.

To integrate, you first run Ollama on a server (or via Testcontainers for testing) and pull an embedding model: `ollama pull nomic-embed-text`.

In **Spring AI**, you add the `spring-ai-ollama-spring-boot-starter` dependency. The framework auto-configures an `OllamaEmbeddingClient`. You simply point the application properties to the Ollama server URL and specify the model name. The application code using the `EmbeddingClient` interface remains exactly the same as if it were using OpenAI, showcasing the power of Spring AI's abstractions.

In **LangChain4j**, you configure an `OllamaEmbeddingModel` with the base URL and model name. 

When you index documents or query the vector database, the Java framework sends the text chunks to your local Ollama instance via HTTP, which returns the vector arrays, completely bypassing the public internet.

### Examples
```java
// application.yml for Spring AI
/*
spring:
  ai:
    ollama:
      base-url: http://localhost:11434
      embedding:
        model: nomic-embed-text
*/

// Java Code (Identical to cloud provider usage)
@Service
public class LocalEmbeddingService {
    private final EmbeddingClient embeddingClient; // Injected OllamaEmbeddingClient

    public void embedAndStore(List<Document> docs) {
        // Text is sent to local Ollama, not OpenAI
        List<List<Double>> vectors = embeddingClient.embed(
            docs.stream().map(Document::getContent).toList()
        );
        // ... store vectors ...
    }
}
```

### Life Analogy
Using OpenAI for embeddings is like sending all your company's sensitive mail to a highly efficient external translation service to be categorized, and then trusting them to send the categorized mail back without keeping copies. 
Using Ollama for local embeddings is like hiring an in-house translator to work on a secure laptop inside your own office. It might require buying the laptop (infrastructure), but the sensitive documents never leave the building (data privacy).

### Key Points
- Data Privacy, Compliance, and Cost are the primary drivers for local embeddings.
- Ollama makes hosting open-source embedding models simple via a REST API.
- Spring AI and LangChain4j provide out-of-the-box clients for Ollama.
- Because of interface abstractions, switching from a cloud provider to Ollama requires only configuration changes, not code rewrites.
