# Structured Output (BeanOutputConverter)

One of the biggest challenges in integrating Large Language Models (LLMs) into traditional Java enterprise applications is the mismatch in data formats. LLMs naturally output unstructured, free-flowing text, whereas Java applications expect highly structured, strictly typed objects (POJOs/Beans). 

To solve this, frameworks provide mechanisms to coerce the LLM into returning specific formats (like JSON) and then automatically deserialize that data into Java objects. In Spring AI, this is handled by **Output Converters**, most notably the `BeanOutputConverter`.

### How Structured Output Works

Achieving structured output requires coordination between the prompt and the application code:
1. **Prompt Engineering:** The application must explicitly instruct the LLM to format its response as JSON that matches a specific schema.
2. **Model Capabilities:** Many modern models (like GPT-4) support specific "JSON Modes" or "Structured Outputs" natively via API flags, guaranteeing valid JSON.
3. **Parsing:** The application receives the JSON string and uses a library (like Jackson) to convert it into a Java Record or Class.

### Using BeanOutputConverter in Spring AI

Spring AI simplifies this entire process with `BeanOutputConverter`. It automatically generates the JSON schema instructions for the prompt and handles the deserialization of the response.

#### 1. Define the Target Record/Class

First, define the Java Record that represents the structure you want the LLM to return.

```java
public record MovieReview(
    String title,
    int releaseYear,
    double score,
    String briefSummary,
    List<String> keyActors
) {}
```

#### 2. Implement the Converter in the Controller

Inject the `ChatClient` and configure it to use the converter.

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.converter.BeanOutputConverter;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

@RestController
public class MovieController {

    private final ChatClient chatClient;

    public MovieController(ChatClient.Builder chatClientBuilder) {
        this.chatClient = chatClientBuilder.build();
    }

    @GetMapping("/analyze-movie")
    public MovieReview analyzeMovie(@RequestParam String movieName) {
        // 1. Initialize the converter for the specific class
        BeanOutputConverter<MovieReview> converter = 
            new BeanOutputConverter<>(MovieReview.class);

        // 2. Get the automatically generated format instructions.
        // This generates a string like: "Please provide the output in JSON format matching this schema..."
        String formatInstructions = converter.getFormat();

        // 3. Create the prompt, injecting the user request and the format instructions
        String prompt = "Analyze the movie: {movieName}. {formatInstructions}";

        // 4. Call the LLM
        String rawResponse = chatClient.prompt()
                .user(u -> u.text(prompt)
                            .param("movieName", movieName)
                            .param("formatInstructions", formatInstructions))
                .call()
                .content();

        // 5. Convert the raw JSON string back into a Java Record
        return converter.convert(rawResponse);
    }
}
```

### Advanced Considerations

1. **Complex Schemas:** `BeanOutputConverter` handles complex nested objects, enums, and lists seamlessly.
2. **JSON Mode vs. Structured Outputs:** While `BeanOutputConverter` relies on prompt engineering, newer APIs (like OpenAI's structured outputs) allow passing a strict JSON Schema directly into the API configuration layer, guaranteeing 100% adherence to the schema without relying purely on prompt instructions. Spring AI supports these native features depending on the underlying `ChatModel` implementation.
3. **Error Handling:** Always anticipate deserialization errors. If the LLM hallucinates an invalid JSON structure or uses the wrong data type (e.g., a string instead of an int), the `convert()` method will throw an exception. Implementing retry logic or fallback prompts is a best practice for production robustness.
