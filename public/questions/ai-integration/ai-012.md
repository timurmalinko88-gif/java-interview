# Streaming responses (Flux/SSE)

A defining characteristic of Large Language Models (LLMs) is their high latency. Generating a comprehensive, multi-paragraph answer can take several seconds. If a web application uses a traditional synchronous HTTP request/response model, the user stares at a loading spinner, creating a poor user experience.

The industry standard solution is **Streaming**. By streaming the response, tokens (words or parts of words) are sent to the client as soon as they are generated by the model, creating the "typing effect" seen in ChatGPT.

In the Java and Spring ecosystem, this is achieved using **Reactive Programming (Project Reactor / WebFlux)** and **Server-Sent Events (SSE)**.

### How AI Streaming Works

1. **LLM API:** The backend Java application connects to the AI provider (e.g., OpenAI) requesting a stream. The provider returns a stream of chunks.
2. **Reactive Backend:** Frameworks like Spring AI use Project Reactor (`Flux<String>`) to consume these chunks asynchronously and non-blockingly.
3. **SSE to Client:** The Spring Web or WebFlux controller returns the `Flux` to the frontend using the `text/event-stream` content type (SSE).
4. **Frontend Rendering:** The browser reads the stream event-by-event and updates the DOM dynamically.

### Implementing Streaming with Spring AI

Spring AI simplifies this drastically. Its `ChatClient` has native support for returning reactive `Flux` objects.

#### 1. Backend Controller

You can use standard Spring MVC or Spring WebFlux. When a controller method returns a `Flux` with `MediaType.TEXT_EVENT_STREAM_VALUE`, Spring automatically manages the SSE protocol connection.

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.http.MediaType;
import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;
import reactor.core.publisher.Flux;

@RestController
public class StreamingChatController {

    private final ChatClient chatClient;

    public StreamingChatController(ChatClient.Builder chatClientBuilder) {
        this.chatClient = chatClientBuilder.build();
    }

    @GetMapping(value = "/chat/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> streamChat(@RequestParam String message) {
        
        // Instead of .call().content(), we use .stream().content()
        // This returns a Flux<String> representing the continuous flow of tokens.
        return chatClient.prompt()
                .user(message)
                .stream()
                .content();
    }
}
```

#### 2. Frontend Consumption (JavaScript)

On the client side, you cannot use a standard `fetch` with `await response.json()`. You must use the `EventSource` API or process the `ReadableStream` directly.

```javascript
// Using basic EventSource (SSE)
const eventSource = new EventSource('/chat/stream?message=Write a story');

eventSource.onmessage = function(event) {
    // Append the newly arrived token to the UI
    document.getElementById('chatBox').innerHTML += event.data;
};

eventSource.onerror = function(error) {
    console.error("Stream closed or failed", error);
    eventSource.close();
};
```

### Challenges and Best Practices

1. **Exception Handling:** Handling errors in a reactive stream can be complex. If the LLM throws a timeout halfway through generating a response, the `Flux` will emit an `onError` signal. The backend must gracefully close the SSE connection, and the frontend must detect this and notify the user.
2. **Backpressure:** Project Reactor handles backpressure automatically. If the client (browser) is on a slow network and cannot consume the SSE events fast enough, Reactor will slow down the consumption from the LLM, preventing OutOfMemory errors on the server.
3. **Memory/History Management:** When streaming, you don't have the complete final string until the stream completes. You must hook into the stream's completion event (e.g., using `doOnComplete` or collecting the elements) to save the final generated message into your `ChatMemory` or database.
