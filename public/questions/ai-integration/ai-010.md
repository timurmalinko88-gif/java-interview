# ChatMemory & persistence

A fundamental characteristic of Large Language Models (LLMs) via REST APIs is that they are completely **stateless**. If you ask an LLM "What is my name?" and then subsequently ask "What did I just ask you?", the LLM will fail to answer unless the entire context of the conversation is sent along with every single request.

In Java applications, managing this conversational context is known as **Chat Memory**.

### Concepts of Chat Memory

To maintain a conversation, the application must store the history of interactions (User messages and Assistant responses) and prepend them to the current prompt. However, LLMs have strict context window limits (token limits). Therefore, memory management strategies are required:

1. **Message Window Memory:** Keeps only the last *N* messages (e.g., the last 10 interactions), discarding older ones.
2. **Token Window Memory:** Keeps as many messages as possible up to a specific token limit (e.g., keep history until it hits 4000 tokens).
3. **Summary Memory:** Uses the LLM itself to periodically summarize the older history into a compact paragraph, saving token space while retaining key facts.

### Implementing Chat Memory

Both Spring AI and LangChain4j offer abstractions for managing chat memory. LangChain4j has particularly robust support for this out-of-the-box.

#### LangChain4j Example

In LangChain4j, memory is managed via the `ChatMemory` interface, often tied to a specific session or user ID using a `ChatMemoryProvider`.

```java
import dev.langchain4j.memory.ChatMemory;
import dev.langchain4j.memory.chat.MessageWindowChatMemory;
import dev.langchain4j.service.spring.AiService;
import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;

// 1. Define the AI Service
@AiService
public interface PersonalAssistant {
    // The @MemoryId annotation tells the framework how to segment memory per user/session
    String chat(@MemoryId String userId, @UserMessage String userMessage);
}

// 2. Configure the Memory Provider
@Configuration
public class AiConfig {

    @Bean
    public ChatMemoryProvider chatMemoryProvider() {
        // Provides a unique MessageWindowChatMemory (last 10 messages) for each User ID
        return memoryId -> MessageWindowChatMemory.builder()
                .id(memoryId)
                .maxMessages(10)
                .build();
    }
}
```

In a controller, you simply pass the Session ID or User ID:
```java
// User 1 says "My name is Alice"
assistant.chat("user-1", "My name is Alice");

// User 1 says "What is my name?". The LLM knows because ChatMemory is injected!
String response = assistant.chat("user-1", "What is my name?"); // Replies: Alice
```

### Persistence of Chat Memory

By default, implementations like `MessageWindowChatMemory` store data in RAM. If the application restarts, the memory is wiped. In enterprise applications, chat memory must be persisted to a database (Redis, PostgreSQL, MongoDB).

LangChain4j handles this via the `ChatMemoryStore` interface.

```java
import dev.langchain4j.store.memory.chat.ChatMemoryStore;
import dev.langchain4j.data.message.ChatMessage;

public class DatabaseChatMemoryStore implements ChatMemoryStore {

    private final JdbcTemplate jdbcTemplate; // Or JPA Repository

    public DatabaseChatMemoryStore(JdbcTemplate jdbcTemplate) {
        this.jdbcTemplate = jdbcTemplate;
    }

    @Override
    public List<ChatMessage> getMessages(Object memoryId) {
        // Query database for messages associated with memoryId
        // Deserialize JSON strings back into ChatMessage objects
        return fetchFromDb(memoryId.toString());
    }

    @Override
    public void updateMessages(Object memoryId, List<ChatMessage> messages) {
        // Serialize messages to JSON and update the database row for memoryId
        saveToDb(memoryId.toString(), messages);
    }

    @Override
    public void deleteMessages(Object memoryId) {
        // Delete row from database
        deleteFromDb(memoryId.toString());
    }
}
```

You then configure the `MessageWindowChatMemory` builder to use this store:
```java
MessageWindowChatMemory.builder()
    .id(memoryId)
    .maxMessages(20)
    .chatMemoryStore(new DatabaseChatMemoryStore(jdbcTemplate))
    .build();
```

By effectively managing and persisting Chat Memory, Java applications can provide seamless, continuous, and highly personalized AI interactions across sessions and application restarts.
