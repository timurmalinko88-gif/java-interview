# RAG Evaluation metrics

Building a Retrieval-Augmented Generation (RAG) application in Java is only the first step. The hardest part of productionizing AI is answering the question: *"How do we know if it's actually working well?"* 

Because LLM outputs are non-deterministic and natural language-based, traditional unit tests (`assertEquals(expected, actual)`) fail. Instead, the industry relies on specialized **RAG Evaluation Metrics** to assess the quality of the pipeline.

### The RAG Triad

A RAG pipeline consists of a Retriever (which finds documents) and a Generator (which writes the answer). To evaluate the system holistically, we measure three core relationships, often called the **RAG Triad**.

#### 1. Context Relevance (Retrieval Quality)
* **What it measures:** Did the system retrieve documents that are actually relevant to the user's question?
* **Why it matters:** If the retriever brings back irrelevant garbage, the LLM cannot answer the question.
* **How it fails:** The vector database returns chunks that use similar words but have a different semantic meaning.

#### 2. Groundedness / Faithfulness (Hallucination Check)
* **What it measures:** Is the LLM's final answer fully supported by the retrieved context? 
* **Why it matters:** We use RAG to prevent hallucinations. If the LLM brings in outside knowledge or invents facts not present in the retrieved documents, the Groundedness score drops.
* **How it fails:** The LLM ignores the retrieved context and answers from its pre-trained memory.

#### 3. Answer Relevance (End-to-End Quality)
* **What it measures:** Does the final answer directly address the user's original question?
* **Why it matters:** An answer can be completely grounded in the context but still be unhelpful to the user.
* **How it fails:** The system retrieves documents about "Java 17 features", the LLM accurately summarizes them (High Groundedness), but the user's question was "How to install Java 17". (Low Answer Relevance).

### LLM-as-a-Judge

How do we actually calculate these scores programmatically? The standard approach is **LLM-as-a-Judge**. You use a powerful model (like GPT-4) to evaluate the output of your RAG pipeline based on strict grading prompts.

Libraries like **Ragas** or **TruLens** (in Python) are industry standards for this. In the Java ecosystem, developers often implement custom LLM-as-a-judge pipelines using Spring AI or LangChain4j.

#### Implementing a Groundedness Check in Java

Here is an example of how you might write an evaluator using Spring AI to check for Groundedness in a CI/CD pipeline or testing suite.

```java
import org.springframework.ai.chat.client.ChatClient;
import org.springframework.ai.converter.BeanOutputConverter;
import org.springframework.stereotype.Service;

@Service
public class RagEvaluator {

    private final ChatClient evalClient; // Configured with a highly capable model like GPT-4

    public RagEvaluator(ChatClient.Builder builder) {
        this.evalClient = builder.build();
    }

    record EvaluationResult(boolean isGrounded, String reasoning, double score) {}

    public EvaluationResult evaluateGroundedness(String retrievedContext, String generatedAnswer) {
        BeanOutputConverter<EvaluationResult> converter = new BeanOutputConverter<>(EvaluationResult.class);

        String prompt = """
            You are an expert grading system.
            Evaluate whether the GENERATED_ANSWER is entirely supported by the RETRIEVED_CONTEXT.
            If the answer contains any information not present in the context, it is not grounded.
            
            RETRIEVED_CONTEXT:
            {context}
            
            GENERATED_ANSWER:
            {answer}
            
            {format}
            """;

        String jsonResponse = evalClient.prompt()
                .user(u -> u.text(prompt)
                        .param("context", retrievedContext)
                        .param("answer", generatedAnswer)
                        .param("format", converter.getFormat()))
                .call()
                .content();

        return converter.convert(jsonResponse);
    }
}
```

### Continuous Evaluation

Evaluating RAG is not a one-time task. Best practices dictate:
1. **Golden Datasets:** Create a curated dataset of 100+ Question/Context/Answer pairs that humans have verified.
2. **Regression Testing:** Run the LLM-as-a-judge evaluator against the Golden Dataset every time you change the chunk size, embedding model, or prompt template.
3. **Production Monitoring:** Sample production traffic and asynchronously run evaluations to detect drift or degradation in user interactions over time.
