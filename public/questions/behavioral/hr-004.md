# Biggest production failure/bug you caused

## Framework
Interviewers want to see accountability, how you handle pressure, and your ability to learn from mistakes.

*   **Situation:** Briefly describe the critical system and the failure.
*   **Task:** Own the mistake. Explain what you were trying to do.
*   **Action:** Detail how you immediately responded to mitigate the damage.
*   **Result:** Explain the post-mortem learnings and the permanent processes you put in place to prevent it from happening again.

## Real-world Java Engineering Scenarios

### Scenario 1: The Infinite Loop Database Outage
*   **Situation:** A bug in my newly deployed batch job caused an infinite loop in a Spring Data JPA repository method.
*   **Task:** I was implementing a retry mechanism for failed payments, but my exit condition logic was flawed.
*   **Action:** As soon as monitoring alerts fired indicating 100% database CPU, I identified my service as the culprit and immediately rolled back the deployment.
*   **Result:** The system recovered within 10 minutes. Afterwards, I wrote a detailed post-mortem, added circuit breakers using Resilience4j, and implemented stricter integration tests for boundary conditions.

### Scenario 2: NullPointerException in Core Checkout Flow
*   **Situation:** During Black Friday, a NullPointerException in a new pricing service I deployed caused 5% of checkouts to fail.
*   **Task:** I had missed a null check for an optional promotional code object in the JSON payload.
*   **Action:** I quickly checked the Kibana logs, pinpointed the NPE, and deployed a hotfix within 30 minutes utilizing `Optional<T>` to handle the missing data gracefully.
*   **Result:** To prevent this, I championed the adoption of strict static analysis tools (SonarQube) and made nullability annotations (`@NonNull`) mandatory across our Java codebase.

### Scenario 3: Memory Leak from Unclosed Resources
*   **Situation:** A microservice I wrote began crashing with OutOfMemory errors every 48 hours in production.
*   **Task:** I had opened a custom `InputStream` to read large files from S3 but failed to close it in a `finally` block.
*   **Action:** I used a profiler to find the dangling references, pushed a fix using Java's try-with-resources statement, and restarted the pods.
*   **Result:** I learned a hard lesson about resource management. I updated our code review checklist to specifically verify resource closures and added load testing to our staging environment.

## Red Flags (What NOT to say)
*   **"I've never caused a bug."** This shows a lack of experience, lack of awareness, or dishonesty.
*   **Blaming QA or other devs:** "QA didn't catch it" or "My colleague reviewed it." Take absolute ownership.
*   **Focusing on the disaster, not the fix:** Don't dwell on how bad it was; focus heavily on the mitigation and the lessons learned.
*   **Failing to learn:** Mentioning a bug but providing no systemic changes (like adding tests or CI/CD checks) to prevent its recurrence.
