### Scoped Values vs ThreadLocal

As Java embraces massive concurrency with Virtual Threads, older constructs like `ThreadLocal` start to show significant limitations. To address this, Java 21 introduces **Scoped Values** (as a preview feature) to provide a modern, efficient alternative for sharing immutable data across a thread and its child threads.

#### The Problem with `ThreadLocal`
For decades, `ThreadLocal` variables have been the standard way to store context implicitly—such as database connections, transaction IDs, or security principals—so that deep method calls don't need to pass these values explicitly as parameters. However, `ThreadLocal` has three major flaws, especially in a world with millions of Virtual Threads:

1. **Mutability:** `ThreadLocal` variables are mutable. Any code that has access to the `ThreadLocal` reference can call `set()` and change the value. This makes data flow difficult to reason about and can lead to bugs if a value is accidentally modified or not cleared properly.
2. **Unbounded Lifetime:** The value inside a `ThreadLocal` persists until the thread dies or `remove()` is explicitly called. If a developer forgets to call `remove()`, the data remains in memory, leading to memory leaks (especially in application servers that use thread pools).
3. **Expensive Inheritance:** When using `InheritableThreadLocal` to pass data to child threads, the child thread must allocate a new map and copy the parent's values. When dealing with millions of Virtual Threads, the overhead of copying `ThreadLocal` maps for every new thread becomes a massive performance and memory bottleneck.

#### The Solution: Scoped Values (`ScopedValue`)
Scoped Values allow a value to be safely and efficiently shared with methods invoked within a bounded lexical scope, without passing it as a method parameter.

Key differences from `ThreadLocal`:
- **Immutable:** Once a Scoped Value is bound, its value cannot be changed within that scope. There is no `set()` method.
- **Bounded Lifetime:** A Scoped Value is only bound for the duration of a specific `Runnable` or `Callable`. Once the execution exits that scope, the value is automatically unbound and garbage collected. There is no need for a `remove()` method, eliminating memory leaks.
- **Efficient Inheritance:** Scoped Values integrate perfectly with Structured Concurrency (`StructuredTaskScope`). When child threads are spawned, they share the parent's Scoped Values without copying the underlying maps, drastically reducing memory overhead.

#### Code Example: How to use Scoped Values

```java
import java.util.concurrent.StructuredTaskScope;

public class ScopedValuesExample {

    // Define a ScopedValue (usually static final)
    private static final ScopedValue<String> CURRENT_USER = ScopedValue.newInstance();

    public void processRequest(String userId) {
        // Bind the value and execute a block of code within that scope.
        // The value is ONLY accessible during the execution of the lambda.
        ScopedValue.where(CURRENT_USER, userId)
                   .run(() -> {
                       System.out.println("Processing started for: " + CURRENT_USER.get());
                       performBusinessLogic();
                   });
        // Here, CURRENT_USER is no longer bound.
    }

    private void performBusinessLogic() {
        // Deep within the call stack, we can retrieve the value without passing it as a parameter.
        // It is safely bound to the current execution scope.
        String user = CURRENT_USER.get();
        System.out.println("Business logic accessing user: " + user);
        
        // Demonstrating inheritance with StructuredTaskScope
        try (var scope = new StructuredTaskScope.ShutdownOnFailure()) {
            scope.fork(() -> {
                // Child threads automatically inherit the ScopedValue for free!
                System.out.println("Background task accessing user: " + CURRENT_USER.get());
                return null;
            });
            scope.join();
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    }

    public static void main(String[] args) {
        var example = new ScopedValuesExample();
        example.processRequest("Alice_Admin");
    }
}
```

#### Summary
`ThreadLocal` is mutable, unbounded, and scales poorly with Virtual Threads. `ScopedValue` is immutable, strictly bound to a lexical scope, and highly optimized for millions of threads, making it the preferred choice for implicit context passing in modern Java applications.
