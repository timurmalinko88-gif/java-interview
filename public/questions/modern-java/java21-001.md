### Virtual Threads vs OS Threads

Java 21 introduces Virtual Threads as part of Project Loom, marking a monumental shift in how concurrent applications are built on the JVM. To understand their impact, we must contrast them with traditional OS threads (platform threads).

#### The Traditional Approach: OS Threads (Platform Threads)
Before Java 21, `java.lang.Thread` was simply a wrapper around an operating system thread. These platform threads are heavy and expensive. When you create a platform thread, the OS must allocate a large block of memory (usually around 1MB or 2MB) for the thread stack and keep track of it in the kernel scheduler. Because of this heavy overhead, you cannot have millions of active platform threads in a typical application. Instead, you're usually limited to a few thousand threads before the system runs out of memory or spends all its time context-switching.

To work around this limitation, developers turned to thread pools (like `ExecutorService`) and reactive programming frameworks (like Project Reactor or RxJava). Thread pools recycle expensive threads, while reactive programming avoids blocking threads altogether by using asynchronous callbacks. However, reactive programming is notoriously difficult to debug and trace, and thread pools limit the maximum concurrency to the pool size.

#### The Modern Approach: Virtual Threads
Virtual threads are lightweight, user-mode threads managed by the JVM rather than the operating system. They are cheap to create, consume very little memory (starting at just a few hundred bytes), and can be instantiated by the millions. When a virtual thread makes a blocking I/O call (e.g., waiting for a database response or reading from a socket), the JVM automatically suspends the virtual thread and yields its underlying platform thread (called the "carrier thread") to another virtual thread. Once the I/O operation completes, the virtual thread is scheduled back onto an available carrier thread to resume execution.

This means you can write code in a simple, synchronous, blocking style—"one thread per request"—without paying the high cost of platform threads. You no longer need complex reactive pipelines to handle high concurrency. 

#### Code Example

```java
import java.time.Duration;
import java.util.concurrent.Executors;
import java.util.stream.IntStream;

public class VirtualThreadExample {
    public static void main(String[] args) {
        long start = System.currentTimeMillis();

        // Using virtual threads, we can easily run 10,000 concurrent tasks.
        // If we used Executors.newFixedThreadPool(10000) with platform threads, 
        // the application would likely crash or suffer severe performance degradation.
        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            IntStream.range(0, 10_000).forEach(i -> {
                executor.submit(() -> {
                    try {
                        // Simulating a blocking I/O operation (e.g., an HTTP call).
                        // The virtual thread unmounts from its carrier thread during this sleep.
                        Thread.sleep(Duration.ofSeconds(1));
                    } catch (InterruptedException e) {
                        Thread.currentThread().interrupt();
                    }
                });
            });
        } // The try-with-resources block waits for all submitted tasks to complete.

        long end = System.currentTimeMillis();
        System.out.println("Completed 10,000 tasks in " + (end - start) + " ms");
    }
}
```

#### Key Differences
1. **Creation Cost:** Platform threads are expensive and slow to create. Virtual threads are almost free.
2. **Scalability:** You can run thousands of platform threads, but millions of virtual threads.
3. **Pooling:** Platform threads should be pooled. Virtual threads should **never** be pooled; they are designed to be created per task and discarded.
4. **Blocking Behavior:** Blocking a platform thread wastes an OS thread. Blocking a virtual thread simply yields the carrier thread to another virtual thread, keeping CPU utilization high.

In summary, Virtual Threads combine the simplicity of blocking synchronous code with the scalability of non-blocking reactive systems, fundamentally changing how Java backend services are designed.
