### What are Unnamed Variables and Patterns?

Introduced as a preview feature in Java 21 (**JEP 443**) and finalized in Java 22 (**JEP 456**), **Unnamed Variables and Unnamed Patterns** allow developers to denote unused variables or pattern components using an underscore (`_`).

In previous versions of Java, declaring a variable required giving it an explicit identifier, even if that variable was never subsequently referenced. This frequently caused compiler warnings about unused variables, required workaround comments (e.g., `// ignored`), or bloated code syntax.

The underscore `_` informs the compiler that the variable or pattern binding is intentionally ignored and does not create a variable binding in local scope.

---

### Key Use Cases and Syntax

#### 1. Unused Exception Catch Blocks

When an exception type is caught solely to enforce control flow without referencing the exception instance:

```java
// Prior to Java 21
try {
    int number = Integer.parseInt(inputStr);
} catch (NumberFormatException ignored) {
    System.out.println("Invalid integer format");
}

// Java 21+ Unnamed Variable
try {
    int number = Integer.parseInt(inputStr);
} catch (NumberFormatException _) {
    System.out.println("Invalid integer format");
}
```

#### 2. Unused Loop Variables

When iterating over collections or ranges where the elements themselves are not needed:

```java
// Java 21+ Unnamed Loop Variable
int count = 0;
for (Order _ : orders) {
    count++;
}
```

#### 3. Lambda Parameters

When a multi-parameter lambda ignores one or more parameters:

```java
Map<String, Integer> map = new HashMap<>();
// Ignore key parameter in Map.merge or Map.forEach
map.computeIfAbsent("Key", _ -> 100);

// Ignore value parameter in Map.forEach
map.forEach((key, _) -> System.out.println("Key: " + key));
```

#### 4. Unnamed Record Pattern Components

When destructuring a record, you can ignore components you don't need without binding named variables:

```java
public record Point(int x, int y) {}
public record ColoredPoint(Point point, String color) {}

// Extract color only, ignoring Point coordinates completely
if (obj instanceof ColoredPoint(_, String color)) {
    System.out.println("Color: " + color);
}

// Extract x coordinate only, ignoring y coordinate and color
if (obj instanceof ColoredPoint(Point(int x, _), _)) {
    System.out.println("X coordinate: " + x);
}
```

#### 5. Pattern Matching in Switch

```java
public sealed interface Response permits Success, ErrorResponse {}
public record Success(String payload) implements Response {}
public record ErrorResponse(int code, String message) implements Response {}

public static void handleResponse(Response response) {
    switch (response) {
        case Success(String data) -> System.out.println("Data: " + data);
        case ErrorResponse(int code, _) -> System.out.println("Error code: " + code); // message ignored
    }
}
```

---

### Important Rules of `_`

1. **No Local Scope Binding**: The underscore `_` is not a variable name. You cannot read from `_` (e.g., `System.out.println(_)` causes a compilation error).
2. **Multiple Uses in Same Scope**: Because `_` does not introduce a named variable, you can use `_` multiple times within the exact same block or statement:

```java
// Valid in Java 21+: Multiple unnamed variables in one lambda
BiFunction<String, String, Integer> lengthSum = (_, _) -> 0;
```

---

### Summary of Benefits

- **Clearer Intent**: Explicitly communicates to code reviewers and static analysis tools that a parameter is intentionally unreferenced.
- **Cleaner Destructuring**: Simplifies pattern matching over complex records by omitting irrelevant fields.
- **Suppresses Compiler Warnings**: Eliminates static analyzer warnings for unused variables naturally.
