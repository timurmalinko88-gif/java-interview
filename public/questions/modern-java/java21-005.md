# Pattern Matching for switch
Describe how Pattern Matching for `switch` works in Java 21. How does it improve upon traditional `switch` statements and `instanceof` checks, and what are "guarded patterns" and "exhaustiveness"?

---ANSWER---

Prior to Java 21, the `switch` statement was highly restricted. It could only evaluate primitives, Strings, and Enums. If you had an object of an unknown type (like an `Object` or a generic interface) and needed to execute different logic based on its actual class, you had to use a long, ugly chain of `if-else` statements with `instanceof` checks and explicit casting. This was verbose, error-prone, and visually cluttered.

Java 21 finalizes Pattern Matching for `switch`, completely transforming the `switch` construct. You can now pass any object into a `switch` statement or expression and match it against type patterns directly in the `case` labels. When a type matches, the `switch` automatically casts the object to that type and binds it to a local pattern variable, removing the need for manual casting inside the block. This leads to significantly cleaner and more declarative code.

Additionally, this feature introduces two critical concepts: Guarded Patterns and Exhaustiveness.
**Guarded Patterns** allow you to add a boolean condition to a `case` label using the `when` keyword. The case will only match if both the type matches and the `when` condition evaluates to true. This eliminates nested `if` statements inside case blocks.
**Exhaustiveness** is a compile-time safety feature. When using a `switch` expression (which returns a value) or when switching over a `sealed` class hierarchy, the Java compiler enforces that every possible type is covered by the `case` labels. If you add a new subclass to a sealed interface but forget to update the `switch` block, the code will fail to compile. This eliminates a whole class of runtime bugs that occurred when `if-else` chains silently missed a new type.

### Examples
```java
// Pre-Java 21: Verbose if-else with manual casting
public String format(Object obj) {
    if (obj instanceof Integer i) {
        return String.format("int %d", i);
    } else if (obj instanceof String s && s.length() > 5) {
        return "Long String";
    } else if (obj instanceof String s) {
        return s;
    } else {
        return "Unknown";
    }
}

// Java 21: Pattern Matching for switch with Guarded Patterns
public String formatModern(Object obj) {
    return switch (obj) {
        case Integer i -> String.format("int %d", i);
        // Guarded pattern using the 'when' keyword
        case String s when s.length() > 5 -> "Long String";
        case String s -> s;
        // switch handles null safety natively as well
        case null -> "It's null";
        case default -> "Unknown";
    };
}
```

### Life Analogy
The old `if-else` approach is like a bouncer at a club checking IDs. They check if you are a VIP. If yes, they ask you for your name again, give you a VIP wristband, and let you in. Then they check if you are staff, ask for your name, and give you a staff badge.

Pattern Matching for `switch` is like an automated scanner. It scans your ID and instantly says, "You are a VIP named John" (binding the variable) and routes you to the VIP line. Guarded patterns add rules like, "You are a VIP named John AND you are wearing a tie (`when tie == true`), so you go to the super VIP lounge." Exhaustiveness is the manager ensuring the scanner has a rule for absolutely every type of person who could possibly walk up to the door.

### Key Points
- `switch` can now evaluate objects based on their actual type.
- Matches automatically cast and bind the object to a local variable, eliminating boilerplate.
- The `when` keyword allows for guarded conditions directly in the `case` label.
- The compiler enforces exhaustiveness, ensuring all possible subtypes (especially in sealed hierarchies) are handled.
