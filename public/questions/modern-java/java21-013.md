### Upgrading to Java 21: Key Considerations

Upgrading enterprise applications to **Java 21** (the Long-Term Support release following Java 17) unlocks major features such as Virtual Threads, Sequenced Collections, and Record Patterns. However, migrating from older JDK versions (especially Java 8 or Java 11, and even Java 17) requires awareness of critical compatibility shifts, JDK internal encapsulations, and runtime behavior changes.

---

### Critical Migration Caveats & Pitfalls

#### 1. Deprecation & Removal of Security Manager (JEP 411)

The legacy **Security Manager** has been deprecated for removal. In Java 21, calling `System.setSecurityManager()` throws a `UnsupportedOperationException` by default unless explicitly allowed via runtime flags.

```bash
# If legacy libraries rely on Security Manager, you must pass:
-Djava.security.manager=allow
```
*Recommendation*: Replace Security Manager checks with container isolation, OS-level security constraints, or security frameworks.

#### 2. Restricting Dynamic Loading of Java Agents (JEP 451)

Java 21 issues stern warnings (and in future releases will block by default) when Java Agents are attached dynamically to a running JVM using the Attach API.

```bash
# JVM Warning at runtime:
# WARNING: A Java agent has been loaded dynamically (...)
```
*Fix*: Attach Java agents at JVM launch time using `-javaagent:agent.jar` or explicitly enable dynamic agent loading with `-XX:+EnableDynamicAgentLoading`.

#### 3. Strong Encapsulation of JDK Internals

Starting in Java 16/17 and enforced further in Java 21, reflective access to JDK internal APIs (e.g., `sun.misc.Unsafe`, `com.sun.*`) is strictly blocked by default.

If legacy third-party dependencies call internal APIs reflectively, applications will throw `InaccessibleObjectException`:

```java
java.lang.reflect.InaccessibleObjectException: Unable to make field private final byte[] java.lang.String.value accessible: module java.base does not "opens java.lang" to unnamed module
```

*Remediation*:
- Upgrade third-party dependencies (Spring, Hibernate, ByteBuddy, Jackson, Lombok) to JDK 21 compatible versions.
- As a temporary workaround, open packages explicitly:

```bash
java --add-opens java.base/java.lang=ALL-UNNAMED -jar app.jar
```

#### 4. Virtual Thread Pitfalls

When adopting Virtual Threads (`Executors.newVirtualThreadPerTaskExecutor()`), common pitfalls include:

- **Virtual Thread Pinning**: Native method calls or `synchronized` blocks wrapping blocking I/O pin virtual threads to carrier OS threads. Replace `synchronized` with `ReentrantLock`.
- **Thread Pooling Anti-Pattern**: Never pool virtual threads! Creating pooled virtual thread executors (`Executors.newFixedThreadPool(10, Thread.ofVirtual().factory())`) defeats their design purpose. Always create a new virtual thread per task (`Thread.ofVirtual().start(...)`).
- **ThreadLocal Memory Leaks**: Holding large context objects in `ThreadLocal` variables when spawning millions of virtual threads causes severe heap consumption.

#### 5. Deprecation of Key Management / 3DES Algorithms

Legacy cryptographic algorithms like 3DES, RC4, and weak RSA keys (< 2048-bit) are disabled by default in Java 21 security providers. Connectors relying on legacy TLS 1.0/1.1 or weak SSL certificates will fail during handshake.

### Library & Framework Compatibility Matrix

To run smoothly on Java 21, check that your toolchain versions meet these minimum requirements:

| Tool / Framework | Minimum Version for Java 21 |
| :--- | :--- |
| **Spring Boot** | 3.2.0+ (requires Spring Framework 6.1+) |
| **Hibernate ORM** | 6.2.0+ |
| **ByteBuddy** | 1.14.9+ |
| **Lombok** | 1.18.30+ |
| **Mockito** | 5.5.0+ |
| **Gradle** | 8.5+ |
| **Apache Maven** | 3.9.0+ (with Compiler Plugin 3.11.0+) |

### Migration Checklist

1. **Run Dependency Analysis**: Use `jdeps --jdk-internals app.jar` to identify legacy internal API usages before upgrading.
2. **Upgrade Build Plugins**: Ensure Maven Compiler Plugin target is configured for Java 21:

```xml
<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <version>3.11.0</version>
    <configuration>
        <release>21</release>
    </configuration>
</plugin>
```

3. **Verify Thread Dumps**: Thread dump formats change when virtual threads are present (`jcmd <pid> Thread.dump_to_file -format=json threads.json`).
4. **Test GC Latency**: Evaluate switching from G1GC to Generational ZGC (`-XX:+UseZGC -XX:+ZGenerational`).
