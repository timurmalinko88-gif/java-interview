### Virtual Threads in Spring Boot 3

With Java 21, **Virtual Threads** (Project Loom, JEP 444) became production-ready. Spring Framework 6.1 and **Spring Boot 3.2+** introduced out-of-the-box support for virtual threads, changing how Java applications achieve massive concurrency for I/O-intensive workloads.

Historically, Java web servers (like embedded Tomcat in Spring Boot) operated on a **thread-per-request** model backed by platform OS threads. Because OS threads are expensive resource-wise (~1 MB stack per thread, context-switching overhead in kernel space), Tomcat default thread pools were capped (typically `max=200`). Under high concurrent I/O load (e.g., waiting for database queries or REST API downstream calls), these threads blocked, causing thread pool exhaustion.

---

### Enabling Virtual Threads in Spring Boot 3.2+

Enabling Virtual Threads in a Spring Boot 3 application requires Java 21+ runtime and a single configuration parameter in `application.properties`:

```properties
# Enable Virtual Threads in Spring Boot 3.2+
spring.threads.virtual.enabled=true
```

When this flag is set to `true`:
1. **Tomcat / Jetty Web Server**: Uses `Executors.newVirtualThreadPerTaskExecutor()` for processing incoming HTTP requests instead of a bounded OS thread pool.
2. **Spring `@Async` Task Execution**: Uses virtual threads for background tasks.
3. **Spring MVC Controller Dispatcher**: Dispatches controller handlers onto lightweight virtual threads.

### Code Example: Controller Comparison

```java
@RestController
@RequestMapping("/api")
public class ProductController {

    private final RestClient restClient;

    public ProductController(RestClient.Builder builder) {
        this.restClient = builder.baseUrl("https://api.inventory-service.com").build();
    }

    @GetMapping("/product/{id}")
    public ProductDetails getProduct(@PathVariable String id) {
        // Blocking I/O call: Under Virtual Threads, this unmounts the virtual thread 
        // from the underlying OS carrier thread rather than blocking the OS thread!
        Inventory inventory = restClient.get()
                .uri("/inventory/" + id)
                .retrieve()
                .body(Inventory.class);

        return new ProductDetails(id, inventory);
    }
}
```

### Performance Implications & Metrics

For **I/O-bound workloads** (applications spending significant time waiting for database queries, Redis calls, or downstream microservices), Virtual Threads provide massive performance gains:

1. **Throughput Scaling**: Serves thousands of concurrent connections using minimal OS threads (matching the available CPU core count).
2. **Resource Efficiency**: Virtual threads consume only a few hundred bytes of heap memory initial stack footprint compared to 1MB per platform thread.
3. **Imperative Simplicity**: Achieves the scale of reactive programming (`Spring WebFlux`) while retaining standard synchronous, imperative code (`Spring MVC`).

---

### Common Pitfalls and Bottlenecks

While Virtual Threads are powerful, improperly applying them can create new bottlenecks:

#### 1. Pinning Virtual Threads (`synchronized` blocks)
If a virtual thread enters a `synchronized` block/method or calls native C code while performing blocking I/O, it gets **pinned** to its carrier OS thread, preventing the carrier thread from serving other virtual threads.

*Remediation*: Replace `synchronized` blocks wrapping I/O calls with `java.util.concurrent.locks.ReentrantLock`:

```java
// BAD: Pins virtual thread during I/O
public synchronized String fetchData() {
    return restClient.get().uri("/data").retrieve().body(String.class);
}

// GOOD: Unmounts virtual thread cleanly
private final ReentrantLock lock = new ReentrantLock();
public String fetchDataSafe() {
    lock.lock();
    try {
        return restClient.get().uri("/data").retrieve().body(String.class);
    } finally {
        lock.unlock();
    }
}
```

#### 2. ThreadLocal Abuse
Because applications can easily launch millions of virtual threads, placing large objects in `ThreadLocal` context variables (e.g., heavy security context objects or reusable parser instances) can result in severe heap memory exhaustion.

#### 3. Database Connection Pool Exhaustion
Virtual threads solve CPU thread starvation, but they do **not** increase database pool capacities. If 10,000 virtual threads attempt to query a HikariCP pool capped at 10 connections simultaneously, the bottleneck shifts to database connection contention. Connection pool sizes must be configured appropriately.
