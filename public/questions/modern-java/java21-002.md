### Thread Pinning: `synchronized` vs `ReentrantLock` in Virtual Threads

While Virtual Threads are a revolutionary feature in Java 21, they are not entirely without pitfalls. One of the most critical concepts to understand when migrating to or using Virtual Threads is **Thread Pinning**.

#### What is Thread Pinning?
When a virtual thread executes, it is "mounted" onto a platform thread (known as the carrier thread). If the virtual thread performs a blocking I/O operation (like reading from a network socket), the JVM normally "unmounts" the virtual thread, freeing up the carrier thread to execute other virtual threads. 

However, in certain situations, a virtual thread cannot be unmounted. When this happens, the virtual thread is said to be **"pinned"** to its carrier thread. If a pinned virtual thread blocks (e.g., does I/O or sleeps), it blocks the underlying carrier thread as well. Because the JVM uses a small pool of carrier threads (equal to the number of CPU cores by default), if too many virtual threads get pinned and block, you can quickly exhaust the carrier thread pool, leading to a condition known as starvation and severely degrading the application's throughput.

#### When Does Thread Pinning Occur?
Currently, a virtual thread is pinned to its carrier thread in two main scenarios:
1. When executing code inside a `synchronized` block or method.
2. When executing a native method or a foreign function.

The restriction involving `synchronized` is particularly problematic because legacy Java code heavily relies on `synchronized` blocks for thread safety. If a virtual thread holds a monitor lock (via `synchronized`) and then performs a blocking I/O operation, it will pin the carrier thread.

#### The Solution: `ReentrantLock`
To avoid pinning issues, the recommended practice when working with Virtual Threads is to replace `synchronized` blocks with `java.util.concurrent.locks.ReentrantLock`. Unlike `synchronized`, acquiring a `ReentrantLock` does not pin the virtual thread. If a virtual thread blocks while holding a `ReentrantLock`, it will successfully unmount, allowing the carrier thread to serve other virtual threads.

#### Code Example: Pinning vs Unpinning

```java
import java.time.Duration;
import java.util.concurrent.Executors;
import java.util.concurrent.locks.ReentrantLock;

public class ThreadPinningExample {

    private final Object syncLock = new Object();
    private final ReentrantLock reentrantLock = new ReentrantLock();

    // Bad Practice with Virtual Threads: Using synchronized
    public void executeWithSynchronized() {
        synchronized (syncLock) {
            try {
                // Simulating blocking I/O inside a synchronized block.
                // This will PIN the virtual thread to the carrier thread!
                // If you do this concurrently across many virtual threads, 
                // you will exhaust the carrier thread pool.
                Thread.sleep(Duration.ofSeconds(2));
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            }
        }
    }

    // Good Practice with Virtual Threads: Using ReentrantLock
    public void executeWithReentrantLock() {
        reentrantLock.lock();
        try {
            // Simulating blocking I/O.
            // This is SAFE. The virtual thread will unmount normally.
            Thread.sleep(Duration.ofSeconds(2));
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            reentrantLock.unlock();
        }
    }

    public static void main(String[] args) {
        ThreadPinningExample example = new ThreadPinningExample();

        try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
            // Submitting many tasks that use ReentrantLock
            for (int i = 0; i < 100; i++) {
                executor.submit(example::executeWithReentrantLock);
            }
        }
    }
}
```

#### Diagnostic Tools
To detect thread pinning in your applications, Java provides a JDK Flight Recorder (JFR) event called `jdk.VirtualThreadPinned`. You can also start the JVM with `-Djdk.tracePinnedThreads=full` or `-Djdk.tracePinnedThreads=short` to get stack traces printed to standard output whenever a virtual thread blocks while pinned.

In conclusion, when adopting Virtual Threads, a careful review of legacy concurrent code is necessary. Replacing long-held `synchronized` blocks containing I/O operations with `ReentrantLock` is crucial for achieving the massive scalability that Virtual Threads promise.
