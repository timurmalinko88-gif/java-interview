# The Silent Web Leak

## Context
Your team has built a Spring Boot REST API. To avoid passing user authentication details through every method signature, a developer introduced a `UserContextHolder` using a `ThreadLocal`. 

After a few days in production, the server starts throwing `OutOfMemoryError: Java heap space`. Additionally, users are reporting a critical security bug: occasionally, User A sees User B's private account data.

## Code Snippet

```java
import org.springframework.stereotype.Component;
import javax.servlet.*;
import javax.servlet.http.HttpServletRequest;
import java.io.IOException;

// Context Holder
public class UserContextHolder {
    private static final ThreadLocal<User> userThreadLocal = new ThreadLocal<>();

    public static void setUser(User user) {
        userThreadLocal.set(user);
    }

    public static User getUser() {
        return userThreadLocal.get();
    }
}

// Spring Interceptor / Filter
@Component
public class AuthenticationFilter implements Filter {

    private final AuthService authService;

    public AuthenticationFilter(AuthService authService) {
        this.authService = authService;
    }

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) 
            throws IOException, ServletException {
            
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        String token = httpRequest.getHeader("Authorization");

        if (token != null) {
            User user = authService.authenticate(token);
            UserContextHolder.setUser(user);
        }

        // Continue processing the request
        chain.doFilter(request, response);
    }
}
```
Review the code above. Identify the bugs causing both the memory leak and the security vulnerability. Explain how the JVM handles threads in a web environment that leads to this outcome, and provide the corrected code.

---ANSWER---

### 2. Bugs & Issues Identified
- **Missing Cleanup:** The `ThreadLocal` is never cleared after the request finishes. 
- **Thread Pooling Mechanism:** Tomcat (Spring Boot's default embedded server) uses a **Thread Pool** to handle incoming HTTP requests. Threads are reused, not destroyed.
- **Memory Leak:** Because the thread stays alive in the pool, its `ThreadLocalMap` retains a strong reference to the `User` object. The Garbage Collector cannot reclaim these objects, eventually causing an `OutOfMemoryError`.
- **Security Vulnerability:** If a subsequent request arrives without an `Authorization` header, the `AuthenticationFilter` skips setting a new user. However, the thread might still hold the `User` object from a previous request. The application will process the new request using the previous user's identity.

- **GC Roots:** Explain that the alive Thread object acts as a GC root, preventing the garbage collection of the `User` object.
- **Data Contamination:** Clearly link the thread reuse to the security bug (cross-request state bleed).
- **The Fix:** Must use a `finally` block to guarantee cleanup (`ThreadLocal.remove()`).

### 4. Corrected Code

```java
@Component
public class AuthenticationFilter implements Filter {
    // ... setup omitted ...

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain) 
            throws IOException, ServletException {
            
        try {
            HttpServletRequest httpRequest = (HttpServletRequest) request;
            String token = httpRequest.getHeader("Authorization");

            if (token != null) {
                User user = authService.authenticate(token);
                UserContextHolder.setUser(user);
            }

            // Continue processing the request
            chain.doFilter(request, response);
            
        } finally {
            // CRITICAL FIX: Always clean up the ThreadLocal to prevent leaks and cross-request contamination
            UserContextHolder.userThreadLocal.remove();
        }
    }
}
```

### Life Analogy
**The Gym Locker:** Using `ThreadLocal` in a web server is like renting a gym locker. You put your clothes in it while you work out (process the request). If you don't empty the locker when you leave (clean up), the next person assigned that locker will find your clothes (security breach). Eventually, all lockers fill up with abandoned clothes, and the gym runs out of space (Memory Leak).

### Key Points
1.  **Thread Pooling:** Must explain that web servers reuse threads; they do not create a new thread per request.
