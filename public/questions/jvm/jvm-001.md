# Unintentional Retention: Memory Leaks in Java
Java has an automated Garbage Collector (GC) that periodically cleans up unreachable objects. Despite this, "memory leaks" are a common problem in Java applications. 

Explain how a memory leak can occur in Java. Provide a practical example involving standard collections, and briefly explain how to prevent it. How does this differ from a native memory leak in languages like C or C++?

---ANSWER---

In Java, a "memory leak" is actually **unintentional object retention**. The Garbage Collector only reclaims objects that are no longer reachable from GC Roots (like active threads, static variables, or local variables in active frames). If a program maintains a reference to an object that is no longer needed, the GC cannot collect it, leading to a gradual exhaustion of the heap space (`OutOfMemoryError: Java heap space`).

### Practical Example
A classic example is using a `HashMap` for caching data but forgetting to remove entries when they are no longer relevant.
If a `static List<Object>` or `static Map<Object, Object>` is used to keep track of processed events, and events are only added but never removed, the collection grows indefinitely.

### Native vs. Java Leak
- **C/C++ (Native)**: You allocate memory explicitly (`malloc`) and lose the pointer to it without calling `free`. The memory is completely unreachable by the program but still marked as in-use by the OS.
- **Java**: The memory is *reachable* by the application (a reference exists), but it is *logically* unused.

2. **GC Roots**: Mention that GC relies on reachability from GC roots; strong references prevent collection.
3. **Example**: Provide a concrete example, such as an unbound static cache or an un-cleared collection.
4. **Prevention**: Mention strategies like using `WeakHashMap`, setting references to `null`, or using bounded caches with eviction policies (like LRU).

### Life Analogy
Imagine a library (the JVM) where a librarian (the GC) walks around collecting books left on tables to put them back. However, if you hoard books in your locker (a static collection) and never read them again, the librarian isn't allowed to touch them. The library eventually runs out of books, not because they are lost, but because they are technically "checked out" forever.

### Key Points
1. **Definition**: Java memory leaks happen when objects are no longer logically needed by the application but are still strongly referenced.
